Week 16 of 17 | CSC 240 – Discrete Structures

Practice Only: This lab is an ungraded practice activity designed to prepare you for Quiz 5. It is not submitted for a grade. Use it to build confidence with AVL tree operations and hash table traces.

Objectives

  • Trace AVL tree insertions step-by-step, computing balance factors after each insertion.
  • Identify and apply single (LL, RR) and double (LR, RL) rotations to restore AVL balance.
  • Trace hash table insertions using separate chaining.
  • Trace hash table insertions using linear probing, recording probe sequences.
  • Justify selection of a data structure (AVL tree vs. hash table) for a given scenario.

Scenario

Context: You have been hired as a junior software engineer at QuickLookup Solutions, a small company that builds high-performance search tools for inventory management. The team is evaluating two candidate data structures for their core lookup engine:

  • Option A: An AVL tree that guarantees O(log n) worst-case search and supports sorted traversal.
  • Option B: A hash table that offers O(1) average-case search but does not maintain order.

Your tech lead has asked you to trace through both structures by hand so the team can compare behavior under realistic insert sequences. The exercises below simulate this evaluation process.

Task 1 – AVL Insertion Trace

Instructions

Starting from an empty AVL tree, insert the following keys in order:

Insert sequence: 45, 25, 65, 15, 35, 55, 10, 30

For each insertion:

  1. Draw (or describe) the tree after the key is placed.
  2. Compute the balance factor (BF) of every node on the insertion path (BF = height(left subtree) − height(right subtree)).
  3. If any node has BF = +2 or −2, identify the imbalance type (LL, RR, LR, or RL) and perform the appropriate rotation.
  4. Show the tree after rotation and confirm all balance factors are in {−1, 0, +1}.

Tip: Use a text-based tree format. For example:

45 / \ 25 65 / \ / 15 35 55

Task 5 – Structure Selection Justification

Instructions

Consider the following scenario:

QuickLookup Solutions is building a feature that must support:

  • Guaranteed O(log n) worst-case lookup time (no degradation under adversarial inputs).
  • The ability to output all stored product IDs in sorted order for monthly reports.
  • Frequent insertions (approximately 10,000 new products per day).

In 3–5 sentences, justify which data structure (AVL tree or hash table) is the better choice for this feature. Address:

  • Worst-case time complexity for search and insert.
  • Support for sorted output.
  • Any trade-offs the team should be aware of.

Hints & Tips

Hint for Task 1 – AVL Insertion Trace
  • After inserting 10 into the tree, check the balance factor at node 25. Does it become +2?
  • Remember: update balance factors bottom-up from the inserted node to the root.
  • The first seven insertions (45, 25, 65, 15, 35, 55, 10) will produce a perfectly balanced tree of height 2 before the last insertion. Think about why.
Hint for Task 2 – Rotation Identification
  • LL case: The imbalance is at a node and the heavy subtree is the left child's left subtree. Fix with a single right rotation.
  • RR case: Mirror of LL. Fix with a single left rotation.
  • LR case: The heavy subtree is the left child's right subtree. Fix with a left rotation on the left child, then a right rotation on the imbalanced node (double rotation).
  • RL case: Mirror of LR. Right rotation on right child, then left rotation on imbalanced node.
  • Common mistake: confusing which child to rotate first in double rotations.
Hint for Task 3 – Separate Chaining
  • Compute: 14 mod 7 = 0, 21 mod 7 = 0, 7 mod 7 = 0. Notice that three keys hash to the same index!
  • Load factor α = 8/7 ≈ 1.14. A load factor greater than 1 is fine with chaining.
  • New keys are typically appended to the end of the chain (or prepended—either convention is acceptable; just be consistent).
Hint for Task 4 – Linear Probing
  • With table size 11: 14 mod 11 = 3, 21 mod 11 = 10, 7 mod 11 = 7, 33 mod 11 = 0.
  • Watch for primary clustering: once a cluster forms, subsequent keys that hash into the cluster will probe through all occupied slots.
  • Key 49: 49 mod 11 = 5. Check if index 5 is already occupied by the time you insert 49.
Hint for Task 5 – Structure Selection
  • Hash tables offer O(1) average-case but O(n) worst-case (e.g., all keys collide).
  • AVL trees guarantee O(log n) for both search and insert in the worst case.
  • An in-order traversal of an AVL tree produces sorted output in O(n) time. A hash table requires extracting all keys and sorting them: O(n log n).
  • Trade-off: AVL trees have higher constant factors per operation due to rotations and pointer overhead.
