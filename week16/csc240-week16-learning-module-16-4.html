Module 16.4 – Introduction to Hashing and Collision Resolution

CSC 240 – Discrete Structures • Week 16 • Estimated time: 60–75 minutes
CSLO 3: Implement graph/tree traversal methods in a programming language.

Where We Are
In Modules 16.1–16.3 you mastered AVL trees, which guarantee O(log n) operations by keeping the tree balanced. Now we pivot: what if you don't need ordering and want even faster average-case lookup? Enter hash tables.

1. Motivation: Why Hash Tables?

Scenario

Imagine you manage 10,000 student records and need to look up a student by their 7-digit ID number. Using a balanced BST (like an AVL tree), each lookup requires about ⌈log₂ 10,000⌉ ≈ 14 comparisons. That's fast—but can we do better?

A hash table promises O(1) average-case lookup—constant time regardless of how many elements are stored. Whether you have 100 records or 10 million, the average lookup takes the same number of steps.

The Tradeoff

FeatureAVL TreeHash Table
Average searchO(log n)O(1)
Worst-case searchO(log n)O(n)
Ordered traversalYes (in-order)No
Range queriesEfficientNot supported
Implementation complexityHigher (rotations)Moderate
Key Insight
Hash tables sacrifice ordering for speed. You cannot efficiently answer "find all students with IDs between 1000 and 2000" with a hash table—that requires a tree.

2. The Hash Table Concept

A hash table consists of two parts:

  1. An array of size m (the table).
  2. A hash function h that maps keys to array indices: h: KeySpace → {0, 1, …, m−1}

Basic Operations

  • Store key k: compute h(k) and place the value at index h(k).
  • Retrieve key k: compute h(k) and look at index h(k).
  • Delete key k: compute h(k) and remove the value at that index.

Visual Concept

Think of a hash function as a "slot assignment machine." You feed it a key, and it tells you exactly which slot in the array to use:

Key: 4527  ──→  h(4527) = 4527 mod 7 = 5  ──→  Store at index 5
Key: 8931  ──→  h(8931) = 8931 mod 7 = 4  ──→  Store at index 4

3. Hash Function Techniques and Quality

3.1 Division Method

The simplest and most common technique:

h(k) = k mod m

where m is the table size.

Best Practice
Choose m to be a prime number not close to a power of 2. This helps distribute keys more uniformly. For example, m = 7, m = 11, or m = 97 are good choices. Avoid m = 8, m = 16, or m = 100.

3.2 Multiplication Method

Multiply the key by a constant A (where 0 < A < 1), extract the fractional part, and multiply by m:

h(k) = ⌊m·(kA mod 1)⌋

A popular choice is A ≈ 0.6180339887 (the golden ratio conjugate). This method works well regardless of the choice of m.

3.3 Polynomial Hashing for Strings

For string keys (e.g., names), treat each character as a coefficient in a polynomial:

h(s) = (s[0]·p⁰ + s[1]·p¹ + … + s[n−1]·p^(n−1)) mod m

where p is a small prime (e.g., 31 or 37) and s[i] is the character code at position i. This ensures that different orderings of characters produce different hash values.

Properties of a Good Hash Function
  • Deterministic: same key always produces the same index.
  • Uniform distribution: keys spread evenly across all m slots.
  • Efficient: computable in O(1) time.
  • Minimizes clustering: similar keys don't map to adjacent slots.

4. Collisions and the Pigeonhole Principle

Here's the fundamental problem: if our key space is larger than the table size, collisions are inevitable.

The Collision Problem

A collision occurs when two different keys k₁ ≠ k₂ hash to the same index: h(k₁) = h(k₂).

Connection to the Pigeonhole Principle (Week 2)

Recall from Week 2: If you place n items into m containers and n > m, then at least one container must hold more than one item.

Apply this to hashing:

  • Items = keys we want to store
  • Containers = table slots (indices 0 through m−1)

If we store more than m keys in a table of size m, at least two keys must collide. In fact, even with fewer than m keys, collisions are likely (similar to the Birthday Problem from Week 9).

Example: With a table of size 7 and 8 keys, the Pigeonhole Principle guarantees at least one collision—no matter how clever our hash function is.

Since collisions cannot be avoided, we need collision resolution strategies. The two main families are:

  1. Separate Chaining – store multiple items at the same index using a linked list.
  2. Open Addressing – find another empty slot within the table itself.

5. Separate Chaining

Concept

In separate chaining, each slot in the hash table holds a linked list (or other collection) of all key–value pairs that hash to that index. When a collision occurs, the new key is simply appended to the list at that slot.

Operations

  • Insert(k): Compute h(k), add k to the front of the list at index h(k). — O(1)
  • Search(k): Compute h(k), traverse the list at index h(k) looking for k. — O(chain length)
  • Delete(k): Compute h(k), find and remove k from the list at index h(k). — O(chain length)

Worked Example: Separate Chaining

Setup: Table size m = 7, hash function h(k) = k mod 7.
Insert keys: 23, 14, 9, 30, 22, 16, 7 (in this order).

6. Open Addressing

In open addressing, all elements are stored directly in the table array—there are no linked lists. When a collision occurs, we probe for the next available slot using a systematic sequence.

Linear Probing

Formula:

h(k, i) = (h(k) + i) mod m, i = 0, 1, 2, …

If slot h(k) is occupied, try h(k)+1, then h(k)+2, and so on (wrapping around).

Advantage: Simple to implement; good cache performance (sequential memory access).

Disadvantage: Primary Clustering. Occupied slots tend to form long contiguous runs. Any key that hashes into a cluster must traverse the entire cluster before finding an empty slot, making the cluster even longer. This creates a "rich get richer" effect.

Quadratic Probing

Formula:

h(k, i) = (h(k) + c₁i + c₂i²) mod m, i = 0, 1, 2, …

Common choice: c₁ = 0, c₂ = 1, giving probe sequence h(k), h(k)+1, h(k)+4, h(k)+9, …

Advantage: Breaks up primary clusters because probes jump by increasing amounts.

Disadvantage: May not visit all slots (depends on table size and constants). Works best when m is prime and the table is less than half full. Can still suffer from secondary clustering: keys that hash to the same initial slot follow the same probe sequence.

Double Hashing

Formula:

h(k, i) = (h₁(k) + i·h₂(k)) mod m, i = 0, 1, 2, …

where h₁ and h₂ are two different hash functions. A common choice:

  • h₁(k) = k mod m
  • h₂(k) = 1 + (k mod (m−1))

Advantage: Best distribution among open addressing methods. Different keys that collide at the same slot will have different probe sequences (since h₂ depends on the key). Eliminates both primary and secondary clustering.

Disadvantage: Slightly more expensive to compute (two hash functions). Must ensure h₂(k) ≠ 0 and that h₂(k) is coprime to m (choosing m prime helps).

Worked Example: Linear Probing

Setup: Table size m = 11, hash function h(k) = k mod 11.
Insert keys: 23, 14, 9, 30, 22, 16, 7 (same keys as the chaining example).

7. Deletion in Open Addressing (Tombstones)

The Problem

In separate chaining, deletion is straightforward—just remove the node from the linked list. But in open addressing, deletion is tricky.

Why Simple Deletion Breaks Probing

Consider this scenario with linear probing (m = 11):

  1. Insert key A at index 5.
  2. Insert key B at index 5 → collision → probes to index 6.
  3. Insert key C at index 5 → collision → probes 6 (occupied) → probes to index 7.
  4. Now delete key B from index 6 (mark it empty).
  5. Search for key C: hash to 5 (occupied by A, not C) → probe 6 (empty!) → search stops and reports "not found."

But key C is in the table at index 7! The empty slot at index 6 broke the probe chain.

The Solution: Lazy Deletion (Tombstones)

Instead of truly emptying a slot, mark it with a special tombstone (or "deleted") marker:

  • During search: treat tombstones as occupied (keep probing past them).
  • During insert: treat tombstones as empty (you can place a new key there).

This preserves probe chains while still allowing reuse of deleted slots.

Performance Note
Too many tombstones degrade performance (long probe sequences with no actual data). When tombstones accumulate, it may be time to rehash—rebuild the table from scratch, inserting only the live keys into a fresh array.

Check 16.4a: You have a hash table of size 13 using h(k) = k mod 13. Which of the following sets of keys will ALL hash to the same slot?

✓ Correct! All keys in {5, 18, 31, 44} are congruent to 5 mod 13: 5 mod 13 = 5, 18 mod 13 = 5, 31 mod 13 = 5, 44 mod 13 = 5. Keys that differ by multiples of m always collide—this is the pattern to watch for when evaluating hash function quality.
✗ Not quite. Compute each key mod 13: {5, 18, 31, 44} → {5, 5, 5, 5} ✓ all same. Check {13, 26, 39, 52} → {0, 0, 0, 0} — these also all collide! But look at option (b): 13 mod 13 = 0, 26 mod 13 = 0, 39 mod 13 = 0, 52 mod 13 = 0. Both (a) and (b) have all keys hashing to the same slot, but the best answer demonstrating the collision pattern (keys differing by exactly m) is (a), where the keys are 5, 5+13, 5+26, 5+39. The key insight: any arithmetic sequence with common difference m produces total collision.

Check 16.4b: Using separate chaining with m = 7 and h(k) = k mod 7, you insert keys [10, 17, 24, 3, 38] in order. What does the chain at index 3 contain (from front to back, assuming front insertion)?

✓ Correct! All five keys hash to index 3: 10 mod 7 = 3, 17 mod 7 = 3, 24 mod 7 = 3, 3 mod 7 = 3, 38 mod 7 = 3. With front insertion, each new key goes to the front: after all insertions the chain is 38 → 3 → 24 → 17 → 10 (most recent first). This is also a demonstration of worst-case behavior—all keys colliding at one slot!
✗ Not quite. Compute each hash: 10 mod 7 = 3, 17 mod 7 = 3, 24 mod 7 = 3, 3 mod 7 = 3, 38 mod 7 = 3. All five keys go to slot 3! With front insertion, the order (front to back) is the reverse of insertion order: 38 → 3 → 24 → 17 → 10. Remember: front insertion means the last key inserted is at the front of the list.

Check 16.4c: Using linear probing with m = 11 and h(k) = k mod 11, you insert keys [22, 33, 44] in order. What is the probe sequence for inserting 44?

✓ Correct! 22 mod 11 = 0 → inserted at index 0. 33 mod 11 = 0 → collision at 0, probe to 1 (empty) → inserted at index 1. 44 mod 11 = 0 → collision at 0 (has 22), probe to 1 (has 33), probe to 2 (empty) → inserted at index 2. This is a perfect example of primary clustering: three keys that hash to the same slot form a contiguous block at indices 0, 1, 2.
✗ Not quite. Let's trace: 22 mod 11 = 0 → index 0. 33 mod 11 = 0 → collision → try 1 (empty) → index 1. 44 mod 11 = 0 → collision → try 1 (occupied by 33) → try 2 (empty) → index 2. The correct probe sequence for 44 is: index 0 (occupied by 22) → index 1 (occupied by 33) → index 2 (empty, insert here). This illustrates primary clustering.

Check 16.4d: A hash table has 13 slots. A classmate claims: "If I choose a really good hash function, I can insert 14 keys with zero collisions." Using a concept from Week 2, explain why this is impossible.

✓ Correct! The Pigeonhole Principle (Week 2) states: if n items are placed into m containers and n > m, at least one container must hold more than one item. Here, 14 keys (pigeons) mapped to 13 slots (pigeonholes) guarantees at least one collision—regardless of how good the hash function is. No hash function can be injective (one-to-one) when the domain size exceeds the codomain size.
✗ Not quite. While options (b) and (c) mention real issues, the fundamental reason is the Pigeonhole Principle: 14 keys into 13 slots guarantees at least one collision. This is a mathematical certainty, not dependent on the probing method or load factor threshold. The Pigeonhole Principle (from Week 2) is the correct justification.
