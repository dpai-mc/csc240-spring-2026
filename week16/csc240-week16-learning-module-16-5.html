1. Load Factor and Hash Table Performance

1.1 Defining the Load Factor

The load factor (denoted α) is the single most important metric for predicting hash table performance:

α = n/m

where n = number of elements stored and m = number of table slots (buckets).

Intuitively, α represents the average number of elements per slot. A higher load factor means more crowding, more collisions, and degraded performance.

1.2 Separate Chaining Performance

With separate chaining, each slot holds a linked list of colliding elements. Under the simple uniform hashing assumption (each key is equally likely to hash to any slot):

  • Expected chain length = α
  • Unsuccessful search: examine α nodes on average → O(1 + α)
  • Successful search: examine about 1 + α/2 nodes → O(1 + α)
  • Insert: O(1) (insert at head of chain) + hash computation
  • Delete: O(1 + α) (search then remove from linked list)
✅ Key Insight

If the table size grows proportionally with n (keeping α bounded by a constant), all operations remain O(1) average case. For example, if we maintain α ≤ 2, the expected chain length never exceeds 2.

1.3 Open Addressing Performance

With open addressing, all elements live directly in the table array. Since every slot can hold at most one element, we must have α ≤ 1. Performance degrades sharply as α approaches 1:

⚠️ The 0.7 Threshold

Most practical implementations resize when α reaches approximately 0.7 (Java's HashMap uses 0.75). Beyond this point, clustering effects cause severe performance degradation for open addressing.

2. Resizing (Rehashing)

2.1 When and Why

When the load factor exceeds the threshold (e.g., α > 0.7), the table must be resized:

  1. Allocate a new, larger table (typically 2m slots, or the next prime > 2m).
  2. Rehash every existing element — recompute h(k) using the new table size and insert into the new table.
  3. Deallocate the old table.

Rehashing is necessary because hash values depend on m (e.g., h(k) = k mod m). Changing m changes where every key belongs.

2.2 Amortized O(1) Insertion

A single resize costs O(n) — we must reinsert all n elements. However, resizes happen rarely. Using a doubling strategy:

  • After inserting n elements, the total work from all resizes is at most n + n/2 + n/4 + … ≤ 2n.
  • Spread over n insertions, the amortized cost per insertion is O(1).
📐 Analogy

Think of resizing like moving to a bigger apartment. Moving day is expensive, but you only move when you've outgrown the current space. Averaged over all the months you lived there, the per-month cost of moving is small.

3. Worst-Case Reminder

🚨 Critical Point for Quiz 5

Hash tables do NOT guarantee O(1) worst-case performance. If all n keys hash to the same slot:

  • Chaining: one chain of length n → search is O(n)
  • Open addressing: all slots probed sequentially → O(n)

This is why hash tables provide O(1) average case (assuming good hash function and bounded load factor) but O(n) worst case.

When does worst case matter?

  • Security-sensitive applications (adversarial input can be crafted to cause collisions)
  • Real-time systems requiring guaranteed response times
  • Any scenario where occasional O(n) spikes are unacceptable

In these cases, an AVL tree (guaranteed O(log n) worst case) is the safer choice.

5. When to Choose Each Structure

  • Data arrives in a random order (average case is O(log n)).
  • You need ordered operations (sorted output, range queries, predecessor/successor).
  • Implementation simplicity is a priority and you can tolerate occasional O(n) worst case.
  • Not suitable when: input may be sorted or nearly sorted (degenerates to linked list).
  • You need guaranteed O(log n) for all operations regardless of input order.
  • You need ordered traversal, range queries, or min/max operations.
  • Worst-case performance matters (real-time systems, security-sensitive applications).
  • Data is frequently searched and relatively infrequently inserted/deleted (AVL is slightly slower on inserts due to rotations).
  • Tradeoff: more complex to implement; slight constant-factor overhead from balance maintenance.
  • You need the fastest possible average-case lookup — O(1).
  • You do not need ordered operations (no sorted traversal, no range queries).
  • You can tolerate O(n) worst case (or mitigate with a good hash function).
  • Keys have a natural hash function (integers, strings).
  • Not suitable when: you need to iterate in sorted order, perform range queries, or guarantee worst-case response time.
💡 Decision Shortcut

Need ordering? → Tree (AVL if worst-case matters, BST if input is random).
Need only lookup by key? → Hash table.
Need both? → Consider using two structures (hash table for lookup + tree for ordering) or a balanced BST as a single compromise.

6. Mini-Case: Student Records System

📋 Scenario

You are designing a student records system with three requirements:

  1. Fast ID lookup: Given a student ID, retrieve the record in the fastest average time.
  2. Ordered name printing: Print all students sorted alphabetically by last name.
  3. GPA range queries: Find all students with GPA between 3.0 and 3.5.

You have 10,000 student records. Which data structure(s) should you use?

Work through the analysis step by step:

Best fit: Hash Table keyed on student ID.

  • Student IDs are unique integers — perfect hash keys.
  • Average lookup: O(1).
  • With 10,000 records and a table of size ~14,300 (load factor ≈ 0.7), we get excellent performance.

Could an AVL tree work? Yes — O(log 10,000) ≈ 14 comparisons. Fast, but not as fast as hash table's O(1).

Best fit: AVL Tree (or BST) keyed on last name.

  • Inorder traversal produces alphabetical output in O(n).
  • A hash table cannot produce sorted output without collecting all elements and sorting them — O(n log n) extra work.

AVL is preferred over unbalanced BST because names inserted alphabetically (e.g., from a sorted file) would degenerate a BST to O(n) height.

Best fit: AVL Tree (or BST) keyed on GPA.

  • Range query "find all GPAs in [3.0, 3.5]" takes O(log n + k) in a tree, where k is the number of results.
  • A hash table would require scanning all 10,000 records — O(n).

Use multiple structures:

StructureKeyPurpose
Hash TableStudent IDFast ID lookup — O(1) average
AVL Tree #1Last NameOrdered alphabetical printing — O(n) inorder
AVL Tree #2GPARange queries — O(log n + k)

Space tradeoff: storing 10,000 records in three structures uses ~3× memory, but each operation achieves optimal time complexity for its requirement. This is a classic example of the space-time tradeoff in data structure design.

Alternative (simpler): If you can only use one structure, an AVL tree keyed on student ID gives O(log n) lookup, supports ordered traversal (by ID), but requires a separate scan for GPA ranges. No single structure optimally satisfies all three requirements.
