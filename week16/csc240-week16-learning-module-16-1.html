Module 16.1 – The Balance Problem and AVL Property

Week 16 · CSC 240 Discrete Structures · Self-Balancing Trees

1. Opening: From BSTs to the Balance Problem

🔗Connection to Week 15: Last week you implemented Binary Search Trees and proved that search, insert, and delete all run in O(h) time, where h is the height of the tree. You saw that a well-shaped BST gives h ≈ log₂ n, making operations fast.

But here's the uncomfortable question we left open: What if the tree isn't well-shaped?

Imagine you are building a phone-book lookup system. You insert names in alphabetical order: Adams, Baker, Clark, Davis, Evans, Foster…

Each new name is larger than the last, so it always goes to the right child. The tree degenerates into a linked list. Your O(log n) promise collapses to O(n).

This week, we study the first (and most elegant) solution to this problem: the AVL tree, a self-balancing BST that guarantees logarithmic height at all times.

2. The Degenerate BST Problem

Worked Example: Inserting Sorted Keys

Insert the keys 10, 20, 30, 40, 50 into an initially empty BST using standard insertion (no balancing):

10 \ 20 \ 30 \ 40 \ 50

⚠️ Problem: 5 nodes but height = 4 (counting edges). To search for 50, we visit all 5 nodes — no better than a linked list!

In general, inserting n keys in sorted order produces height n − 1, giving O(n) operations.

A balanced BST holding the same 5 keys would look like this:

30 / \ 10 40 \ \ 20 50

Height = 2. Searching for any key takes at most 3 comparisons. We need a mechanism that automatically maintains this kind of shape.

3. The AVL Balance Property

3.1 Balance Factor Definition

Definition: For any node v in a binary tree, the balance factor is:

BF(v) = height(left subtree of v) − height(right subtree of v)

Convention: the height of an empty (null) subtree is −1.

AVL Property: A binary search tree is an AVL tree if and only if for every node v in the tree:

BF(v) ∈ {−1, 0, +1}

A node with |BF(v)| ≥ 2 is said to be unbalanced or in violation of the AVL property.

3.2 Computing Balance Factors — Worked Examples

Consider this tree:

20 [BF = 0] / \ 10 30 [BF=0] [BF=−1] \ 35 [BF=0]

Calculations:

  • Node 10: left height = −1, right height = −1 → BF = −1 − (−1) = 0
  • Node 35: left height = −1, right height = −1 → BF = 0
  • Node 30: left height = −1, right height = 0 → BF = −1 − 0 = −1
  • Node 20: left height = 0, right height = 1 → BF(20) = 0 − 1 = −1… wait, let me recalculate. Left subtree rooted at 10 has height 0. Right subtree rooted at 30 has height 1. So BF(20) = 0 − 1 = −1. ✓ Still valid!

All balance factors are in {−1, 0, +1}. This is a valid AVL tree. ✅

Now insert 40 into the previous tree (standard BST insert, no rebalancing):

20 [BF = −2] ← VIOLATION / \ 10 30 [BF=0] [BF=−2] ← VIOLATION \ 35 [BF=−1] \ 40 [BF=0]

Calculations:

  • Node 40: BF = −1 − (−1) = 0 ✓
  • Node 35: BF = −1 − 0 = −1 ✓
  • Node 30: left height = −1, right height = 1 → BF = −1 − 1 = −2 ❌ Violation!
  • Node 20: left height = 0, right height = 2 → BF = 0 − 2 = −2 ❌ Violation!

The AVL property is broken. A rotation (covered in Module 16.2) would fix this.

3.3 Key Observations

  • A balance factor of +1 means the node is "left-heavy" (left subtree is one level taller).
  • A balance factor of −1 means the node is "right-heavy."
  • A balance factor of 0 means both subtrees have equal height.
  • After an insertion or deletion, violations can only occur on the path from the modified node to the root.

4. The O(log n) Height Guarantee

4.1 Intuition: How Tall Can an AVL Tree Get?

We want to prove that an AVL tree with n nodes has height O(log n). The strategy: ask the reverse question — what is the minimum number of nodes an AVL tree of height h can have?

Deeper Dive: The Fibonacci-Recurrence Argument

Let N(h) = minimum number of nodes in an AVL tree of height h.

  • N(0) = 1 (a single root)
  • N(1) = 2 (root + one child)

For height h ≥ 2: the root must have two subtrees. To minimize nodes while achieving height h, make one subtree have height h−1 and the other height h−2 (the minimum allowed by the AVL property). So:

N(h) = 1 + N(h−1) + N(h−2)

This is essentially the Fibonacci recurrence (shifted by 1). The Fibonacci numbers grow exponentially as φh where φ = (1+√5)/2 ≈ 1.618.

Therefore N(h) ≥ φh, which means:

n ≥ φh  ⟹  h ≤ logφ(n) ≈ 1.44 × log₂(n)

Conclusion: The height of an AVL tree is at most about 1.44 × log₂ n. This is still O(log n).

5. Historical Context

🏛️ The AVL tree was invented in 1962 by Soviet mathematicians Georgy Adelson-Velsky and Evgenii Landis, published in their paper "An algorithm for the organization of information." It was the first self-balancing binary search tree data structure ever proposed.

The name "AVL" comes from the initials of its inventors: Adelson-Velsky and Landis.

Their work was groundbreaking because it showed that maintaining balance dynamically (after every insertion/deletion) was not only possible but efficient — requiring at most O(log n) additional work per operation. This opened the door to many later balanced-tree structures (Red-Black trees, B-trees, etc.).

6. Visualizer Walk-Through Activity

For this activity, open the AVL Tree Visualizer (linked in this week's module page). Follow the steps below and observe what happens at each stage.

Activity: Observing the Balance Problem and AVL Rotations

  1. Degenerate insertion: Insert the keys 10, 20, 30, 40, 50 one at a time (in that order) into a plain BST mode (if available) or simply imagine the result. Notice the tree becomes a right-leaning chain with height 4.
  2. AVL insertion — same sequence: Clear the tree and switch to AVL mode. Insert the same keys: 10, 20, 30, 40, 50. Watch how the visualizer performs rotations after inserting 30 (the first violation). Note the resulting tree height.
  3. Observe balance factors: After all 5 keys are inserted, hover over (or inspect) each node. Confirm that every node has a balance factor in {−1, 0, +1}.
  4. Worst-case AVL sequence: Clear the tree and insert: 50, 30, 70, 20, 40, 60, 80, 10. This builds a taller AVL tree. Note the height and where (if any) rotations occur.
  5. Reflection: Compare the final heights from steps 1 and 2. The plain BST had height 4 for 5 nodes; the AVL tree should have height ≤ 2. This is the power of automatic balancing.
What to look for during rotations
  • When a rotation triggers, note which node was the "violator" (the first ancestor with |BF| = 2).
  • Notice whether it's a single or double rotation (we'll formalize these in Module 16.2).
  • After the rotation, verify the BST ordering property still holds — rotations preserve it!
  • Count the nodes that actually moved. Rotations are local operations (only 2–3 nodes change position).
