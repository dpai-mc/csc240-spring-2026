Module 17.4 – Code Quality, Documentation, and Final Submission

This is the final module of CSC 240. You've built graph algorithms from scratch, implemented tree-based data structures, measured runtime performance, and written technical analysis. Now it's time to polish, package, and submit a professional-quality project.

Think of this module as your project's finishing stage: the difference between a working prototype and a deliverable that you'd be proud to show a future employer or transfer institution. Clean code, clear documentation, and a complete submission package demonstrate mastery not just of discrete structures, but of software engineering practice.

Code Quality Checklist

The grading rubric allocates 20% (50 points) to code quality. This isn't about whether your algorithms produce correct output — that's functionality. Code quality is about whether another developer (or your future self) can read, understand, and maintain your code. Expand each criterion below to understand what the grader evaluates:

What it means: Every variable, function, and class name communicates its purpose without requiring a comment to explain it.

Ask yourself:

  • Could someone reading vertexCount immediately know what it stores? (Yes!) Could they know what n stores? (Maybe not.)
  • Does findShortestPath() describe what the function does? (Yes!) Does doStuff()? (No.)
  • Are abbreviations consistent? If you use adj for "adjacency" in one place, don't use adjList, adjacency, and aList elsewhere.

Common pitfalls: single-letter names (x, i, n outside of loop counters), generic names (temp, data, result), misleading names (a variable called list that's actually an array).

What it means: Each function does one thing and does it well. Functions are short (ideally under 50 lines), and complex operations are broken into named steps.

Ask yourself:

  • Is your main() method under 50 lines? If not, extract logical blocks into helper functions.
  • Can you describe what each function does in one sentence? If you need "and" in the sentence, the function probably does too much.
  • Are there repeated code blocks? Extract them into a shared helper.

Common pitfalls: a 500-line main() method, copy-pasted code with minor variations, functions that both compute a result and print it (separate computation from display).

What it means: Indentation, brace placement, spacing, and blank lines follow a single consistent style throughout the entire codebase.

Ask yourself:

  • Are you using tabs or spaces? Pick one and use it everywhere.
  • Is your brace style consistent? (K&R, Allman, or other — just be consistent.)
  • Do you have consistent spacing around operators? (x = y + z not sometimes x=y+z)
  • Are blank lines used consistently to separate logical sections?

Tip: Most IDEs have an "auto-format" feature. Run it on your entire project before submission.

What it means: Your submitted code contains nothing that isn't actively used: no commented-out blocks, no unused variables, no unreachable code paths, no leftover debugging output.

Ask yourself:

  • Search for System.out.println("DEBUG") or cout << "test" — remove them all.
  • Are there commented-out blocks of code? Delete them. (Version control preserves history; your submission shouldn't.)
  • Does your compiler report any "unused variable" warnings? Fix them.

Common pitfalls: leftover TODO comments with no resolution, functions that are defined but never called, import/include statements for unused libraries.

What it means: Your application handles unexpected or invalid input gracefully — it doesn't crash, and it communicates the problem clearly to the user.

Ask yourself:

  • What happens if the user enters an invalid vertex number? (Should print an error message, not throw an unhandled exception.)
  • What happens on an empty graph (0 vertices)? Does each algorithm handle this case?
  • What happens if Dijkstra's is called on a disconnected graph? (Should report "no path" for unreachable vertices.)
  • What happens if topological sort is called on a graph with a cycle? (Should detect and report the cycle.)

Common pitfalls: NullPointerException / segfault on empty input, array index out of bounds on edge cases, infinite loops on cyclic input to topological sort.

Before/After: Common Code Quality Improvements

Below are three concrete examples of transformations that move code from "it works" to "it's professional." Study these patterns and apply them to your own project.

1. Renaming Vague Variables

Before:

int n = 20;
int[][] arr = new int[n][n];
for (int i = 0; i < n; i++) {
    for (int j = 0; j < n; j++) {
        if (arr[i][j] != 0) {
            // process edge
        }
    }
}

After:

int vertexCount = 20;
int[][] adjacencyMatrix = new int[vertexCount][vertexCount];
for (int source = 0; source < vertexCount; source++) {
    for (int destination = 0; destination < vertexCount; destination++) {
        if (adjacencyMatrix[source][destination] != 0) {
            int edgeWeight = adjacencyMatrix[source][destination];
            processEdge(source, destination, edgeWeight);
        }
    }
}

Why it matters: The "after" version requires zero comments to understand. A reader immediately knows what source, destination, and edgeWeight represent.

2. Extracting Helper Functions

Before: (a 40-line block buried inside main())

// inside main(), after building the graph...
boolean[] visited = new boolean[vertexCount];
int[] distance = new int[vertexCount];
Arrays.fill(distance, Integer.MAX_VALUE);
distance[startVertex] = 0;
MinHeap pq = new MinHeap(vertexCount);
pq.insert(startVertex, 0);
while (!pq.isEmpty()) {
    int u = pq.extractMin();
    if (visited[u]) continue;
    visited[u] = true;
    for (Edge edge : adjacencyList[u]) {
        int v = edge.destination;
        int w = edge.weight;
        if (!visited[v] && distance[u] + w < distance[v]) {
            distance[v] = distance[u] + w;
            pq.insert(v, distance[v]);
        }
    }
}
// ... print results ...

After:

/**
 * Computes shortest paths from startVertex to all reachable vertices
 * using Dijkstra's algorithm with a student-implemented min-heap.
 *
 * @param graph       the adjacency list representation
 * @param startVertex the source vertex
 * @return array where distance[v] = shortest distance from startVertex to v
 */
public static int[] dijkstra(List<Edge>[] graph, int startVertex) {
    int vertexCount = graph.length;
    boolean[] visited = new boolean[vertexCount];
    int[] distance = new int[vertexCount];
    Arrays.fill(distance, Integer.MAX_VALUE);
    distance[startVertex] = 0;

    MinHeap priorityQueue = new MinHeap(vertexCount);
    priorityQueue.insert(startVertex, 0);

    while (!priorityQueue.isEmpty()) {
        int current = priorityQueue.extractMin();
        if (visited[current]) continue;
        visited[current] = true;
        relaxNeighbors(graph, current, distance, visited, priorityQueue);
    }
    return distance;
}

private static void relaxNeighbors(List<Edge>[] graph, int current,
        int[] distance, boolean[] visited, MinHeap priorityQueue) {
    for (Edge edge : graph[current]) {
        int neighbor = edge.destination;
        int newDist = distance[current] + edge.weight;
        if (!visited[neighbor] && newDist < distance[neighbor]) {
            distance[neighbor] = newDist;
            priorityQueue.insert(neighbor, newDist);
        }
    }
}

Why it matters: The algorithm is now reusable, testable in isolation, and self-documenting. The main() method simply calls dijkstra(graph, 0).

3. Removing Debug Output

Before:

public int extractMin() {
    System.out.println("DEBUG: extracting min, heap size = " + size);
    System.out.println("DEBUG: root value = " + heap[0].priority);
    int minVertex = heap[0].vertex;
    heap[0] = heap[size - 1];
    size--;
    System.out.println("DEBUG: about to heapify down");
    heapifyDown(0);
    System.out.println("DEBUG: done with extractMin");
    return minVertex;
}

After:

/**
 * Removes and returns the vertex with the smallest priority (distance).
 * Maintains the heap property via downward percolation.
 *
 * @return the vertex index with minimum priority
 * @throws IllegalStateException if the heap is empty
 */
public int extractMin() {
    if (size == 0) {
        throw new IllegalStateException("Cannot extract from empty heap");
    }
    int minVertex = heap[0].vertex;
    heap[0] = heap[size - 1];
    size--;
    if (size > 0) {
        heapifyDown(0);
    }
    return minVertex;
}

Why it matters: Debug output clutters the user experience and the grader's terminal. The "after" version also adds proper error handling for the empty-heap edge case.
