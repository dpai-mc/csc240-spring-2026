Why Runtime Comparison Matters

Throughout this course you have analyzed algorithms using Big-O notation — a theoretical tool that describes how an algorithm's resource usage grows with input size. But theory alone cannot tell you whether your specific implementation achieves the expected growth rate. That is where empirical runtime measurement comes in: you time your code on real inputs and compare the measured growth to the theoretical prediction.

If the two match, you gain confidence that your implementation is correct and efficient. If they diverge, you have a powerful debugging signal — perhaps your priority queue performs linear scans instead of logarithmic extractions, or your adjacency-matrix traversal adds an unexpected |V|² factor.

This module prepares you to produce the side-by-side runtime output required by Component 3 of the Final Project and to write the complexity analysis section of your Analysis Document. By the end, you will have measured, recorded, and interpreted the performance of every graph algorithm in your application.

How to Measure Runtime

Both Java and C++ provide high-resolution timing utilities. The key pattern is: record a start time immediately before algorithm execution, record an end time immediately after, and compute the difference. Wrap only the algorithm — not graph construction or output printing — inside the timing window.

Java 
// Measure a single algorithm run
long startTime = System.nanoTime();

// --- Algorithm execution (e.g., BFS) ---
bfs(graph, sourceVertex);

long endTime = System.nanoTime();
long durationMs = (endTime - startTime) / 1_000_000; // convert to ms

System.out.println("BFS duration: " + durationMs + " ms");

// --- Averaging multiple runs ---
int trials = 5;
long totalNanos = 0;
for (int t = 0; t < trials; t++) {
    resetVisited(); // clear state between runs
    long s = System.nanoTime();
    bfs(graph, sourceVertex);
    long e = System.nanoTime();
    totalNanos += (e - s);
}
double avgMs = (totalNanos / (double) trials) / 1_000_000.0;
System.out.printf("BFS avg (%d trials): %.2f ms%n", trials, avgMs);

💡 Tip

Use System.nanoTime() — not System.currentTimeMillis(). The latter has only millisecond granularity and can jump due to system clock adjustments.

C++ 
#include <chrono>
#include <iostream>

// Measure a single algorithm run
auto start = std::chrono::high_resolution_clock::now();

// --- Algorithm execution (e.g., BFS) ---
bfs(graph, sourceVertex);

auto end = std::chrono::high_resolution_clock::now();
auto duration = std::chrono::duration_cast<std::chrono::microseconds>(end - start);
std::cout << "BFS duration: " << duration.count() / 1000.0 << " ms\n";

// --- Averaging multiple runs ---
int trials = 5;
long long totalMicro = 0;
for (int t = 0; t < trials; t++) {
    resetVisited(); // clear state between runs
    auto s = std::chrono::high_resolution_clock::now();
    bfs(graph, sourceVertex);
    auto e = std::chrono::high_resolution_clock::now();
    totalMicro += std::chrono::duration_cast<std::chrono::microseconds>(e - s).count();
}
double avgMs = (totalMicro / static_cast<double>(trials)) / 1000.0;
std::cout << "BFS avg (" << trials << " trials): " << avgMs << " ms\n";

💡 Tip

Compile with optimizations disabled (-O0) during timing experiments so the compiler does not eliminate your algorithm calls. Alternatively, use the result (print a checksum) to prevent dead-code elimination.

⚠ Common Pitfalls

  • Including I/O in the timing window — printing traversal orders inside the timed block inflates results. Time the algorithm alone, then print results separately.
  • Forgetting to reset state — if your visited array is not cleared between trials, subsequent runs will return instantly.
  • JVM warm-up (Java) — the first 1–2 runs may be slower due to JIT compilation. Discard warm-up runs or include a warm-up loop before measurement begins.

Designing Your Runtime Experiment

A well-designed experiment isolates the variable you care about (input size) and controls everything else. Follow this structure:

1. Choose Graph Sizes

  • Minimum requirement: Two sizes (e.g., 20 vertices / 50 edges and 100 vertices / 500 edges).
  • Recommended: Three sizes to see a growth trend (e.g., 20, 60, 200 vertices).
  • Keep the edge-to-vertex ratio consistent across sizes so you are scaling both |V| and |E| proportionally (e.g., always ~2.5 edges per vertex for sparse, or ~|V|·(|V|−1)/4 for moderate density).

2. Generate or Construct Test Graphs

  • Hard-coded small graph: Useful for correctness verification (you can check results by hand).
  • Programmatic random graph: Write a helper that creates a connected graph with a given |V| and |E|. Strategy: first create a spanning tree (ensures connectivity), then add random edges until you reach the target |E|.
  • File-based input: Store graphs in a file and load them — ensures identical input across runs.

4. Number of Trials

  • Run each algorithm 3–5 times on the same graph and report the average.
  • This reduces noise from background system processes, garbage collection (Java), or cache effects.
  • Optionally discard the highest and lowest times (trimmed mean) for cleaner data.

Interpreting Your Results

Use this guide to compare your measured growth ratios against theoretical expectations. The key question: when I scale the input by factor k, does the runtime scale by the factor Big-O predicts?

AlgorithmTime ComplexityIf Input ×5Expected Runtime Factor
BFS O(|V| + |E|) |V| ×5, |E| ×5 ≈ 5×
DFS (recursive or iterative) O(|V| + |E|) |V| ×5, |E| ×5 ≈ 5×
Prim's (binary heap) O((|V| + |E|) log |V|) |V| ×5, |E| ×5 ≈ 5 × (log(5n) / log(n)) — slightly more than 5×
Kruskal's (Union-Find) O(|E| log |E|) |E| ×5 ≈ 5 × (log(5m) / log(m)) — slightly more than 5×
Dijkstra's (binary heap) O((|V| + |E|) log |V|) |V| ×5, |E| ×5 ≈ 5 × (log(5n) / log(n)) — slightly more than 5×

Reading the Ratio

Suppose your Graph 1 has 20 vertices and Graph 2 has 100 vertices (a 5× increase), with edges scaled proportionally:

  • BFS/DFS: If Graph 1 takes 0.08 ms and Graph 2 takes 0.35 ms, the ratio is 0.35/0.08 ≈ 4.4×. Close to 5× — consistent with O(|V| + |E|). ✓
  • Prim's/Dijkstra's: If Graph 1 takes 0.20 ms and Graph 2 takes 1.30 ms, the ratio is 6.5×. Slightly above 5× due to the log factor — consistent with O((|V|+|E|) log |V|). ✓

When measured results don't match theory, use this diagnostic table:

SymptomLikely CauseFix
BFS/DFS ratio ≈ 25× when input scaled 5× Iterating over full adjacency matrix rows → O(|V|²) instead of O(|V|+|E|) Switch neighbor iteration to adjacency list
Dijkstra's ratio ≈ 25× when input scaled 5× Linear scan to find minimum instead of heap extract → O(|V|²) Use your min-heap's extractMin() for O(log |V|)
Prim's much slower than expected Rebuilding heap from scratch each iteration instead of using decrease-key Implement decrease-key or use lazy deletion with a visited check
Runtime varies wildly between trials Garbage collection (Java) or background processes Increase trials to 10; discard outliers; close other applications
Second run is much faster than first JVM JIT warm-up (Java) or CPU cache effects Add 2–3 warm-up iterations before measurement begins
Recursive DFS crashes on large graphs Stack overflow from deep recursion Increase stack size or note limitation in README; use iterative DFS for large inputs

⚠ Important

If your results don't match theory, don't fabricate data. Report what you measured and explain the discrepancy in your Analysis Document. Honest analysis of unexpected results demonstrates deeper understanding and earns more credit than faked numbers.

Self-Check: Runtime Analysis

Ungraded Knowledge Check – Module 17.2

Test your understanding before moving on. These items align with the Final Project runtime comparison requirement.

Check 17.2a: If you double both the number of vertices and the number of edges in your graph, by what factor should BFS runtime approximately increase? Why?

Answer: Approximately 2×. BFS has time complexity O(|V| + |E|). If both |V| and |E| double, then (2|V| + 2|E|) / (|V| + |E|) = 2. The runtime should roughly double because BFS visits each vertex and each edge exactly once, and you doubled both quantities.

Check 17.2b: You run Prim's MST algorithm (with a binary heap) on Graph 1 (20 vertices, 50 edges) and get 0.18 ms. On Graph 2 (100 vertices, 250 edges) you get 1.20 ms. The ratio is 6.7×. The input scaled by 5× in both |V| and |E|. Is this consistent with O((|V|+|E|) log |V|)? Explain in one sentence.

Answer: Yes, this is consistent. The expected ratio is approximately 5 × (log 100 / log 20) ≈ 5 × (2 / 1.3) ≈ 7.7×. The measured 6.7× is in the right ballpark — slightly below the theoretical maximum due to constant factors and measurement noise. The key insight: the log factor causes Prim's to grow faster than linearly, which is exactly what the 6.7× (greater than 5×) demonstrates.

Check 17.2c: Your Dijkstra's implementation shows a 10× runtime increase when you only 5× the input size (vertices and edges). What is the most likely implementation problem, and how would you confirm it?

Answer: The most likely problem is that your "priority queue" is performing a linear scan to find the minimum-distance vertex instead of using a true O(log |V|) heap extraction. This would give you O(|V|²) behavior instead of O((|V|+|E|) log |V|). A 5× increase in |V| with O(|V|²) yields a 25× increase — your 10× is between the expected 5–7× (correct heap) and 25× (linear scan), suggesting a partial issue. To confirm: add a counter inside your extractMin operation. If it iterates over all vertices each time (count ≈ |V| per extraction), you have a linear scan. A correct binary heap should touch at most O(log |V|) elements per extraction.

Check 17.2d: You implement Kruskal's algorithm. On a graph with |E| = 50 edges it takes 0.05 ms; on |E| = 500 edges it takes 0.65 ms. The ratio is 13×. The expected ratio for O(|E| log |E|) with a 10× increase in |E| is 10 × (log 500 / log 50) ≈ 10 × 1.59 ≈ 15.9×. Is your measurement reasonable?

Answer: Yes, 13× is reasonable and consistent with O(|E| log |E|). It is slightly below the theoretical 15.9× — this is normal because: (1) constant factors and overhead make the smaller input slightly "slower per unit" than the larger one, (2) Union-Find with path compression makes the union operations nearly O(1) in practice (inverse Ackermann), so the sorting step dominates and the actual growth is close to O(|E| log |E|). A ratio of 13× is well within the expected range and definitely not linear (which would be exactly 10×) or quadratic (which would be 100×).
