Module 17.3 – Analysis Document and Technical Writing

CSLO 3~2–2.5 hours

What Is the Analysis Document?

Deliverable: A 1–2 page PDF that explains your algorithm choices, compares BFS and DFS, provides time/space complexity analysis, and connects your application to a real-world scenario.
Weight: 37.5 points (15% of the Final Project's 250 points).
What the grader looks for: Accurate complexity analysis, clear and specific writing, a justified MST selection, a substantive BFS/DFS comparison, and a concrete real-world connection.

This module walks you through exactly what to write, how to write it well, and what distinguishes strong analysis from weak analysis. By the end, you will have a complete draft of your Analysis Document ready for submission as part of Module 17.4's final package.

Required Document Structure

Organize your Analysis Document into six clearly labeled sections. The grader will look for each one.

  1. Introduction — One paragraph identifying your application, the language you used, and which algorithms you implemented. Set the stage; don't repeat the assignment prompt.
  2. Algorithm Selection: MST — State whether you chose Kruskal's or Prim's. Justify the choice based on your graph representation, graph density, and implementation tradeoffs. Acknowledge when the other algorithm would be preferable.
  3. BFS vs. DFS Comparison — Explain when BFS is preferable and when DFS is preferable, citing specific properties (level-order guarantee, memory profile, topological ordering). State which was more useful in your application and why.
  4. Complexity Analysis — Provide a table listing each algorithm with its time and space complexity. Follow the table with 1–2 paragraphs explaining why each bound holds and referencing your empirical runtime data from Module 17.2.
  5. Real-World Application — Describe one specific scenario your application models. Explain which algorithms apply and why. Be concrete (name the entities, the edges, the weights).
  6. Conclusion — Summarize what you learned, what you would do differently, and one insight about the relationship between theory and practice.

Writing Tips for Technical Documents

Good technical writing is precise, justified, and honest. Use these four principles as a checklist before you finalize each section.

Be Specific

Don't write: "Dijkstra's is fast."

Do write: "Dijkstra's runs in O((|V| + |E|) log |V|) with a binary heap, which is efficient for sparse graphs where |E| = O(|V|)."

Action step: Every time you use a word like "fast," "efficient," or "better," ask yourself: compared to what? measured how? Replace the vague word with a Big-O bound, a runtime measurement, or a specific property.

Justify Choices

Don't write: "I used Prim's algorithm."

Do write: "I chose Prim's over Kruskal's because my graph uses an adjacency list and Prim's with a min-heap naturally iterates over neighbors, achieving O((|V| + |E|) log |V|) without needing to sort all edges first."

Action step: After every statement of what you did, write one sentence starting with "because" or "since" that explains why.

Acknowledge Tradeoffs

Don't write: "Prim's is the best MST algorithm."

Do write: "Kruskal's would be preferable if the graph were given as an edge list, since sorting edges is its dominant operation and Union-Find provides near-constant-time cycle detection."

Action step: For each choice you justify, add one sentence that begins "However…" or "The alternative would be better when…" This shows the grader you understand the full picture.

Connect Theory to Empirical Data

Don't write: "The runtime results matched expectations."

Do write: "My empirical results show a 4.2× increase in Dijkstra's runtime when input size increases 5×, which is consistent with the O((|V| + |E|) log |V|) bound (expected ratio ≈ 5 × log(100)/log(20) ≈ 4.3×)."

Action step: Pull at least one number from your Module 17.2 runtime comparison and show the math connecting it to the theoretical bound.

Strong vs. Weak Analysis: Side-by-Side

Use the buttons below to examine two key sections of the Analysis Document. Study the contrast between weak and strong writing, then model your own document on the strong versions.

MST Algorithm Selection

✗ Weak

"I used Prim's because it was easier to implement. It finds the MST correctly."

Problems: No reference to graph representation, no complexity reasoning, no tradeoff acknowledgment. "Easier" is subjective and unexplained.

✓ Strong

"I chose Prim's algorithm because my application stores graphs as adjacency lists. Prim's with a student-implemented min-heap achieves O((|V| + |E|) log |V|) by iterating over each vertex's neighbor list and extracting the minimum-weight crossing edge from the heap. Kruskal's would require sorting all |E| edges first — O(|E| log |E|) — which is comparable asymptotically but would not leverage my adjacency list structure. However, for very sparse graphs given as edge lists, Kruskal's with Union-Find would avoid building an adjacency structure entirely."

BFS vs. DFS Comparison

✗ Weak

"BFS uses a queue and DFS uses a stack. They both visit all vertices. BFS goes level by level and DFS goes deep first."

Problems: Only describes mechanics, not applicability. Doesn't answer when each is preferable or connect to the student's application.

✓ Strong

"BFS is preferable when finding the shortest path in an unweighted graph because it explores vertices in level-order, guaranteeing that the first time a vertex is discovered, it is via the minimum number of edges. In my campus network scenario, BFS identifies the fewest-hop route between two buildings. DFS is preferable for topological sorting because its post-order completion times naturally encode dependency ordering — I used DFS-based topological sort to determine a valid course registration sequence in my real-world scenario extension. DFS also uses O(|V|) stack space in the worst case versus BFS's O(|V|) queue space, but DFS's recursive form can overflow the call stack on very deep graphs, which is why I implemented both recursive and iterative versions."

Real-World Scenario Ideas

Your Analysis Document must describe one real-world scenario and explain how your application models it. Choose a scenario that lets you discuss at least two of your implemented algorithms. Here are five curated options:

Campus Wi-Fi Network Design Vertices = buildings; edges = possible cable routes; weights = installation cost. Find the cheapest way to connect all buildings. Primary: MST (Prim's/Kruskal's) · Supporting: BFS (connectivity check)
Social Network Analysis Vertices = users; edges = friendships (unweighted). Find friend-of-friend suggestions and detect communities. Primary: BFS (distance-2 neighbors) · Supporting: DFS (connected components)
Course Prerequisite Planning Vertices = courses; directed edges = prerequisite relationships. Determine a valid semester-by-semester schedule. Primary: Topological Sort (DFS-based) · Supporting: BFS (longest path = minimum semesters)
Emergency Evacuation Routing Vertices = intersections; edges = road segments; weights = travel time. Find fastest routes from any location to the nearest hospital. Primary: Dijkstra's (shortest path) · Supporting: BFS (unweighted fallback)
Internet Packet Routing Vertices = routers; edges = links; weights = latency. Minimize total latency for backbone infrastructure and find lowest-latency paths. Primary: Dijkstra's + MST · Supporting: BFS (hop-count metric)
Tip: Pick the scenario that best matches your implementation. If you chose Prim's MST, the campus Wi-Fi or Internet routing scenarios let you naturally discuss why MST minimizes total cost. If your application emphasizes traversal, the social network or prerequisite scenarios work well.
