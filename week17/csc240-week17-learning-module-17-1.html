Integration testing matters because the Final Project rubric evaluates Functionality (50% = 125 pts) holistically — the grader will run your application end-to-end, not test each function in isolation. Catching integration bugs now, before submission, is the single highest-value activity you can do this week.

End-to-End Integration Testing Strategy

Once each component passes its individual checks, test the complete data flow in a single execution. Integration testing verifies that components communicate correctly — that the graph your Builder creates is the same graph your algorithms traverse.

The Integration Pipeline

Your application should support this end-to-end flow:

🔗 Data Flow: Single Execution Path
  1. Graph Input → Build graph using Graph Builder (add vertices, add weighted edges)
  2. Construction → Store in adjacency list (and/or matrix); display graph properties
  3. Traversal → Run BFS and DFS from a specified source; output traversal orders
  4. Advanced Algorithms → Run MST (output edges + total weight); Run Dijkstra's (output shortest paths)
  5. Tree Structure Active → During step 4, your student-built min-heap/BST performs insert/extract operations
  6. Output → All results displayed; runtime recorded

Verifying Tree Structure Is Actually Called

A common mistake: the tree-based structure exists in the code but the algorithm bypasses it (e.g., using a simple linear scan instead of the heap). To verify meaningful use:

  • Add temporary print/log statements inside your heap's insert() and extractMin() methods.
  • Run Dijkstra's (or Prim's) on your small test graph.
  • Confirm you see insert/extract output for every relaxation step.
  • Remove or disable the print statements before final submission (or guard them behind a DEBUG flag).

Debugging Tips

BFS/DFS Traversal Order Seems Wrong

  • Check your visited-set logic: are you marking a vertex as visited when you enqueue/push it, or when you dequeue/pop it? For BFS, mark on enqueue to avoid duplicates in the queue.
  • Check your adjacency list ordering: neighbor processing order affects traversal output. If your expected output assumes alphabetical/numerical order, sort your neighbor lists.
  • For iterative DFS: remember that a stack reverses the processing order compared to recursive DFS (last-pushed neighbor is processed first).

MST Weight Is Incorrect

  • Verify you are using edge weights, not vertex labels or indices.
  • For undirected graphs, ensure you are not double-counting edges (each undirected edge appears in both adjacency lists).
  • For Kruskal's: confirm your Union-Find correctly detects cycles (check that find(u) != find(v) before adding edge).
  • For Prim's: confirm you update the key/priority when a cheaper edge to a vertex is found (decrease-key operation).

Dijkstra's Produces Wrong Distances

  • Check your priority queue update logic: if using lazy deletion (re-inserting with new priority), make sure you skip already-finalized vertices when you extract them.
  • Verify initialization: distance to source = 0, all others = ∞ (use a large sentinel value like Integer.MAX_VALUE).
  • Confirm you are relaxing all neighbors of the extracted vertex, not just one.

Tree Structure Not Being Used

  • Trace through your algorithm with print statements inside insert() and extractMin().
  • If you see no output, your algorithm is likely using a different mechanism (e.g., linear scan of a distance array). Refactor to route through your heap.
  • Common mistake: importing a library priority queue alongside your custom one and accidentally calling the library version.

General Strategies

  • Small-graph debugging: Use a 4–5 vertex graph where you can trace every step by hand. Compare your hand trace to your program's output line by line.
  • Regression testing: After fixing a bug, re-run all previous test cases to make sure the fix didn't break something else.
  • Isolate the problem: If the integrated pipeline fails, test each component with hardcoded input to determine which component introduced the error.

Self-Check: Integration Readiness

These ungraded checks help you confirm your project is integration-ready. Work through each one and use the feedback to guide your next steps.

Check 17.1a: Run your application on the 5-vertex sample graph from PE-09. Does BFS from vertex 0 produce the same traversal order as your PE-09 submission?

✅ Great! Matching output confirms your Graph Builder is feeding the same structure to your BFS as PE-09 did. Proceed to the next check.
🔍 Investigate: A discrepancy means your Graph Builder may store neighbors in a different order, or your BFS visited-set logic changed. Compare your adjacency list output to PE-09's expected input. Also check whether you mark visited on enqueue vs. dequeue.
⏳ Action needed: This is the most important verification step. Run the test now — it takes under 2 minutes and can reveal integration bugs immediately.

Check 17.1b: Run your MST algorithm on a graph where you can compute the MST by hand (e.g., 4 vertices, 5 edges with distinct weights). Does your output match your hand calculation?

✅ Correct! Your MST algorithm is producing valid output. (Note: if edge weights are distinct, the MST is unique. If some weights are equal, multiple valid MSTs may exist.)
⚠️ Possibly OK: If edge weights are not all distinct, multiple valid MSTs exist with the same total weight. Verify that your output forms a spanning tree (connected, |V| − 1 edges) with the minimum total weight.
🐛 Bug detected: Check for double-counting undirected edges, incorrect Union-Find cycle detection (Kruskal's), or missing decrease-key updates (Prim's). See the Debugging Tips section above.

Check 17.1c: Add print statements inside your tree-based structure's insert() and extractMin() (or equivalent) methods. Run Dijkstra's on your small test graph. Do you see insert/extract output for each relaxation step?

✅ Excellent! Your tree-based structure is meaningfully integrated. This directly satisfies the Tree-Based Structure Integration rubric criterion (15% = 37.5 pts). Remove or disable debug prints before submission.
🚨 Critical issue: Your algorithm is bypassing the tree structure. Check if you accidentally imported a library priority queue, or if your code uses a linear scan of the distance array instead of extracting from the heap. This must be fixed — it affects 37.5 points.
🔍 Investigate: You may be inserting into the heap but not extracting from it (or vice versa). Trace through Dijkstra's pseudocode: every iteration should extract the minimum, and every successful relaxation should insert (or decrease-key). Count expected operations for your test graph and compare.

Check 17.1d: Test on a disconnected graph with 3 components. Does your connected-components algorithm correctly report 3 components? Does your MST algorithm handle this gracefully?

✅ Robust! Handling disconnected graphs gracefully demonstrates mature error handling and earns full marks in the Functionality category for edge-case testing.
⚠️ Partial fix needed: Your MST algorithm assumes a connected graph. Add a check: if the graph has multiple components, either report a minimum spanning forest (run MST on each component) or output a clear message like "Graph is disconnected — MST does not exist for the full graph." Do not let it crash.
🐛 Fix needed: Your connected-components algorithm likely has a bug in its "restart BFS/DFS from unvisited vertices" logic. Review your Week 13 module content on connected components. Ensure you iterate over all vertices and start a new traversal for each unvisited vertex.
