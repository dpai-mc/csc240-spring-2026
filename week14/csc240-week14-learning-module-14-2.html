1 Opening Context

📌 Where We Are

In Module 14.1 you learned that a Minimum Spanning Tree (MST) is a subset of edges that connects every vertex of a weighted, connected, undirected graph with the smallest possible total weight — using exactly |V| − 1 edges and no cycles. You also met the Cut Property: the lightest edge crossing any cut must belong to some MST.

Module 14.1 introduced the greedy paradigm for MST construction. Now we put it into action with Kruskal's algorithm — the edge-centric approach. Kruskal's considers edges globally, from cheapest to most expensive, and adds each edge as long as it doesn't create a cycle.

To make that "does it create a cycle?" check lightning-fast, we'll learn a purpose-built data structure called Union-Find (also known as Disjoint Set Union, or DSU). Union-Find is the engine that makes Kruskal's efficient, and it appears throughout computer science — from network connectivity to image segmentation.

2 Kruskal's Algorithm

2.1 The Big Idea

Kruskal's algorithm builds an MST by repeatedly choosing the globally cheapest edge that does not form a cycle with edges already selected. It is "edge-centric" because it processes a sorted list of all edges, rather than growing a tree from a single source vertex.

💡 Greedy Choice

Kruskal's greedy rule: Among all edges not yet considered, pick the one with the smallest weight. Add it to the MST if it connects two different components (i.e., it does not create a cycle). Skip it otherwise.

2.2 Algorithm Steps

  1. Sort all edges by weight (non-decreasing).
  2. Initialize each vertex as its own component (forest of single-node trees).
  3. Iterate through the sorted edges. For each edge (u, v, w):
    • If u and v are in different components → add the edge to the MST and merge the two components.
    • If u and v are in the same component → skip (adding it would create a cycle).
  4. Stop when the MST has exactly |V| − 1 edges.

2.3 Pseudocode

KRUSKAL(G = (V, E, w)):
    MST ← ∅
    for each vertex v ∈ V:
        MAKE-SET(v)

    sort E by weight w in non-decreasing order

    for each edge (u, v, w) in sorted E:
        if FIND(u) ≠ FIND(v):        // u and v in different components?
            MST ← MST ∪ {(u, v, w)}
            UNION(u, v)              // merge components
            if |MST| == |V| − 1:
                break                // MST is complete

    return MST
🔗 Week 13 Connection

Recall from Week 13 that adding an edge between two vertices that are already in the same connected component creates a cycle. Kruskal's algorithm avoids cycles by checking whether two vertices share a component — exactly the connectivity / cycle-detection reasoning you practiced last week.

2.4 Why Does It Work?

Correctness follows from the Cut Property (Module 14.1). Each time Kruskal's selects an edge (u,v), consider the cut that separates the component containing u from everything else. The selected edge is the lightest edge crossing that cut (because we process edges in sorted order and all lighter cross-edges were already added or would have been cross-edges of a different cut). The Cut Property guarantees this edge belongs to some MST.

3 Union-Find (Disjoint Set Union)

Kruskal's pseudocode relies on three operations — MAKE-SET, FIND, and UNION. Together they form the Union-Find data structure, which maintains a collection of disjoint sets and supports fast merging and membership queries.

Three Core Operations

OperationPurposeKruskal's Usage
MAKE-SET(x)Create a new set containing only x.Initialize each vertex as its own component.
FIND(x)Return the representative (root) of the set containing x.Determine which component a vertex belongs to.
UNION(x, y)Merge the sets containing x and y into one set.Merge two components when a safe edge is added.

Two vertices are in the same component if and only if FIND(u) == FIND(v).

Parent-Array Representation

Each element points to a parent. The root of a set points to itself — it is the representative.

parent[] array, indexed by vertex
Initially: parent[v] = v for every vertex v

FIND follows parent pointers up to the root:

FIND(x):
    while parent[x] ≠ x:
        x ← parent[x]
    return x

UNION (naïve) makes one root point to the other:

UNION(x, y):
    rootX ← FIND(x)
    rootY ← FIND(y)
    if rootX ≠ rootY:
        parent[rootX] ← rootY
⚠️ Problem

Without optimizations, trees can degenerate into long chains, making FIND take O(n) time. We fix this with union by rank and path compression.

Union by Rank

Idea: Always attach the shorter tree under the taller tree's root. This keeps trees shallow.

We maintain a rank[] array (an upper bound on tree height).

MAKE-SET(x):
    parent[x] ← x
    rank[x] ← 0

UNION(x, y):
    rootX ← FIND(x)
    rootY ← FIND(y)
    if rootX == rootY: return      // already same set

    if rank[rootX] < rank[rootY]:
        parent[rootX] ← rootY
    else if rank[rootX] > rank[rootY]:
        parent[rootY] ← rootX
    else:
        parent[rootY] ← rootX
        rank[rootX] ← rank[rootX] + 1

With union by rank alone, tree height is at most O(log n), so FIND runs in O(log n).

Path Compression

Idea: During FIND, make every node on the path point directly to the root. Future queries on these nodes become O(1).

FIND(x):
    if parent[x] ≠ x:
        parent[x] ← FIND(parent[x])   // recursive path compression
    return parent[x]

Visually, path compression "flattens" the tree:

Before FIND(D):        After FIND(D):
    A                      A
    |                    / | \
    B                   B  C  D
    |
    C
    |
    D

Every node that was visited during the FIND now points directly to root A.

Combined Performance

When both union by rank and path compression are used together:

⚡ Key Result

Any sequence of m FIND/UNION operations on n elements takes O(m · α(n)) total time, where α(n) is the inverse Ackermann function.

α(n) grows incredibly slowly — for all practical values of n (even n = 1080, the estimated number of atoms in the observable universe), α(n) ≤ 5. So each operation is effectively O(1) amortized.

📝 Exam Note

You will not be asked to prove the O(α(n)) bound. You should be able to state it and explain that it makes Union-Find operations effectively constant time.

5 Time Complexity Analysis

Let's break down the cost of each phase of Kruskal's algorithm.

PhaseOperationCost
1. Initialize|V| calls to MAKE-SETO(|V|)
2. Sort edgesComparison-based sort of |E| edgesO(|E| log |E|)
3. Process edgesUp to |E| calls to FIND + up to |V| − 1 calls to UNIONO(|E| · α(|V|))

The sorting step dominates, so the overall complexity is:

TKruskal = O(|E| log |E|)

For a simple graph, |E| ≤ |V|(|V|−1)/2 < |V|². Therefore:

log |E| ≤ log(|V|²) = 2 log |V|

Since constant factors are absorbed by Big-O:

O(|E| log |E|) = O(|E| · 2 log |V|) = O(|E| log |V|)

Both forms are used interchangeably in the literature. For this course, either is acceptable.

Phase 3 costs O(|E| · α(|V|)). Since α(|V|) ≤ 5 for all practical |V|, this is effectively O(|E|) — which is strictly less than O(|E| log |E|) for any non-trivial graph. The sort step is always the bottleneck.

🧠 Rule of Thumb

If the graph is sparse (few edges relative to vertices) or you already have an edge list, Kruskal's is typically the better choice. If the graph is dense, Prim's (Module 14.3) often wins.

6.2 Java Code Walkthrough

Below is a clean Java implementation. Read through it carefully — comments map each section back to the pseudocode.

class UnionFind {
    private int[] parent;
    private int[] rank;

    // MAKE-SET for all n elements (0-indexed)
    public UnionFind(int n) {
        parent = new int[n];
        rank   = new int[n];
        for (int i = 0; i < n; i++) {
            parent[i] = i;   // each element is its own root
            rank[i]   = 0;
        }
    }

    // FIND with path compression
    public int find(int x) {
        if (parent[x] != x) {
            parent[x] = find(parent[x]);  // path compression
        }
        return parent[x];
    }

    // UNION by rank; returns true if merge happened
    public boolean union(int x, int y) {
        int rootX = find(x);
        int rootY = find(y);
        if (rootX == rootY) return false;  // already same set

        if (rank[rootX] < rank[rootY]) {
            parent[rootX] = rootY;
        } else if (rank[rootX] > rank[rootY]) {
            parent[rootY] = rootX;
        } else {
            parent[rootY] = rootX;
            rank[rootX]++;
        }
        return true;
    }
}
import java.util.*;

class Edge implements Comparable<Edge> {
    int u, v, weight;
    Edge(int u, int v, int weight) {
        this.u = u; this.v = v; this.weight = weight;
    }
    @Override
    public int compareTo(Edge other) {
        return Integer.compare(this.weight, other.weight);
    }
}

public class KruskalMST {

    public static List<Edge> kruskal(int V, List<Edge> edges) {
        Collections.sort(edges);          // Step 1: sort by weight
        UnionFind uf = new UnionFind(V);  // Step 2: make-set for all
        List<Edge> mst = new ArrayList<>();

        for (Edge e : edges) {            // Step 3: iterate sorted edges
            if (uf.union(e.u, e.v)) {     // returns true if merged
                mst.add(e);
                if (mst.size() == V - 1) break;  // Step 4: done
            }
        }
        return mst;
    }

    public static void main(String[] args) {
        // Example: 7 vertices (0=A,1=B,2=C,3=D,4=E,5=F,6=G)
        List<Edge> edges = Arrays.asList(
            new Edge(0,3,5),  new Edge(2,4,5),  new Edge(3,5,6),
            new Edge(0,1,7),  new Edge(1,4,7),  new Edge(1,2,8),
            new Edge(4,5,8),  new Edge(1,3,9),  new Edge(4,6,9),
            new Edge(5,6,11), new Edge(3,4,15)
        );
        List<Edge> mst = kruskal(7, edges);

        int total = 0;
        System.out.println("MST Edges:");
        for (Edge e : mst) {
            System.out.printf("  (%d, %d) weight %d%n", e.u, e.v, e.weight);
            total += e.weight;
        }
        System.out.println("Total MST weight: " + total);
    }
}

✓ Module 14.2 Summary

📋 Key Takeaways
  • Kruskal's algorithm builds an MST by sorting all edges and greedily adding the cheapest edge that doesn't create a cycle.
  • The Union-Find data structure supports MAKE-SET, FIND, and UNION to efficiently track connected components.
  • Union by rank keeps trees balanced; path compression flattens trees during FIND. Together they achieve O(α(n)) amortized per operation — effectively constant.
  • Kruskal's overall time is O(|E| log |E|), dominated by the sort step.
  • Kruskal's is ideal for sparse graphs and edge-list representations.
