PE-09: Graph Algorithms Implementation

Programming Exercise — CSC 240 Discrete Structures, Week 14
Points: 30 Type: Programming Exercise Language: Java or C++

Overview & Objectives

PE-09 is the capstone programming exercise for the graph algorithms unit. You will implement four fundamental graph algorithms—BFS, DFS, one MST algorithm, and Dijkstra’s shortest path—on adjacency list representations, then compare their performance on at least two different graph structures.

CSLO Alignment
CSLO 3: Implement graph/tree traversal methods in a programming language.
Supporting: CSLO 4: Explain the relationship between mathematical induction and recursion/recurrence equations, and their relationship to computing systems. (DFS recursion connects to induction from Weeks 6 and 11.)

What This Exercise Assesses

  • Your ability to build a correct adjacency list representation supporting weighted, directed graphs.
  • Correct implementation of BFS and DFS (both recursive and iterative) with traversal-order output.
  • Implementation of one MST algorithm (Kruskal’s or Prim’s)—your choice, with written justification.
  • Implementation of Dijkstra’s algorithm with distance output and path reconstruction.
  • Empirical performance comparison on at least two graph structures (e.g., sparse vs. dense, or small vs. large).
Final Project Scaffolding: PE-09 directly scaffolds the Final Project (assigned this week). The algorithms you implement here map to Final Project Components 2 (Traversal Algorithms, 50 pts) and 3 (Advanced Graph Algorithms, 75 pts). Treat PE-09 as a working prototype you can extend into your Final Project.

Scenario: NetGraph Analysis Library

You are a software engineer at NetGraph Solutions, a startup building a network infrastructure analysis tool. Your team needs a reusable graph analysis library that network engineers will use to:

  1. Map network topology — represent routers, switches, and connections as a weighted, directed graph stored in adjacency lists.
  2. Discover reachable devices — use BFS (level-by-level hop counting) and DFS (deep path exploration) to traverse the network from any source device.
  3. Design minimum-cost cabling — compute a Minimum Spanning Tree so the engineering team can connect all nodes at the lowest total cable cost.
  4. Find fastest routes — run Dijkstra’s algorithm to determine the shortest (lowest-latency) path between any two devices, including the full route.

Your library must be tested on at least two network configurations (graph structures) so the team can verify correctness and compare algorithm performance across different topologies.

Why this matters: Every modern routing protocol (OSPF, IS-IS) and network design tool relies on exactly these algorithms. The code you write in PE-09 mirrors real infrastructure software.

Tasks & Procedure

Complete the four phases below in order. Each phase builds on the previous one. For every phase, your code must compile, run, and produce the specified output.

Phase 1

Graph Representation — Adjacency List

Build the core data structure that all subsequent algorithms will operate on.

Requirements

  • Implement a weighted, directed graph using an adjacency list. (Undirected graphs are modeled by adding edges in both directions.)
  • Support the following operations:
    • addVertex(v)
    • addEdge(u, v, weight) — directed edge from u to v
    • getNeighbors(v) — returns list of (neighbor, weight) pairs
    • getVertexCount() and getEdgeCount()
    • printGraph() — human-readable adjacency list output

Input / Output Specification

  • Input: Your driver or test code constructs at least two graphs: one sparse (e.g., |E| ≈ |V|) and one dense (e.g., |E| ≈ |V|² / 2).
  • Output: Print the adjacency list for each graph, showing each vertex followed by its neighbor list with weights.

Testing Guidance

  • Verify edge counts match expected values.
  • For undirected graphs, confirm each edge appears in both directions.
  • Test with a single-vertex graph (edge case) and a disconnected graph.
Hints & Tips
  • In Java, use Map<Integer, List<int[]>> or a custom Edge class. In C++, use unordered_map<int, vector<pair<int,int>>>.
  • Store edge weights as integers (latency in ms) to simplify testing. Use values between 1 and 100.
  • Consider creating a helper method addUndirectedEdge(u, v, w) that calls addEdge in both directions.
  • This adjacency list is the same representation required for Final Project Component 1 (Graph Builder, 50 pts)—design it well now and you can reuse it directly.
Phase 2

BFS and DFS Traversals

Implement both traversal algorithms on your adjacency list from Phase 1. You studied these in Week 13—now you implement them.

Requirements

  • BFS: Queue-based, level-by-level traversal from a given source vertex. Output the traversal order and the level (distance in hops) of each vertex.
  • DFS — Recursive: Implement DFS using the call stack. Output discovery order.
  • DFS — Iterative: Implement DFS using an explicit stack. Output discovery order.
  • All three must use a visited set/array to avoid revisiting vertices.

Input / Output Specification

  • Input: A graph (from Phase 1) and a source vertex.
  • Output (BFS): BFS from vertex 0: 0(L0) → 1(L1) → 3(L1) → 2(L2) → 4(L2)
  • Output (DFS): DFS-Recursive from vertex 0: 0 → 1 → 2 → 4 → 3
  • Both recursive and iterative DFS should produce output; note that traversal order may differ due to neighbor processing order—this is acceptable as long as the algorithm is correct.

Testing Guidance

  • Run BFS and DFS on a small graph (5–7 vertices) and verify output by hand-tracing.
  • Test on a disconnected graph—confirm that only the source’s component is traversed.
  • Test on a single-vertex graph.
  • Compare recursive and iterative DFS outputs—they may differ in order but both must visit all reachable vertices exactly once.
Hints & Tips
  • For BFS, use a Queue (Java: LinkedList; C++: queue). Track levels by processing all vertices at the current level before moving to the next, or store (vertex, level) pairs in the queue.
  • For iterative DFS, use a Stack (Java: ArrayDeque; C++: stack). Push neighbors in reverse order if you want to match the recursive DFS order.
  • Mark vertices as visited when you enqueue/push them, not when you dequeue/pop—this prevents duplicates in the queue/stack.
  • Review the pseudocode from Week 13 Modules 13.1 (BFS) and 13.2 (DFS).
Phase 3

MST Algorithm — Kruskal’s or Prim’s (Your Choice)

Implement one Minimum Spanning Tree algorithm. You choose which one—but you must justify your choice in the Observations section below.

Option A: Kruskal’s Algorithm

  • Sort all edges by weight.
  • Use a Union-Find (Disjoint Set Union) data structure with union by rank and path compression.
  • Process edges in order; add an edge to the MST if its endpoints are in different sets.
  • Time complexity: O(|E| log |E|) dominated by sorting.

Option B: Prim’s Algorithm

  • Start from an arbitrary vertex; grow the MST by always adding the cheapest edge connecting the MST to a non-MST vertex.
  • Use a priority queue (min-heap) keyed on edge weight.
  • Time complexity: O((|V| + |E|) log |V|) with a binary heap.

Input / Output Specification

  • Input: A connected, weighted, undirected graph (from Phase 1).
  • Output:
    • List of MST edges with weights, e.g., (0-1, w=4), (1-3, w=2), ...
    • Total MST weight.
    • Number of edges in the MST (should be |V| − 1).

Testing Guidance

  • Verify MST edge count equals |V| − 1.
  • Hand-compute the MST on your small test graph and compare.
  • Test on a graph with duplicate edge weights—MST may not be unique, but total weight must be minimal.
  • Test on a graph that is already a tree—MST should be the graph itself.
Hints & Tips
  • Kruskal’s: Implement Union-Find as a separate class/struct. The find() method should use path compression; the union() method should use union by rank. This keeps the amortized cost nearly O(1) per operation.
  • Prim’s: In Java, use PriorityQueue. In C++, use priority_queue with greater<> for a min-heap. Since standard library priority queues don’t support decreaseKey, you can insert duplicate entries and skip stale ones (lazy deletion).
  • Kruskal’s tends to be simpler to implement and performs well on sparse graphs. Prim’s is a better choice for dense graphs and shares structural similarity with Dijkstra’s (Phase 4).
  • Your MST algorithm choice and justification will also be relevant for the Final Project Analysis Document (Component 5, 25 pts).
Phase 4

Dijkstra’s Shortest Path Algorithm

Implement Dijkstra’s algorithm to find single-source shortest paths in a weighted graph with non-negative edge weights.

Requirements

  • Maintain a distance array d[v] initialized to ∞ for all vertices (source initialized to 0).
  • Use a priority queue (min-heap) to always process the vertex with the smallest tentative distance.
  • Perform edge relaxation: if d[u] + w(u,v) < d[v], update d[v] and record u as v’s predecessor.
  • After the algorithm completes, reconstruct the shortest path from source to each reachable vertex using predecessor pointers.

Input / Output Specification

  • Input: A weighted graph (from Phase 1) and a source vertex. All edge weights must be non-negative.
  • Output:
    • Distance table: Vertex 0: dist=0 | Vertex 1: dist=4 | Vertex 2: dist=7 | ...
    • Shortest path reconstruction: Path to vertex 4: 0 → 1 → 3 → 4 (total: 11)
    • If a vertex is unreachable, output dist=∞, no path.

Testing Guidance

  • Hand-trace Dijkstra’s on your small test graph and compare distances and paths.
  • Test with a disconnected graph—unreachable vertices should show ∞.
  • Test with a graph where the direct edge is not the shortest path (requires going through intermediate vertices).
  • Verify that path reconstruction produces a valid path whose edge weights sum to the reported distance.
Hints & Tips
  • Dijkstra’s is structurally similar to Prim’s—both use a min-heap and process vertices greedily. The key difference: Prim’s keys on edge weight to MST; Dijkstra’s keys on tentative distance from source.
  • Use the same lazy deletion approach as Prim’s: when you pop a vertex from the priority queue, check if it has already been finalized (visited). If so, skip it.
  • For path reconstruction, store a predecessor[] array. After the algorithm finishes, trace back from the destination to the source using predecessors.
  • Use Integer.MAX_VALUE (Java) or INT_MAX (C++) for infinity—but be careful with overflow when adding weights. Use Long or check before adding.
  • If you implemented Prim’s in Phase 3, you can refactor that code—change the priority queue key from edge weight to cumulative distance and add predecessor tracking.

Common Pitfalls

Expand to view common implementation mistakes

General

  • Forgetting to reset the visited array between algorithm runs. If you run BFS, then DFS on the same graph without resetting, DFS will think all vertices are already visited.
  • Off-by-one errors in vertex numbering. Decide whether vertices are 0-indexed or 1-indexed and be consistent everywhere.
  • Not handling disconnected graphs. BFS/DFS from a single source will not visit all vertices in a disconnected graph—this is correct behavior, not a bug.

BFS / DFS

  • Marking visited on pop instead of push. If you mark a vertex as visited when you dequeue/pop it (instead of when you enqueue/push it), you may add duplicate entries to the queue/stack, wasting time and potentially producing incorrect output.
  • Iterative DFS producing different order than recursive DFS. This is normal—the explicit stack processes neighbors in a different order than the call stack. Both are valid DFS traversals.

Kruskal’s Algorithm

  • Not implementing path compression in Union-Find. Without path compression, find() degrades to O(|V|) per call, making the overall algorithm much slower.
  • Forgetting to stop after |V| − 1 edges. Once the MST has |V| − 1 edges, you can stop processing—no more edges are needed.

Prim’s Algorithm

  • Not handling the decreaseKey limitation. Standard library priority queues typically do not support decreaseKey. Use lazy deletion: insert a new entry with the updated key and skip stale entries when popping.
  • Starting from a vertex not in the graph. Always validate the starting vertex.

Dijkstra’s Algorithm

  • Integer overflow when adding distances. If d[u] is set to Integer.MAX_VALUE and you compute d[u] + w(u,v), the result overflows. Check for infinity before adding, or use long.
  • Using Dijkstra’s on graphs with negative edge weights. Dijkstra’s assumes non-negative weights. If any weight is negative, the algorithm may produce incorrect results.
  • Forgetting to track predecessors for path reconstruction. The distance array alone tells you the shortest distance, but without predecessor pointers you cannot reconstruct the actual path.
  • Not skipping already-finalized vertices. When using lazy deletion with the priority queue, you must check whether a popped vertex has already been processed. If it has, skip it.
