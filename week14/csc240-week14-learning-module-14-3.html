Module 14.3 – Prim's Algorithm & MST Comparison

The vertex-centric approach to Minimum Spanning Trees
📅 Week 14 ⏱️ Estimated Time: 60–75 minutes 📖 Module 3 of 3

Quick Recap: Kruskal's Algorithm (Module 14.2)

In Module 14.2 you learned Kruskal's algorithm, an edge-centric greedy strategy for building a Minimum Spanning Tree (MST):

  1. Sort every edge by weight.
  2. Process edges lightest-first; add an edge if it connects two different components (Union-Find cycle check).
  3. Stop after $$|V| - 1$$ edges.

Kruskal's works globally—it doesn't care where in the graph the next cheapest edge lives. Now we shift to a fundamentally different perspective.

Introducing Prim's Algorithm — Vertex-Centric Growth

Prim's algorithm grows the MST from a single starting vertex, one vertex at a time, always choosing the lightest edge that crosses the boundary between "already in the tree" and "not yet in the tree." Think of it as a greedy frontier expansion.

💡 Key Intuition
If you have ever seen Breadth-First Search (BFS), Prim's feels similar—but instead of a FIFO queue that explores by hop count, Prim's uses a min-priority queue that explores by edge weight. Swap the queue type, and the traversal strategy changes from "nearest by hops" to "nearest by cost."

Two Algorithms, One Theorem

Both Kruskal's and Prim's are proven correct by the same foundation—the Cut Property (Module 14.1). Every greedy choice each algorithm makes selects the minimum-weight edge crossing some cut, guaranteeing the result is an MST.

🔗 Final Project Connection
Your Final Project Analysis Document (Component 4) requires you to compare two graph algorithms and justify which is more suitable for a given scenario. The Kruskal's-vs-Prim's comparison table in Section 6 of this module is designed to give you a reusable framework for that analysis.

High-Level Idea

  1. Initialize: Pick any start vertex $$s$$. Set its key (cheapest known connection cost) to 0; set every other vertex's key to $$\infty$$. Insert all vertices into a min-priority queue keyed on these values.
  2. Repeat until the queue is empty:
    1. Extract-Min: Remove the vertex $$u$$ with the smallest key. Add $$u$$ (and its connecting edge) to the MST.
    2. Relax neighbors: For every neighbor $$v$$ of $$u$$ still in the queue, if $$w(u,v) < \text{key}[v]$$, then update $$\text{key}[v] \leftarrow w(u,v)$$ and record $$\text{parent}[v] \leftarrow u$$ (this is the decreaseKey operation).
  3. Result: The parent pointers form the MST.
💡 The Greedy Choice
At every step, Prim's selects the lightest edge crossing the cut $$(S, V \setminus S)$$, where $$S$$ is the set of vertices already extracted. By the Cut Property, this edge is safe to add to the MST.

Pseudocode

1function Prim(G, s): 2 for each vertex v ∈ G.V: 3 key[v] ← ∞ 4 parent[v] ← NIL 5 inQueue[v] ← TRUE 6 key[s] ← 0 7 Q ← MinHeap(G.V, key) // Build heap on key values 8 9 while Q ≠ ∅: 10 u ← extractMin(Q) // Vertex with smallest key 11 inQueue[u] ← FALSE 12 13 for each (u, v) ∈ G.Adj[u]: // All neighbors of u 14 if inQueue[v] AND w(u,v) < key[v]: 15 key[v] ← w(u,v) // decreaseKey 16 parent[v] ← u 17 18 return parent // parent[] encodes the MST

Key Data Structures

Arrays maintained during Prim's
ArrayPurpose
key[v]Cheapest known edge weight connecting $$v$$ to the growing MST. Initialized to $$\infty$$; updated by decreaseKey.
parent[v]The vertex in the MST that connects to $$v$$ via the edge of weight $$\text{key}[v]$$. Together, all parent pointers form the MST edges.
inQueue[v]Boolean flag — TRUE if $$v$$ is still in the priority queue (not yet added to the MST).
✅ Remember
key[v] is not the total path cost from the start (that would be Dijkstra's). It is only the weight of the single cheapest edge from any MST vertex to $$v$$.

Why a Min-Heap?

Prim's algorithm repeatedly needs the vertex with the smallest key. A binary min-heap provides this efficiently:

Min-Heap Operations Used by Prim's
OperationWhat Prim's Uses It ForBinary Heap Cost
buildHeapInitialize the queue with all $$|V|$$ vertices$$O(|V|)$$
extractMinGet the next vertex to add to the MST$$O(\log |V|)$$
decreaseKeyUpdate key[v] when a cheaper connecting edge is found$$O(\log |V|)$$

How decreaseKey Works

  1. Locate vertex $$v$$ in the heap (using an index map for $$O(1)$$ lookup).
  2. Replace its key with the new, smaller value.
  3. Bubble up (sift up): swap $$v$$ with its parent while $$v$$'s key is smaller than its parent's key.

This restores the min-heap property in $$O(\log |V|)$$ time because the heap has height $$\lfloor \log_2 |V| \rfloor$$.

💡 decreaseKey Is the Critical Operation
Without decreaseKey, you would need to remove and reinsert a vertex every time you find a cheaper edge—losing the efficiency advantage. The ability to update a key in place and restore heap order is what makes the binary-heap implementation of Prim's run in $$O((|V|+|E|) \log |V|)$$.

Alternative: Fibonacci Heap

A Fibonacci heap achieves amortized $$O(1)$$ decreaseKey, which drops Prim's total to $$O(|E| + |V| \log |V|)$$. In practice, the constant factors are large, so binary heaps are used more often for moderate graph sizes. For this course, we focus on the binary min-heap implementation.

🔗 Final Project Connection (Component 4)
When your Final Project asks you to analyze an algorithm's data-structure choices, this is a perfect example: the choice of heap type directly affects the asymptotic complexity. Documenting why a binary heap is chosen over a Fibonacci heap (or vice versa) is the kind of justified trade-off analysis the rubric rewards.

Deriving the Total

Summing all contributions:

$$T(V,E) = O(|V|) + O(|V| \log |V|) + O(|E| \log |V|) = O\bigl((|V| + |E|) \log |V|\bigr)$$

Since a connected graph has $$|E| \geq |V| - 1$$, the $$|E| \log |V|$$ term dominates, so we often write simply:

$$O(|E| \log |V|)$$

💡 Same Big-O as Kruskal's!
Kruskal's is $$O(|E| \log |E|)$$. Since $$\log |E| = O(\log |V|^2) = O(2 \log |V|) = O(\log |V|)$$, both algorithms have the same asymptotic complexity for connected graphs. The practical difference lies in constant factors and data-structure overhead, which we explore in the comparison table.

Dense Graphs: Adjacency Matrix Variant

For dense graphs where $$|E| \approx |V|^2$$, a simpler implementation avoids the heap entirely:

  • Store key[] in a plain array.
  • Each extractMin scans the array in $$O(|V|)$$.
  • decreaseKey is $$O(1)$$ (just update the array).
  • Total: $$|V|$$ extractions × $$O(|V|)$$ + $$|E|$$ relaxations × $$O(1)$$ = $$O(|V|^2)$$.

For dense graphs, $$O(|V|^2)$$ beats $$O(|E| \log |V|) = O(|V|^2 \log |V|)$$, making the array version preferable.

Decision Flowchart (Text Version)

✅ When to Choose Which
  1. Is the graph sparse ($$|E|$$ close to $$|V|$$)? → Lean toward Kruskal's.
  2. Is the graph dense ($$|E|$$ close to $$|V|^2$$)? → Lean toward Prim's (array variant).
  3. Is the graph potentially disconnected and you need a spanning forest? → Kruskal's handles this natively.
  4. Do you already have an adjacency list and a priority queue library? → Prim's avoids building an edge list and sorting.
  5. Tie? Both produce correct MSTs with the same asymptotic complexity. Choose based on available data structures and implementation comfort.

Key Takeaways

  • Prim's algorithm is a vertex-centric greedy MST algorithm that grows a tree from a single source by always adding the cheapest crossing edge.
  • It relies on a min-priority queue with extractMin and decreaseKey as its critical operations.
  • With a binary min-heap, Prim's runs in $$O((|V|+|E|) \log |V|)$$; with a plain array on dense graphs, it runs in $$O(|V|^2)$$.
  • Kruskal's and Prim's have the same asymptotic complexity but differ in approach, data structures, and practical suitability depending on graph density and connectivity.
  • Both algorithms are justified by the Cut Property (Module 14.1).
