Dijkstra's Shortest Path Algorithm

From total network cost (MST) to optimal individual routes — the single-source shortest path problem, the relaxation operation, and path reconstruction.

1 · Opening Context: MST vs. Shortest Path

In Modules 14.2 and 14.3 you learned two algorithms — Kruskal's and Prim's — that solve the Minimum Spanning Tree problem. An MST connects every vertex in a network at the lowest total cost. Think of it as an infrastructure planner's question: "What is the cheapest set of cables that keeps every office connected?"

Now change the question. You open a GPS app and ask: "What is the shortest route from my house to the airport?" You don't care about connecting every intersection in the city — you care about one source and the cheapest individual path to a destination. That is the single-source shortest path (SSSP) problem, and the classic solution for graphs with non-negative weights is Dijkstra's algorithm.

🔗 Structural Connection to Prim's
Dijkstra's algorithm looks remarkably similar to Prim's: both use a priority queue, both process one vertex per iteration, and both update keys of neighboring vertices. The only difference is what the key represents — Prim's keys on edge weight to the MST, while Dijkstra's keys on tentative distance from the source. Keep this parallel in mind throughout the module.

MST (Prim's / Kruskal's)

  • Goal: Connect all vertices
  • Minimizes: Total edge weight of spanning tree
  • Output: A tree with |V| − 1 edges
  • Analogy: Cheapest cable network

SSSP (Dijkstra's)

  • Goal: Find cheapest path from one source to all others
  • Minimizes: Individual path cost from source
  • Output: Distance array + predecessor array (shortest-path tree)
  • Analogy: GPS turn-by-turn route
⚠️ Key Distinction
The MST of a graph is not the same as the shortest-path tree. An edge that appears in the MST may not appear on any shortest path, and vice versa. These are different optimization problems that happen to share a greedy, priority-queue-based structure.

2 · The Single-Source Shortest Path Problem

Formal Definition

Given a weighted graph G = (V, E) with weight function w: E → ℝ≥0 and a designated source vertex s ∈ V, find, for every vertex v ∈ V:

  1. The shortest-path distance d[v] — the minimum total weight of any path from s to v.
  2. A shortest path itself (the actual sequence of vertices from s to v).

If no path from s to v exists, we define d[v] = ∞.

Two Key Arrays

Distance array d[v]
Stores the current best-known (tentative) distance from the source s to vertex v. Initialized to ∞ for every vertex except the source, where d[s] = 0. When the algorithm finishes, d[v] holds the true shortest-path distance.
Predecessor array prev[v] (also called π[v])
Stores the vertex immediately before v on the current best-known shortest path from s. Initialized to null (or −1) for every vertex. Used to reconstruct the actual shortest path after the algorithm finishes.
🔁 Callback: Prim's Arrays
In Prim's algorithm (Module 14.3), you maintained a key[v] array (minimum edge weight connecting v to the MST) and a parent[v] array (which MST vertex connects to v). Dijkstra's d[v] and prev[v] serve analogous roles but measure distance from the source rather than edge weight to the growing tree.

3 · The Non-Negative Weight Requirement

Dijkstra's algorithm requires all edge weights to be non-negative (w(u,v) ≥ 0 for every edge). This is not just a suggestion — negative weights can cause the algorithm to produce incorrect results. Let's see exactly why.

The Greedy Assumption

Dijkstra's is a greedy algorithm: once a vertex u is extracted from the priority queue (i.e., "finalized"), the algorithm assumes d[u] is the true shortest distance and never revisits u. This assumption is valid only when no future edge can reduce a path cost — which is guaranteed when all weights are non-negative.

Worked Counterexample (3 Vertices)

Consider this directed graph with a negative edge:

(S)──5──▶(A) │ │ └──2──▶(B)──(-4)──▶(A) Edges: S→A (weight 5), S→B (weight 2), B→A (weight -4)
Figure 14.4.2 — A 3-vertex graph where Dijkstra's fails due to a negative edge weight.
Step-by-step: Why Dijkstra's gives the wrong answer
  1. Initialize: d[S] = 0, d[A] = ∞, d[B] = ∞. Priority queue: {S:0}.
  2. Extract S (distance 0). Relax neighbors:
    • d[A] = 0 + 5 = 5
    • d[B] = 0 + 2 = 2
    Priority queue: {B:2, A:5}.
  3. Extract B (distance 2). B is now finalized. Relax neighbors:
    • d[A]: check d[B] + w(B,A) = 2 + (−4) = −2. Since −2 < 5, update d[A] = −2.
    Priority queue: {A:−2}.
  4. Extract A (distance −2). Dijkstra's reports d[A] = −2.

In this tiny example Dijkstra's actually happens to get the right answer (−2), but the problem appears in larger graphs: if a vertex has already been finalized and a later negative edge would reduce its distance, Dijkstra's will not go back and fix it. The algorithm's correctness proof relies on the invariant that finalized distances are optimal — negative edges violate this invariant.

Here is a slightly expanded scenario where the failure is clear: suppose after finalizing A at distance 5, a later path through B discovers d[A] = −2. If A had already been used to relax its neighbors, those relaxations used the wrong distance (5 instead of −2), and the errors propagate.

🚫 Rule
Dijkstra's algorithm is only correct when all edge weights are non-negative. For graphs with negative edges (but no negative cycles), the Bellman-Ford algorithm can be used instead. Bellman-Ford is not assessed in this course, but you should know it exists and when it's needed.

4 · Dijkstra's Algorithm

4.1 · High-Level Idea

  1. Start at the source s. Set d[s] = 0 and d[v] = ∞ for all other vertices.
  2. Put all vertices in a min-priority queue, keyed by their tentative distance d[v].
  3. Repeat until the priority queue is empty:
    1. Extract the vertex u with the smallest d[u]. This vertex is now finalized.
    2. For each neighbor v of u, perform the RELAX operation.
  4. When finished, d[v] contains the shortest distance from s to every reachable vertex, and prev[v] encodes the shortest paths.

4.2 · The RELAX Operation

Relaxation is the core operation that updates distance estimates. It asks: "Can we improve the best-known distance to v by going through u?"

RELAX(u, v, w):
    if d[u] + w(u, v) < d[v]:
        d[v] ← d[u] + w(u, v)
        prev[v] ← u
        decreaseKey(PQ, v, d[v])
Why "relaxation"?
The term comes from the idea of relaxing an overestimate. Initially, d[v] = ∞ is a loose upper bound. Each successful relaxation tightens that bound, bringing it closer to the true shortest distance. The name is standard in algorithm textbooks (Cormen et al., CLRS).
🔗 Connection to Prim's
In Prim's algorithm, the analogous update is: if w(u,v) < key[v]: key[v] ← w(u,v); parent[v] ← u. Notice Prim's compares the edge weight alone, while Dijkstra's compares the accumulated distance plus the edge weight. This single difference is what makes one algorithm solve MST and the other solve SSSP.

4.3 · Full Algorithm Pseudocode

DIJKSTRA(G, w, s):
    // Initialization
    for each vertex v in G.V:
        d[v] ← ∞
        prev[v] ← null
    d[s] ← 0

    PQ ← MinPriorityQueue()
    for each vertex v in G.V:
        PQ.insert(v, d[v])

    // Main loop
    while PQ is not empty:
        u ← PQ.extractMin()          // Finalize vertex with smallest d[u]
        for each neighbor v of u:     // Iterate over adjacency list of u
            RELAX(u, v, w)            // Update d[v] and prev[v] if improved

    return d[], prev[]

💡 Key Insight
If you understand Prim's algorithm, you already understand 90% of Dijkstra's. The only change is in the RELAX/update step: replace the edge weight comparison with an accumulated distance comparison. Everything else — the priority queue, the extract-min loop, the adjacency list iteration — is identical.

Path Reconstruction Using the Predecessor Array

To find the actual shortest path from A to any vertex, trace backward through prev[] from the destination to the source, then reverse.

Example: Shortest path from A to F
  1. Start at F. prev[F] = D.
  2. Go to D. prev[D] = B.
  3. Go to B. prev[B] = A.
  4. A is the source. Stop.

Reverse the chain: A → B → D → F

Verify the cost: w(A,B) + w(B,D) + w(D,F) = 2 + 1 + 3 = 6 = d[F] ✓

Example: Shortest path from A to E
  1. Start at E. prev[E] = B.
  2. Go to B. prev[B] = A.
  3. A is the source. Stop.

Reverse: A → B → E

Verify: 2 + 3 = 5 = d[E] ✓

Path Reconstruction Pseudocode

RECONSTRUCT_PATH(prev[], s, target):
    path ← empty list
    current ← target
    while current ≠ null:
        path.addFirst(current)    // prepend
        current ← prev[current]
    if path[0] = s:
        return path               // valid path found
    else:
        return "no path exists"   // target unreachable from s

6 · Time Complexity Analysis

Operation Counts

Dijkstra's algorithm performs two types of priority queue operations:

  1. extractMin — called once per vertex → |V| times.
  2. decreaseKey — called at most once per edge (each edge triggers at most one relaxation that succeeds) → at most |E| times.

With a Binary Min-Heap

Both extractMin and decreaseKey take O(log |V|) on a binary heap:

T(V,E) = |V| · O(log |V|) + |E| · O(log |V|) = O((|V| + |E|) log |V|)

🔗 Same as Prim's!
This is the same complexity as Prim's algorithm with a binary heap. Both algorithms have the same loop structure; they differ only in what the key represents.

Connection to BFS

Recall from Module 13.1 that BFS finds shortest paths in unweighted graphs in O(|V| + |E|) time using a simple FIFO queue. Dijkstra's generalizes BFS to weighted graphs by replacing the FIFO queue with a priority queue. The priority queue adds a log |V| factor per operation:

AlgorithmGraph TypeQueue TypeTime Complexity
BFSUnweightedFIFO queueO(|V| + |E|)
Dijkstra'sWeighted (non-negative)Min-priority queueO((|V| + |E|) log |V|)

If all edge weights are 1, Dijkstra's algorithm produces the same result as BFS — but with an unnecessary log |V| overhead. Use BFS for unweighted graphs; use Dijkstra's when weights vary.

Advanced note: Fibonacci heap optimization

With a Fibonacci heap, decreaseKey runs in O(1) amortized time, reducing the total complexity to O(|V| log |V| + |E|). This is better for dense graphs where |E| is large. Fibonacci heaps are not covered in this course, but you may encounter them in CSC 340 or algorithms electives.

7.1 · PE-09 Version (Library PriorityQueue)

import java.util.*;

public class Dijkstra {

    // Edge in adjacency list: (neighbor, weight)
    static class Edge {
        int to, weight;
        Edge(int to, int weight) {
            this.to = to;
            this.weight = weight;
        }
    }

    // PQ entry: (vertex, tentative distance)
    static class Node implements Comparable<Node> {
        int vertex, dist;
        Node(int vertex, int dist) {
            this.vertex = vertex;
            this.dist = dist;
        }
        public int compareTo(Node other) {
            return Integer.compare(this.dist, other.dist);
        }
    }

    /**
     * Runs Dijkstra's algorithm from source vertex s.
     * @param adj  adjacency list: adj[u] = list of Edge(v, w)
     * @param s    source vertex
     * @return     int[] d where d[v] = shortest distance from s to v
     *             (also populates prev[] for path reconstruction)
     */
    static int[] dijkstra(List<List<Edge>> adj, int s) {
        int n = adj.size();
        int[] d    = new int[n];
        int[] prev = new int[n];
        Arrays.fill(d, Integer.MAX_VALUE);
        Arrays.fill(prev, -1);
        d[s] = 0;

        // Min-priority queue
        PriorityQueue<Node> pq = new PriorityQueue<>();
        pq.offer(new Node(s, 0));

        boolean[] finalized = new boolean[n];

        while (!pq.isEmpty()) {
            Node cur = pq.poll();          // extractMin
            int u = cur.vertex;

            if (finalized[u]) continue;    // skip stale entries
            finalized[u] = true;

            for (Edge e : adj.get(u)) {    // relax neighbors
                int v = e.to;
                int newDist = d[u] + e.weight;
                if (newDist < d[v]) {      // RELAX
                    d[v] = newDist;
                    prev[v] = u;
                    pq.offer(new Node(v, newDist));
                }
            }
        }
        return d;  // also return prev[] for path reconstruction
    }
}
⚠️ Lazy Deletion Pattern

Java's PriorityQueue does not support a true decreaseKey operation. Instead, we insert a new entry with the updated distance and use a finalized[] boolean array to skip stale (outdated) entries when they are extracted. This is called lazy deletion and is the standard approach in competitive programming and most textbook Java implementations.

The theoretical complexity becomes O((|V| + |E|) log |E|) instead of O((|V| + |E|) log |V|), but since log |E| ≤ log |V|² = 2 log |V|, this is the same asymptotic class.

7.2 · Path Reconstruction in Java

static List<Integer> reconstructPath(int[] prev, int s, int target) {
    LinkedList<Integer> path = new LinkedList<>();
    for (int v = target; v != -1; v = prev[v]) {
        path.addFirst(v);
    }
    if (path.getFirst() == s) {
        return path;           // valid path
    }
    return Collections.emptyList();  // unreachable
}
