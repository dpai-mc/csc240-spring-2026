2   Application 1 — Connected Components

2.1   Definition Review

Recall from Module 12.1:

🔑 Connected Component
A connected component of an undirected graph $$G = (V, E)$$ is a maximal set of vertices $$C \subseteq V$$ such that there exists a path between every pair of vertices in $$C$$. A graph is connected if it has exactly one connected component.

A disconnected graph has two or more connected components. The question we want to answer algorithmically is: Given a graph $$G$$, how many connected components does it have, and which vertices belong to each?

2.2   The FindComponents Algorithm

The idea is beautifully simple: start a BFS (or DFS) from any unvisited vertex. The traversal will reach exactly the vertices in that vertex's connected component — no more, no less. Then move to the next unvisited vertex and repeat.

⚙️ Algorithm: FindComponents(G)
Algorithm FindComponents(G):
    Input:  Undirected graph G = (V, E) as adjacency list
    Output: Component label for each vertex

    visited[v] ← false for all v ∈ V
    componentID[v] ← -1 for all v ∈ V
    componentCount ← 0

    for each vertex v ∈ V:
        if visited[v] = false:
            componentCount ← componentCount + 1
            // BFS (or DFS) from v; label everything reached
            BFS(G, v):
                for each vertex u reached:
                    visited[u] ← true
                    componentID[u] ← componentCount

    return componentCount, componentID[]

Why does it work?

  • BFS (or DFS) from vertex $$v$$ visits every vertex reachable from $$v$$ — that is exactly the connected component containing $$v$$.
  • Once a vertex is marked visited, it is never processed again, so no vertex appears in two components.
  • The outer loop ensures that every vertex is eventually visited, even if the graph is disconnected.

Time Complexity

Each vertex is enqueued/pushed exactly once across all BFS/DFS calls. Each edge is examined exactly once (twice for undirected, but still $$O(1)$$ per edge). Therefore the total time is:

$$T(V, E) = O(|V| + |E|)$$

This is the same as a single BFS or DFS — the outer loop adds only $$O(|V|)$$ overhead for checking the visited array.

💻 Java Pseudocode — FindComponents 
int findComponents(List<List<Integer>> adj, int n) {
    boolean[] visited = new boolean[n];
    int[] compID = new int[n];
    Arrays.fill(compID, -1);
    int count = 0;

    for (int v = 0; v < n; v++) {
        if (!visited[v]) {
            count++;
            // BFS from v
            Queue<Integer> queue = new LinkedList<>();
            queue.add(v);
            visited[v] = true;
            compID[v] = count;

            while (!queue.isEmpty()) {
                int u = queue.poll();
                for (int w : adj.get(u)) {
                    if (!visited[w]) {
                        visited[w] = true;
                        compID[w] = count;
                        queue.add(w);
                    }
                }
            }
        }
    }
    return count;  // compID[] has labels
}

3   Application 2 — Topological Sorting

3.1   Motivation

Many real-world problems involve tasks with prerequisite constraints:

  • Course scheduling: You must complete CSC 221 before CSC 240.
  • Build systems: Source file A must be compiled before library B can link against it.
  • Package managers: Package X depends on package Y, which depends on package Z — install in the right order.

All of these are naturally modeled as directed acyclic graphs (DAGs) — a concept you first saw in Module 12.2. The question is: In what order should we process the vertices so that every prerequisite is handled before the task that depends on it?

🔑 Topological Ordering
A topological ordering (or topological sort) of a directed acyclic graph $$G = (V, E)$$ is a linear ordering of all vertices such that for every directed edge $$(u, v) \in E$$, vertex $$u$$ appears before vertex $$v$$ in the ordering.
⚠️ DAGs Only
A topological ordering exists if and only if the graph is a DAG (has no directed cycles). If a cycle exists, there is no way to order the vertices so that every edge points forward — you would need $$u$$ before $$v$$ and $$v$$ before $$u$$, a contradiction. We will formalize this in Section 4 (Cycle Detection).

3.2   The DFS-Based Algorithm

The key insight comes from DFS finish times (from Module 13.2):

🔑 Finish-Time Property
In a DFS of a DAG, if there is a directed edge $$(u, v)$$, then vertex $$v$$ finishes before vertex $$u$$. That is, $$\text{finish}[v] < \text{finish}[u]$$.

Therefore, if we output vertices in reverse order of finish time (highest finish time first), every edge $$(u, v)$$ will have $$u$$ appearing before $$v$$ — exactly what a topological ordering requires.

⚙️ Algorithm: TopologicalSort(G)
Algorithm TopologicalSort(G):
    Input:  Directed acyclic graph G = (V, E) as adjacency list
    Output: List of vertices in topological order

    visited[v] ← false for all v ∈ V
    stack ← empty stack  // to collect reverse finish order
    time ← 0

    for each vertex v ∈ V:
        if visited[v] = false:
            DFS-Visit(G, v, visited, stack, time)

    return stack  // pop order = reverse finish order = topological order

Procedure DFS-Visit(G, u, visited, stack, time):
    visited[u] ← true
    time ← time + 1
    discovery[u] ← time

    for each neighbor v ∈ adj[u]:
        if visited[v] = false:
            DFS-Visit(G, v, visited, stack, time)

    time ← time + 1
    finish[u] ← time
    stack.push(u)  // push when finished (all descendants done)

Why does it work? (Proof Sketch)

Consider any directed edge $$(u, v)$$ in the DAG. When DFS explores $$u$$, one of two things happens:

  1. $$v$$ has not yet been visited: DFS will visit $$v$$ as a descendant of $$u$$, so $$v$$ finishes before $$u$$. Thus $$\text{finish}[v] < \text{finish}[u]$$.
  2. $$v$$ has already been visited and finished: Then $$\text{finish}[v]$$ is already set, and since $$u$$ hasn't finished yet, $$\text{finish}[v] < \text{finish}[u]$$.

The third possibility — $$v$$ is visited but not yet finished (i.e., $$v$$ is an ancestor of $$u$$ on the recursion stack) — would mean $$(u, v)$$ is a back edge, creating a cycle. But we assumed the graph is a DAG, so this case cannot occur.

Since $$\text{finish}[v] < \text{finish}[u]$$ for every edge $$(u, v)$$, outputting vertices from highest to lowest finish time gives a valid topological ordering. ∎

💻 Java Pseudocode — TopologicalSort 
List<Integer> topologicalSort(List<List<Integer>> adj, int n) {
    boolean[] visited = new boolean[n];
    Deque<Integer> stack = new ArrayDeque<>();

    for (int v = 0; v < n; v++) {
        if (!visited[v]) {
            dfsVisit(adj, v, visited, stack);
        }
    }

    // Stack pop order = reverse finish order = topological order
    List<Integer> order = new ArrayList<>();
    while (!stack.isEmpty()) {
        order.add(stack.pop());
    }
    return order;
}

void dfsVisit(List<List<Integer>> adj, int u,
              boolean[] visited, Deque<Integer> stack) {
    visited[u] = true;
    for (int v : adj.get(u)) {
        if (!visited[v]) {
            dfsVisit(adj, v, visited, stack);
        }
    }
    stack.push(u);  // push after all descendants finished
}

4   Application 3 — Cycle Detection

4.1   Why Cycle Detection Matters

  • Deadlock detection: In an operating system, a cycle in the resource-allocation graph means processes are deadlocked.
  • Dependency validation: A circular dependency in a build system or package manager means the build cannot proceed.
  • Topological sort precondition: Topological ordering is only possible on DAGs. Cycle detection tells us whether topological sort is even applicable.

4.2   Edge Classification Review

From Module 13.2, DFS classifies every edge into one of four types based on vertex colors (white/gray/black):

Edge TypeTarget ColorMeaning
Tree edgeWhiteTarget is unvisited; edge is part of the DFS tree
Back edgeGrayTarget is an ancestor on the current recursion path → indicates a cycle
Forward edgeBlackTarget is a descendant (already finished); directed graphs only
Cross edgeBlackTarget is in a different branch (already finished); directed graphs only

The critical insight: a graph contains a cycle if and only if DFS encounters a back edge.

4.3   Cycle Detection in Directed Graphs

🔑 Directed Cycle Detection Rule
During DFS on a directed graph, if we encounter an edge $$(u, v)$$ where $$v$$ is gray (currently on the recursion stack / in progress), then $$(u, v)$$ is a back edge and the graph contains a directed cycle.

No parent check is needed — in a directed graph, if we reach a gray vertex through any edge, it means we have found a path from that vertex back to itself.

⚙️ Algorithm: HasCycleDirected(G)
Algorithm HasCycleDirected(G):
    Input:  Directed graph G = (V, E) as adjacency list
    Output: true if G contains a directed cycle, false otherwise

    color[v] ← WHITE for all v ∈ V  // WHITE=unvisited, GRAY=in-progress, BLACK=finished

    for each vertex v ∈ V:
        if color[v] = WHITE:
            if DFS-Cycle(G, v, color):
                return true
    return false

Procedure DFS-Cycle(G, u, color):
    color[u] ← GRAY

    for each neighbor v ∈ adj[u]:
        if color[v] = GRAY:       // Back edge found → cycle!
            return true
        if color[v] = WHITE:
            if DFS-Cycle(G, v, color):
                return true

    color[u] ← BLACK          // Finished processing u
    return false

4.4   Cycle Detection in Undirected Graphs

Undirected cycle detection is similar but requires one crucial difference: the parent check.

⚠️ Why the Parent Check is Necessary

In an undirected graph, every edge $$\{u, v\}$$ appears in both adjacency lists: $$v$$ is in $$\text{adj}[u]$$ and $$u$$ is in $$\text{adj}[v]$$. When DFS visits $$u$$ and then moves to $$v$$, the very next step will see $$u$$ in $$v$$'s adjacency list — but this is just the edge we came from, not a cycle.

Without a parent check, every edge in an undirected graph would be falsely reported as a cycle.

🔑 Undirected Cycle Detection Rule
During DFS on an undirected graph, if we encounter a visited vertex $$v$$ that is not the parent of the current vertex $$u$$, then the graph contains a cycle.
⚙️ Algorithm: HasCycleUndirected(G)
Algorithm HasCycleUndirected(G):
    Input:  Undirected graph G = (V, E) as adjacency list
    Output: true if G contains a cycle, false otherwise

    visited[v] ← false for all v ∈ V

    for each vertex v ∈ V:
        if visited[v] = false:
            if DFS-Cycle-Undir(G, v, -1):  // -1 = no parent
                return true
    return false

Procedure DFS-Cycle-Undir(G, u, parent):
    visited[u] ← true

    for each neighbor v ∈ adj[u]:
        if visited[v] = false:
            if DFS-Cycle-Undir(G, v, u):
                return true
        else if v ≠ parent:   // visited AND not the parent → cycle!
            return true

    return false

🔗 Connection to Topological Sort

If cycle detection finds a cycle in a directed graph, topological sort is impossible. In practice, you can combine both into a single DFS pass: if you encounter a back edge (gray vertex), report "cycle found — no topological ordering exists." Otherwise, push vertices onto the stack as they finish, and the pop order is a valid topological ordering.

💻 Java Pseudocode — Directed Cycle Detection 
enum Color { WHITE, GRAY, BLACK }

boolean hasCycleDirected(List<List<Integer>> adj, int n) {
    Color[] color = new Color[n];
    Arrays.fill(color, Color.WHITE);

    for (int v = 0; v < n; v++) {
        if (color[v] == Color.WHITE) {
            if (dfsCycleDir(adj, v, color))
                return true;
        }
    }
    return false;
}

boolean dfsCycleDir(List<List<Integer>> adj, int u, Color[] color) {
    color[u] = Color.GRAY;

    for (int v : adj.get(u)) {
        if (color[v] == Color.GRAY)
            return true;           // back edge → cycle
        if (color[v] == Color.WHITE && dfsCycleDir(adj, v, color))
            return true;
    }

    color[u] = Color.BLACK;
    return false;
}
💻 Java Pseudocode — Undirected Cycle Detection 
boolean hasCycleUndirected(List<List<Integer>> adj, int n) {
    boolean[] visited = new boolean[n];

    for (int v = 0; v < n; v++) {
        if (!visited[v]) {
            if (dfsCycleUndir(adj, v, -1, visited))
                return true;
        }
    }
    return false;
}

boolean dfsCycleUndir(List<List<Integer>> adj, int u,
                       int parent, boolean[] visited) {
    visited[u] = true;

    for (int v : adj.get(u)) {
        if (!visited[v]) {
            if (dfsCycleUndir(adj, v, u, visited))
                return true;
        } else if (v != parent) {
            return true;   // visited & not parent → cycle
        }
    }
    return false;
}

5   Putting It Together — One DFS, Multiple Answers

A powerful insight: you do not need to run separate DFS passes for cycle detection and topological sorting. A single DFS pass can do both simultaneously:

⚙️ Combined Algorithm: CycleCheckAndTopoSort(G)
Algorithm CycleCheckAndTopoSort(G):
    Input:  Directed graph G = (V, E)
    Output: Topological order if DAG; "Cycle detected" otherwise

    color[v] ← WHITE for all v ∈ V
    stack ← empty stack
    hasCycle ← false

    for each vertex v ∈ V:
        if color[v] = WHITE:
            DFS-Combined(G, v)

    if hasCycle:
        return "Cycle detected — no topological ordering exists"
    else:
        return stack  // pop order = topological order

Procedure DFS-Combined(G, u):
    color[u] ← GRAY

    for each neighbor v ∈ adj[u]:
        if color[v] = GRAY:
            hasCycle ← true       // back edge → cycle
        if color[v] = WHITE:
            DFS-Combined(G, v)

    color[u] ← BLACK
    stack.push(u)               // record finish order regardless

This combined approach is exactly what you will implement in Final Project Component 2. The logic is:

  1. Run DFS with 3-color marking.
  2. If any back edge is encountered (gray neighbor), set a flag — the graph has a cycle.
  3. Push each vertex onto a stack when it finishes (turns black).
  4. After DFS completes: if the cycle flag is set, report no topological ordering. Otherwise, pop the stack for the topological order.
💡 Complexity
The combined algorithm still runs in $$O(|V| + |E|)$$ — exactly the same as a single DFS. We are simply doing a small amount of extra bookkeeping (checking colors and pushing to a stack) at each vertex and edge, all in $$O(1)$$ per operation.

Both are valid cycles caused by the same back edge. Since the graph has a cycle, topological sort would be impossible.
✗ Not quite. Trace DFS from P: P→Q→R→T. At T, we check neighbor Q — Q is gray (still on the recursion stack), so T→Q is a back edge. This means there is a cycle. In fact, there are two cycles involving this back edge: Q→R→T→Q and Q→S→T→Q. Both (B) and (C) correctly identify cycles, so the answer is (D).
✅ Check 13.4d — Undirected Cycle Detection (Parent Check)

Consider this undirected graph:

W: [X, Y]
X: [W, Y]
Y: [W, X, Z]
Z: [Y]

DFS starts at W. The first recursive call goes to X (parent = W). From X, we examine neighbor W.

Why do we NOT report a cycle when X sees W in its neighbor list?

✓ Correct! W is visited and is X's parent. The edge {X, W} is the same edge we used to arrive at X — traversing it "backwards" is not a cycle, just the undirected nature of the edge. The parent check prevents this false positive. However, this graph does contain a cycle: from X, neighbor Y is unvisited, so DFS goes to Y. At Y, neighbor W is visited and W ≠ parent (Y's parent is X), so {Y, W} is a back edge revealing the cycle W–X–Y–W.
✗ Not quite. When DFS moves from W to X, X's parent is set to W. When X then examines its neighbor list and sees W, W is visited — but W is also X's parent. The parent check says: "if the visited neighbor is my parent, skip it." This is necessary because in an undirected graph, the edge {W, X} appears in both adjacency lists, and following it back to the parent is not a cycle. The correct answer is (B).
✅ Check 13.4e — Real-World Scenario: Build System

A software build system has the following compilation dependencies (each arrow means "must be compiled before"):

utils.java     → parser.java
parser.java    → compiler.java
compiler.java  → optimizer.java
optimizer.java → parser.java      ← suspicious!
utils.java     → logger.java

Can this build system produce a valid compilation order? Why or why not?

✓ Correct! The dependency graph contains the directed cycle: parser → compiler → optimizer → parser. This means there is no valid compilation order — parser needs compiler to be done, but compiler (via optimizer) needs parser to be done first. DFS would detect the back edge optimizer→parser (parser would be gray when optimizer tries to visit it). Since the graph is not a DAG, topological sort is impossible. This is exactly the kind of circular dependency that package managers and build tools must detect and report as an error.
✗ Not quite. Look at the chain: parser.java → compiler.java → optimizer.java → parser.java. This is a directed cycle! You cannot compile parser before compiler (parser depends on... well, compiler depends on optimizer which depends on parser). No linear ordering can satisfy all these constraints simultaneously. The answer is (B): the circular dependency makes topological sort impossible.
✅ Check 13.4f — Conceptual: Edge to a Black Vertex

During DFS on a directed graph, you encounter an edge $$(u, v)$$ where $$v$$ is colored BLACK (already finished). Does this indicate a cycle?

✓ Correct! A black vertex has already finished processing — it is not on the current recursion stack. An edge to a black vertex is either a forward edge (to a descendant that finished earlier) or a cross edge (to a vertex in a different DFS subtree). Neither indicates a cycle. Only an edge to a gray vertex (still on the recursion stack) is a back edge, which is the sole indicator of a cycle in directed graphs.
✗ Not quite. In directed graph DFS, vertex colors tell us everything. A gray vertex is on the current recursion path — an edge to it creates a cycle. A black vertex has completely finished — it is no longer on the recursion stack, so an edge to it is a forward or cross edge, not a cycle indicator. The answer is (B).
