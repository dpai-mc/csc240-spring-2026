1   Context and Motivation

Last week you learned how to define, classify, and represent graphs. You built adjacency lists and adjacency matrices, identified special graph families, and proved the Handshaking Lemma. Those structures were static — snapshots of relationships frozen in place.

This week we bring those static structures to life with traversal algorithms that systematically visit every vertex in a graph. A traversal is the graph equivalent of iterating through every element of an array — except that graphs have no inherent "first" or "next" element, so we need a principled strategy for choosing where to go.

Motivating Question
Given a social-network graph where vertices are people and edges are friendships, how would you find the shortest chain of friendships between two people? For example, how many "friend-of-a-friend" hops separate you from a particular celebrity? By the end of this module you will know the answer: Breadth-First Search (BFS).

Module 13.1 covers BFS — the first of two fundamental graph traversal algorithms. Module 13.2 will introduce Depth-First Search (DFS), and the remaining modules this week will compare the two and apply them to connected components, topological sorting, and cycle detection.

2   BFS Intuition — Level-by-Level Exploration

2.1   The Pond-Ripple Analogy

Imagine dropping a stone into a still pond. Ripples spread outward in concentric circles — first the ring closest to the impact point, then the next ring, then the next. BFS explores a graph in exactly this way:

  1. Start at a source vertex s.
  2. Visit every neighbor of s (distance 1).
  3. Then visit every unvisited neighbor of those vertices (distance 2).
  4. Continue until every reachable vertex has been visited.

The key property: BFS visits all vertices at distance d before visiting any vertex at distance d + 1. This is why BFS naturally discovers shortest paths in unweighted graphs.

3   The BFS Algorithm

3.1   Core Mechanism: The Queue

BFS uses a queue (first-in, first-out) to manage the frontier of vertices waiting to be explored. This is the structural reason BFS explores level by level — vertices discovered earlier are processed earlier.

Connection to CSC 221
The queue data structure you implemented in your prerequisite course is the engine that drives BFS. FIFO ordering is what guarantees the level-by-level property.

3.2   Vertex Coloring Convention

To keep track of each vertex's status, the classic BFS algorithm uses a three-color scheme:

ColorMeaning
WhiteUndiscovered — has not yet been seen by BFS.
GrayDiscovered and in the queue — seen but not yet fully explored.
BlackFinished — dequeued and all neighbors examined.
Practical Simplification
In many implementations you will see a simple boolean[] visited array instead of three colors. Setting visited[v] = true when you enqueue v is equivalent to coloring it gray. The three-color scheme is useful for proofs and for understanding DFS edge classification (Module 13.2).

3.3   BFS Pseudocode

// BFS from source vertex s in graph G
procedure BFS(G, s):
    for each vertex u in G.V:          // initialize all vertices
        color[u]  ← WHITE
        dist[u]   ← ∞
        parent[u] ← NIL

    color[s]  ← GRAY                     // discover the source
    dist[s]   ← 0
    parent[s] ← NIL

    Q ← new empty Queue
    Q.enqueue(s)

    while Q is not empty:
        u ← Q.dequeue()                  // process front of queue
        for each v in G.adj[u]:          // examine each neighbor
            if color[v] == WHITE:          // undiscovered?
                color[v]  ← GRAY
                dist[v]   ← dist[u] + 1  // one hop farther
                parent[v] ← u            // record BFS tree edge
                Q.enqueue(v)
        color[u] ← BLACK                 // u is fully explored

3.4   Key Arrays Maintained

  • color[] — tracks white/gray/black status (or a simple visited[] boolean).
  • dist[] — stores the shortest distance (number of edges) from the source to each vertex.
  • parent[] — stores the predecessor of each vertex on the shortest path from the source, enabling path reconstruction.

4   Complexity Analysis

4.1   BFS on an Adjacency List — O(|V| + |E|)

Let's count the work BFS performs:

  1. Initialization: The for each vertex loop runs |V| times → O(|V|).
  2. Queue operations: Every vertex is enqueued at most once and dequeued at most once → O(|V|) total queue operations.
  3. Neighbor scanning: When vertex u is dequeued, we scan its adjacency list, which has deg(u) entries. Summing over all vertices gives the total scanning work:

∑u ∈ V deg(u) = 2|E|    (Handshaking Lemma from Module 12.4)

So the total neighbor-scanning work is O(|E|). Combining: O(|V|) + O(|E|) = O(|V| + |E|).

Key Result
BFS on an adjacency-list representation runs in O(|V| + |E|) time. This is optimal — you cannot do better than looking at every vertex and every edge at least once.

4.2   BFS on an Adjacency Matrix — O(|V|²)

If the graph is stored as an adjacency matrix, finding the neighbors of vertex u requires scanning the entire row of length |V|. We do this for each of the |V| vertices dequeued:

|V| × |V| = |V|²  ⟹  O(|V|²)

For sparse graphs (|E| ≪ |V|²), the adjacency list is dramatically faster. For dense graphs (|E| ≈ |V|²), both representations give similar performance.

RepresentationBFS TimeBest For
Adjacency ListO(|V| + |E|)Sparse graphs
Adjacency MatrixO(|V|²)Dense graphs (or when edge-lookup speed matters)

5   BFS Tree and Shortest Paths

5.1   What Is the BFS Tree?

As BFS runs, each vertex v (other than the source) is discovered from exactly one vertex u, and we set parent[v] = u. The collection of these parent–child edges forms a BFS tree rooted at the source.

Definition — BFS Tree
The BFS tree of a graph G from source s is the spanning tree (of the connected component containing s) formed by the edges {(parent[v], v) | v ≠ s}. Each vertex appears at a level equal to its distance from s.

5.2   Shortest-Path Property

Theorem — BFS Shortest-Path Property
For every vertex v reachable from source s in an unweighted graph, the value dist[v] computed by BFS equals the length of a shortest path from s to v (measured in number of edges).

Intuition for why this holds: BFS explores vertices in non-decreasing order of distance. When v is first discovered via some vertex u at level d, we set dist[v] = d + 1. No shorter path to v can exist, because all vertices at distance ≤ d have already been processed.

5.3   Reconstructing a Shortest Path

To find the actual shortest path from s to a target vertex t, follow the parent[] pointers backward:

procedure PrintPath(s, t):
    if t == s:
        print s
    else if parent[t] == NIL:
        print "No path exists"
    else:
        PrintPath(s, parent[t])
        print t
Unweighted Graphs Only
The shortest-path guarantee applies to unweighted graphs (or equivalently, graphs where every edge has weight 1). For weighted graphs, you need Dijkstra's algorithm or Bellman-Ford — topics coming in Week 14.

7   Java Code Walkthrough

Expand: BFS Implementation in Java

The following Java method implements BFS on a graph stored as an adjacency list (List<List<Integer>>). Vertices are numbered 0 through |V| − 1.

import java.util.*;

public class GraphBFS {

    // adj.get(u) returns list of u's neighbors
    private List<List<Integer>> adj;
    private int V;

    public GraphBFS(int V) {
        this.V = V;
        adj = new ArrayList<>();
        for (int i = 0; i < V; i++)
            adj.add(new ArrayList<>());
    }

    public void addEdge(int u, int v) {
        adj.get(u).add(v);
        adj.get(v).add(u);
    }

    // Returns dist[] where dist[v] = shortest edge-distance from s,
    // or -1 if unreachable.
    public int[] bfs(int s) {

        int[] dist   = new int[V];   // ① distance array
        int[] parent = new int[V];   // ② parent array
        Arrays.fill(dist, -1);       // -1 = unvisited (white)
        Arrays.fill(parent, -1);

        Queue<Integer> queue = new LinkedList<>();

        dist[s] = 0;                  // ③ source: distance 0
        queue.add(s);                   // ④ enqueue source

        while (!queue.isEmpty()) {
            int u = queue.poll();       // ⑤ dequeue front vertex

            for (int v : adj.get(u)) { // ⑥ scan neighbor list
                if (dist[v] == -1) {      // ⑦ unvisited?
                    dist[v]   = dist[u] + 1; // ⑧ set distance
                    parent[v] = u;           // ⑨ record parent
                    queue.add(v);            // ⑩ enqueue neighbor
                }
            }
        }
        return dist;
    }
}

Line-by-Line Annotations

MarkerPurpose
① ②Allocate distance and parent arrays. Using -1 as "unvisited" replaces the white/gray/black coloring — a vertex is "white" if dist[v] == -1.
③ ④Initialize the source: distance 0, then enqueue it.
⑤queue.poll() removes and returns the front (FIFO). This is the vertex currently being explored.
⑥Iterate over the adjacency list of u. Across all iterations this is where the O(|E|) work happens.
⑦Check if v has been discovered. If dist[v] != -1, it was already enqueued — skip it.
⑧ ⑨ ⑩Discover v: record its distance (one more than u), set its parent to u, and enqueue it.
Why check before enqueuing?
We check dist[v] == -1 before enqueuing, not when dequeuing. This prevents a vertex from being enqueued multiple times and keeps the queue size bounded by O(|V|).

8   Knowledge Checks

These are ungraded self-checks with immediate explanatory feedback. Use them to confirm your understanding before moving to Module 13.2.

Check 1 — BFS Trace on a New Graph

Consider the undirected graph with vertices {1, 2, 3, 4, 5} and edges {1-2, 1-3, 2-4, 3-4, 3-5}. If you run BFS from vertex 1 (processing neighbors in numerical order), what is the BFS traversal order?

Correct! Starting from 1, we enqueue neighbors 2 and 3 (level 1). Dequeue 2 → enqueue 4 (level 2; 1 already visited). Dequeue 3 → 4 is already gray, enqueue 5 (level 2). Dequeue 4 and 5 → no new neighbors. Order: 1, 2, 3, 4, 5.
Not quite. Remember BFS uses a queue (FIFO). From vertex 1, neighbors 2 and 3 are enqueued in numerical order. Vertex 2 is dequeued first, then 3. Work through each dequeue step and track what gets enqueued. The correct order is 1, 2, 3, 4, 5.

Check 2 — Complexity: List vs. Matrix

A graph has 1,000 vertices and 3,000 edges. Approximately how many operations does BFS perform on an adjacency list vs. an adjacency matrix?

Correct! On an adjacency list, BFS runs in O(|V| + |E|) = O(1,000 + 3,000) = O(4,000). On an adjacency matrix, BFS runs in O(|V|²) = O(1,000²) = O(1,000,000). The list representation is roughly 250× faster for this sparse graph.
Not quite. Recall: adjacency list → O(|V| + |E|); adjacency matrix → O(|V|²). Plug in |V| = 1,000 and |E| = 3,000. The list gives ~4,000 operations and the matrix gives ~1,000,000. The correct answer is A.

Check 3 — True or False: Shortest-Path Property

"BFS guarantees the shortest path between two vertices in any graph, including graphs with weighted edges."

Correct — the statement is False. BFS guarantees shortest paths only in unweighted graphs (every edge has the same weight). In weighted graphs, BFS may find a path with fewer edges but greater total weight. For weighted shortest paths, use Dijkstra's algorithm (Week 14).
Not quite. The statement is False. BFS counts edges, not weights. Consider a graph where the direct edge A→B has weight 10, but the path A→C→B has total weight 3. BFS would report A→B as "shortest" (1 edge), but the cheaper path uses 2 edges. The shortest-path guarantee applies only to unweighted graphs.

Check 4 — BFS Tree Level = Distance

Refer to the worked example (Section 6). In the BFS tree rooted at A, vertex G appears at level 3. Which of the following is the best interpretation?

Correct! The level of a vertex in the BFS tree equals its shortest distance (in edges) from the source. G at level 3 means the shortest path from A to G uses exactly 3 edges (A → B → D → G). Option A confuses discovery order with level. Option C is coincidentally true for G (it has 3 neighbors) but is unrelated to its BFS level.
Not quite. The BFS tree level of a vertex equals the shortest distance (number of edges) from the source. G at level 3 means the shortest A-to-G path has 3 edges. The correct answer is B. Review Section 5.2 for the shortest-path property.
