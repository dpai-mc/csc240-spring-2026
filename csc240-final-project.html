🌐 Final Project: Network Analysis Application

CSC 240 – Discrete Structures  |  250 Points

Overview & Objectives

The Final Project is a capstone application that replaces the traditional final exam. You will design and implement a complete Network Analysis Application that integrates the graph theory, tree structures, and algorithm analysis topics studied throughout the second half of the course (Weeks 12–16) into a single, coherent system.

What This Project Assesses

  • Your ability to build data structures by hand (no built-in library classes for the required tree-based component).
  • Correct implementation of core graph and tree algorithms.
  • Thoughtful algorithm selection, complexity analysis, and written justification.
  • Software engineering practices: clean code, testing, documentation.

CSLO Alignment

CSLO 3 Solve problems using trees and graphs including traversals, spanning trees, shortest path, and related algorithms.

CSLO 4 Analyze the complexity of algorithms using asymptotic notation and apply algorithm design strategies.

The project also reinforces CSLO 1 (formal logic & proof reasoning used in your analysis document) and CSLO 2 (combinatorics/counting when analyzing graph properties).

Scenario & Context

Imagine you have been hired as a junior software engineer at a logistics company. The company needs an internal tool to model its distribution network—warehouses (vertices) connected by shipping routes (edges) with associated distances or costs (weights).

Your tool must let analysts:

  1. Build a network graph interactively (add/remove warehouses and routes).
  2. Explore the network using traversal algorithms to discover connectivity and dependencies.
  3. Optimize the network by finding minimum spanning trees (cheapest way to connect all warehouses) and shortest paths (fastest route between two locations).
  4. Store internal data efficiently using a hand-built tree-based structure (priority queue or BST).
  5. Document design choices and complexity trade-offs for the engineering team.

This mirrors real-world network analysis tasks in logistics, telecommunications, social-network analysis, and infrastructure planning.

Tasks / Procedure

The project is organized into five scored phases. Complete them in order; each phase builds on the previous one.

Phase 1 — Graph Builder and Representation  50 pts

Build the foundation: a graph data structure that supports both adjacency list and adjacency matrix representations.

Requirements

  • Support weighted, directed, and undirected graphs (configurable at creation or via toggle).
  • Operations: add vertex, remove vertex, add edge, remove edge.
  • Display graph properties: vertex count, edge count, degree of each vertex, whether the graph is connected.
  • Provide a way to switch between or display both adjacency list and adjacency matrix representations.

Input / Output Specification

InputExpected Output
User adds vertices A, B, C and edges A→B (weight 4), B→C (weight 2)Adjacency list and matrix both display correctly; vertex count = 3, edge count = 2
User removes vertex BAll edges involving B are removed; counts update accordingly
User toggles directed → undirectedEach edge is mirrored; matrix becomes symmetric

Testing Guidance

  • Test with an empty graph (0 vertices), a single vertex, and a graph with 8+ vertices.
  • Verify adjacency list and matrix stay in sync after every add/remove.
  • Test edge cases: duplicate edges, self-loops, removing a non-existent vertex.
Hints & Tips
  • Use a Map<String, List<Edge>> (or dictionary equivalent) for the adjacency list—this makes vertex lookup $$O(1)$$ on average.
  • For the adjacency matrix, consider a 2-D array indexed by vertex ID. You'll need a mapping from vertex names to integer indices.
  • Keep a single source of truth (e.g., the adjacency list) and generate the matrix from it, or maintain both and update together.
  • Encapsulate graph operations in a class so Phases 2–4 can call them cleanly.

Phase 2 — Traversal Algorithms  50 pts

Implement fundamental graph traversal and ordering algorithms.

Requirements

  • Breadth-First Search (BFS) from a user-specified start vertex.
  • Depth-First Search (DFS) — provide both a recursive and an iterative implementation.
  • Connected components detection — list each component's vertices.
  • Topological sort on directed acyclic graphs (DAGs); gracefully report if the graph contains a cycle.

Input / Output Specification

InputExpected Output
BFS from vertex A on a sample 6-vertex graphVisit order printed level by level (e.g., A → B, C → D, E, F)
DFS (recursive) from vertex AVisit order showing back-tracking behavior
DFS (iterative) from vertex ASame reachable set as recursive (order may differ due to stack vs. call-stack)
Connected components on a disconnected graphList of components, e.g., {A,B,C}, {D,E}
Topological sort on a DAGValid linear ordering; on a cyclic graph → "Cycle detected"

Testing Guidance

  • Compare recursive and iterative DFS on the same graph—both must visit the same set of vertices.
  • Test BFS/DFS on a disconnected graph to confirm they only visit the reachable component.
  • For topological sort, test with a known DAG (e.g., course prerequisites) and verify ordering validity; also test with a graph containing a cycle.
Hints & Tips
  • Use a visited set (or boolean array) and reset it before each new traversal call.
  • For iterative DFS, use an explicit stack. Push neighbors in reverse order if you want the same visit sequence as the recursive version.
  • Connected components: run BFS or DFS from every unvisited vertex; each run discovers one component.
  • Topological sort can be done via DFS post-order (reverse finish times) or Kahn's algorithm (in-degree approach). Either is acceptable.

Phase 3 — Advanced Graph Algorithms  75 pts

Implement optimization algorithms and perform an empirical runtime comparison.

Requirements

  • Minimum Spanning Tree (MST) — implement either Kruskal's or Prim's algorithm (your choice). You must justify your choice in writing (Phase 5 analysis document).
  • Dijkstra's shortest path algorithm from a user-specified source vertex.
  • Output: MST edges with total weight; shortest path and distance to each reachable vertex.
  • Side-by-side runtime comparison of your MST and shortest-path algorithms on at least two different graph sizes (e.g., 10 vertices vs. 100+ vertices). Display timing results clearly.

Input / Output Specification

InputExpected Output
Run MST on a connected, weighted, undirected graphList of MST edges and total MST weight
Run Dijkstra from source SShortest distance to every vertex; reconstructed shortest path (e.g., S→A→D→F, cost 11)
Runtime comparison on graph with ~10 vertices and ~100 verticesTable or console output showing elapsed time for each algorithm at each size

Testing Guidance

  • Verify MST output against a hand-calculated example (use a small 5–6 vertex graph you can check on paper).
  • Test Dijkstra with no negative weights (Dijkstra does not handle negative edges). Include a test that confirms correct behavior when multiple shortest paths exist.
  • For timing, use your language's high-resolution timer. Run each algorithm multiple times and average the results to reduce noise.
Hints & Tips
  • Kruskal's needs a Union-Find (Disjoint Set) structure—this is a great data structure to implement from scratch. Time complexity: $$O(E \log E)$$.
  • Prim's pairs naturally with a min-heap priority queue (which you may also build in Phase 4). Time complexity with a binary heap: $$O((V + E) \log V)$$.
  • For Dijkstra's, use a priority queue keyed on tentative distance. If you build a min-heap in Phase 4, you can reuse it here—this demonstrates integration.
  • Generate larger test graphs programmatically (random edges with random weights) for the runtime comparison.
  • When comparing runtimes, discuss why the results match (or don't match) theoretical Big-O in your analysis document.

Phase 4 — Tree-Based Internal Structure  50 pts

Demonstrate mastery of tree data structures by building one from scratch.

Requirements

  • Implement one of the following from scratch (you cannot use a built-in library class):
    • Option A: A min-heap / priority queue used internally by Dijkstra's or Prim's algorithm.
    • Option B: A Binary Search Tree (BST) for storing and retrieving graph node data (e.g., lookup by name, ordered traversal of vertices).
  • Demonstrate at least one tree traversal (e.g., in-order for BST; level-order for heap) with visible output.
  • Your tree structure must be integrated into the larger application—not a standalone demo.

Input / Output Specification

InputExpected Output
Option A: Insert several (vertex, distance) pairs into min-heapExtract-min returns the pair with smallest distance each time; heap property maintained
Option B: Insert graph vertices into BST by nameIn-order traversal prints vertices in alphabetical order
Demonstrate integration (e.g., run Dijkstra using your custom heap)Correct shortest-path results produced using your hand-built structure

Testing Guidance

  • Test insert and extract/delete with edge cases: single element, duplicate keys, already-sorted input.
  • For min-heap: verify the heap property after every insertion and extraction (parent ≤ children).
  • For BST: test with data that would create a degenerate (linked-list) tree to observe worst-case behavior.
  • Print the tree structure (or array representation for heap) at each step to visually confirm correctness.
Hints & Tips
  • A min-heap can be stored in a simple array where the children of index $$i$$ are at $$2i+1$$ and $$2i+2$$, and the parent is at $$\lfloor(i-1)/2\rfloor$$.
  • Key operations for a min-heap: insert (sift up), extractMin (sift down), and decreaseKey (needed for Dijkstra's relaxation step).
  • For a BST, implement insert, search, delete, and at least one traversal (in-order is the most natural).
  • If you choose Option A and use it in Dijkstra's, you get strong integration credit—the heap is doing real work, not just a demo.

Phase 5 — Analysis Document  25 pts

Write a 1–2 page PDF document that demonstrates your understanding of the algorithms you implemented.

Requirements

  • Justify your MST algorithm choice — Why did you choose Kruskal's or Prim's? Under what graph conditions is your choice more efficient?
  • Compare BFS vs. DFS — When would you prefer one over the other? Reference your implementation experience.
  • Time and space complexity analysis for each algorithm implemented (BFS, DFS, topological sort, MST, Dijkstra's, and your tree operations). Use asymptotic notation: $$O$$, $$\Theta$$, or $$\Omega$$ as appropriate.
  • One real-world scenario — Describe a real-world problem (different from the logistics scenario given here) and explain which of your implemented algorithms would apply and why. Analyze the expected complexity for that scenario.

Format & Submission

  • 1–2 pages, 11–12 pt font, standard margins.
  • Submit as a PDF alongside your source code.
  • Use section headings that match the four bullet points above.
Hints & Tips
  • Reference your runtime comparison data from Phase 3 to support your complexity claims.
  • Don't just state Big-O—explain why. For example: "BFS is $$O(V + E)$$ because each vertex is enqueued at most once and each edge is examined at most once (twice for undirected graphs)."
  • For the real-world scenario, think about social networks (friend recommendations via BFS), GPS navigation (Dijkstra), network cabling (MST), or build systems (topological sort).
  • Proofread carefully—clarity and correctness of analysis matter as much as code.

Common Pitfalls

Expand: Common Implementation Mistakes

Graph Basics (Phase 1)

  • Forgetting to update both representations — If you maintain both an adjacency list and matrix, an add/remove must update both. A mismatch is a common source of bugs.
  • Off-by-one index errors in the adjacency matrix when vertices are added or removed dynamically.

Traversals (Phase 2)

  • Not resetting the visited set between separate traversal calls — this causes later traversals to skip vertices.
  • Confusing stack vs. queue — BFS uses a queue (FIFO); iterative DFS uses a stack (LIFO). Mixing them up gives wrong traversal order.
  • Topological sort on a cyclic graph — Make sure you detect cycles and report them instead of producing invalid output.

Advanced Algorithms (Phase 3)

  • Incorrect priority queue key updates in Dijkstra's — When you relax an edge and find a shorter path, you must update (decrease) the key in the priority queue. If your heap doesn't support decreaseKey, you can use lazy deletion (insert a new entry and skip stale ones on extraction).
  • Running MST on a directed graph — Standard Kruskal's and Prim's assume undirected graphs. Make sure your test graph is undirected, or handle the conversion.
  • Negative edge weights with Dijkstra's — Dijkstra's algorithm does not produce correct results with negative weights. Use only non-negative weights in your test data.

Tree Structure (Phase 4)

  • Using a built-in library class (e.g., Java's PriorityQueue, Python's heapq) — This violates the requirement. You must write the data structure yourself.
  • Heap array indexing errors — Double-check parent/child index formulas: children of $$i$$ are at $$2i+1$$ and $$2i+2$$; parent of $$i$$ is at $$\lfloor(i-1)/2\rfloor$$.
  • BST deletion — The case where a node has two children is tricky; replace with the in-order successor (or predecessor) and delete that node recursively.

General

  • No README or missing test cases — These are required deliverables and contribute to the code quality score.
  • Monolithic code — Break your program into well-named classes/modules. A single 500-line file is hard to debug and loses code quality points.
  • Waiting until Week 17 to start — This is a multi-phase project. Start Phase 1 immediately and build incrementally.
