This connects Week 15's tree traversal work to a powerful real-world application — and ties back to the stack-based algorithms you studied in CSC 221.

1. What Is an Expression Tree?

An expression tree is a binary tree that represents an arithmetic (or logical) expression:

  • Every leaf node stores an operand (a variable or constant).
  • Every internal node stores an operator (such as +, −, ×, ÷).
  • The structure of the tree encodes the order of operations — no parentheses are needed.

Key Insight

Because the tree's branching determines which operands belong to which operator, an expression tree is inherently unambiguous. Parentheses, operator precedence rules, and associativity rules are all "baked in" to the tree structure.

Example: (a + b) × (c − d)

          ×
        /   \
       +     −
      / \   / \
     a   b c   d

Figure 15.3.1 — Expression tree for (a + b) × (c − d)

Notice:

  • The root × is the last operation performed (lowest precedence in the original expression with its parentheses).
  • The left subtree represents (a + b); the right subtree represents (c − d).
  • Reading the tree via different traversals yields different notations — explored next.

2. Traversal ↔ Notation Correspondence

Each standard binary-tree traversal produces a well-known expression notation when applied to an expression tree:

Traversal Order Notation Produced Also Known As Example Output
Inorder (Left, Root, Right) Infix Standard algebraic notation a + b × c − d *
Preorder (Root, Left, Right) Prefix Polish Notation × + a b − c d
Postorder (Left, Right, Root) Postfix Reverse Polish Notation (RPN) a b + c d − ×

* The raw inorder traversal of the tree in Figure 15.3.1 visits nodes in the sequence a + b × c − d. Without parentheses this is ambiguous — it could be misread using standard precedence. A fully parenthesized inorder traversal adds parentheses around each subtree: ((a + b) × (c − d)).

Why Prefix and Postfix Are Unambiguous

Both prefix and postfix notations are unambiguous without parentheses because the position of the operator relative to its operands, combined with a known arity (binary = 2 operands), uniquely determines grouping. This is why stack-based calculators and many compilers prefer postfix internally.

Deriving All Three Notations from the Tree

Using the tree for (a + b) × (c − d):

NotationTraversal AppliedResult
Infix (fully parenthesized)Inorder with parens((a + b) × (c − d))
Prefix (Polish)Preorder× + a b − c d
Postfix (RPN)Postordera b + c d − ×

3. Building an Expression Tree from a Fully Parenthesized Infix Expression

A fully parenthesized infix expression has parentheses around every sub-expression, making parsing straightforward. For example: ((a + b) × (c − d)).

Algorithm: Build from Fully Parenthesized Infix 
BuildFromInfix(tokens):
    Create an empty stack S
    Create a new empty node, currentNode
    Set tree root = currentNode
    Push currentNode onto S

    For each token in tokens:
        If token == '(':
            Create a new node as left child of currentNode
            Push currentNode onto S
            Set currentNode = left child

        Else if token is an operand:
            Set currentNode.value = token
            Pop S to set currentNode = top of S

        Else if token is an operator:
            Set currentNode.value = token
            Create a new node as right child of currentNode
            Push currentNode onto S
            Set currentNode = right child

        Else if token == ')':
            Pop S to set currentNode = top of S

    Return root

Intuition: Each ( means "descend left," each operator fills the current node and means "descend right," each operand fills a leaf, and each ) means "ascend to parent." The stack tracks ancestors.

Worked Example — Build from Infix

Expression: ((3 + 5) × (4 − 2))

Try each step yourself before revealing.

Token ( (outer): Create root node. Create left child; push root; descend left.

Stack: [root]   |   Current: root.left

Token (: Create left child of current; push current; descend left.

Stack: [root, root.left]   |   Current: root.left.left

Token 3 (operand): Set current.value = 3. Pop stack → current = root.left.

Stack: [root]   |   Current: root.left

Token + (operator): Set current.value = +. Create right child; push current; descend right.

Stack: [root, root.left(+)]   |   Current: root.left.right

Token 5 (operand): Set current.value = 5. Pop → current = root.left(+).

Stack: [root]   |   Current: root.left(+)

Token ): Pop → current = root.

Stack: []   |   Current: root

Token × (operator): Set root.value = ×. Create right child; push root; descend right.

Stack: [root(×)]   |   Current: root.right

Token (: Create left child; push current; descend left.

Stack: [root(×), root.right]   |   Current: root.right.left

Token 4: current.value = 4. Pop → current = root.right.

Token −: current.value = −. Create right child; push; descend right.

Token 2: current.value = 2. Pop → current = root.right(−).

Token ): Pop → current = root(×).

Token ) (outer): Pop → stack empty. Done!

Final tree:

      ×
    /   \
   +     −
  / \   / \
 3   5 4   2

4. Building an Expression Tree from Postfix — Stack Algorithm

The postfix (RPN) build algorithm is elegant and directly mirrors the stack-based evaluation you learned in CSC 221. Instead of computing a numeric result, we build tree nodes.

Algorithm: Build Expression Tree from Postfix 
BuildFromPostfix(tokens):
    Create an empty stack S   // stack of tree-node pointers

    For each token in tokens:
        If token is an operand:
            Create a new leaf node with value = token
            Push node onto S

        Else if token is an operator:
            Create a new node with value = token
            Pop S → set node.right = popped node
            Pop S → set node.left  = popped node
            Push node onto S

    // After processing all tokens, the stack has exactly one item:
    Return Pop(S)   // this is the root of the expression tree

Key point: The first pop becomes the right child and the second pop becomes the left child (because a stack reverses order).

Worked Example — Build from Postfix

Postfix expression: 3 5 + 4 2 − ×

Attempt each step mentally, then reveal to check.

Operand → create leaf [3]. Push onto stack.

Stack (bottom→top): [3]

Operand → create leaf [5]. Push.

Stack: [3] [5]

Operator → create node [+].

Pop → [5] becomes right child of [+].

Pop → [3] becomes left child of [+].

Push [+] (subtree rooted at +).

Stack: [ +{3,5} ]

   +
  / \
 3   5

Operand → create leaf [4]. Push.

Stack: [ +{3,5} ] [4]

Operand → create leaf [2]. Push.

Stack: [ +{3,5} ] [4] [2]

Operator → create node [−].

Pop → [2] becomes right child. Pop → [4] becomes left child.

Push [−].

Stack: [ +{3,5} ] [ −{4,2} ]

Operator → create node [×].

Pop → [−{4,2}] becomes right child. Pop → [+{3,5}] becomes left child.

Push [×]. Stack has one item — done!

      ×
    /   \
   +     −
  / \   / \
 3   5 4   2

This is the same tree as before. ✓

5. Evaluating an Expression Tree (Recursive Postorder)

Evaluation follows a postorder pattern: evaluate left subtree, evaluate right subtree, then apply the operator at the root.

Algorithm: Recursive Evaluation 
Evaluate(node):
    If node is a leaf:
        Return numeric value of node.value

    leftVal  ← Evaluate(node.left)
    rightVal ← Evaluate(node.right)

    Switch node.value:
        case '+': Return leftVal + rightVal
        case '−': Return leftVal − rightVal
        case '×': Return leftVal × rightVal
        case '÷': Return leftVal ÷ rightVal

This is essentially a postorder traversal that computes a value instead of printing.

Comprehensive Worked Example — Full Pipeline

Let us walk through the complete pipeline for the expression ((3 + 5) × (4 − 2)):

Using either the infix or postfix algorithm, we obtain:

      ×
    /   \
   +     −
  / \   / \
 3   5 4   2
  • Infix (fully parenthesized): ((3 + 5) × (4 − 2))
  • Prefix: × + 3 5 − 4 2
  • Postfix: 3 5 + 4 2 − ×

Evaluate(root):

  1. Evaluate left subtree (rooted at +):
    • Evaluate(3) → 3
    • Evaluate(5) → 5
    • Apply +: 3 + 5 = 8
  2. Evaluate right subtree (rooted at −):
    • Evaluate(4) → 4
    • Evaluate(2) → 2
    • Apply −: 4 − 2 = 2
  3. Apply × at root: 8 × 2 = 16

Final result: 16 ✓

6. Application: Compilers and Abstract Syntax Trees

Expression trees are a simplified form of the Abstract Syntax Tree (AST) that compilers build during the parsing phase. When you write source code, the compiler's parser converts your text into an AST that captures the hierarchical structure of every expression and statement — eliminating ambiguity from the flat text. The compiler then traverses this AST to perform type checking, optimization, and code generation. In particular, a postorder traversal of the AST naturally produces instructions for a stack-based virtual machine (like the Java Virtual Machine's bytecode), which is exactly why postfix notation and stack evaluation are so central to compiler design.

Module 15.3 Summary

  • Expression trees encode arithmetic expressions with operators at internal nodes and operands at leaves — no parentheses needed.
  • Inorder → infix; preorder → prefix (Polish); postorder → postfix (RPN).
  • Build from fully parenthesized infix using a stack to track parent nodes.
  • Build from postfix using a stack of subtrees — mirrors the evaluation algorithm from CSC 221.
  • Evaluate recursively via postorder: evaluate children, then apply operator.
  • Compilers use ASTs (generalized expression trees) for parsing, optimization, and code generation.
