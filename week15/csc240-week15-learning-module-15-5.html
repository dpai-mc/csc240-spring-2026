Final Project & Quiz 5 Planning

Component 4 Overview

Component 4 asks you to integrate a tree-based data structure into the graph application you have been building across the semester. You must choose one of two options and implement it so that it meaningfully supports your graph's operations.

Key Requirements (Both Options)

  • The tree structure must be integrated with your existing graph — not a standalone demo.
  • You must implement core operations (insert, delete/extract, search/traverse) with clear documentation.
  • Your code must include comments explaining how the tree supports the graph algorithm or data management.
  • Component 5 (Analysis Document) will require you to analyze the complexity of your tree choice — plan for that now.

Option A vs. Option B Comparison

Comparison of Option A (Min-Heap / Priority Queue) and Option B (BST for Node Data)
Design Factor Option A: Min-Heap / Priority Queue Option B: BST for Node Data
Purpose in Graph Supports Dijkstra's shortest-path or Prim's MST by efficiently extracting the minimum-weight edge/vertex Stores and retrieves graph node metadata (labels, weights, attributes) in sorted order
Core Operations Insert, Extract-Min, Decrease-Key, Heapify Insert, Search, Delete, In-Order Traversal
Time Complexity (average) Insert: O(log n); Extract-Min: O(log n) Insert: O(log n); Search: O(log n); Traversal: O(n)
Underlying Structure Complete binary tree (array-based representation) Binary search tree (pointer/reference-based nodes)
Traversal Used Level-order (implicit in array); primarily extraction-focused In-order (sorted output), pre-order (serialization), post-order (deletion)
Pros
  • Guaranteed O(log n) extract-min
  • Cache-friendly array layout
  • Natural fit for greedy algorithms
  • Supports range queries and sorted iteration
  • Flexible node-level operations
  • Demonstrates multiple traversal strategies
Cons
  • Decrease-Key can be tricky to implement
  • No efficient arbitrary search
  • Less demonstration of traversal variety
  • Worst case O(n) if unbalanced
  • Deletion logic is more complex
  • May require balancing (AVL) for guarantees
Best If Your Project… Focuses on shortest-path or MST algorithms Focuses on data management, lookup, or sorted reporting

Choosing Your Option

Neither option is inherently harder or easier — choose based on your graph project's algorithmic focus. If your project already implements or plans to implement Dijkstra's or Prim's, Option A is the natural fit. If your project emphasizes node management and reporting, Option B aligns better.

Connecting to the Analysis Document (Component 5)

Component 5 asks you to provide a written analysis of your project's data structures and algorithms. For your tree structure, be prepared to address the following questions:

Complexity Questions to Address in Your Analysis

  1. Time Complexity: What is the best-case, average-case, and worst-case time complexity for each core operation (insert, delete/extract, search)?
  2. Space Complexity: How much additional memory does your tree structure require relative to the number of graph nodes/edges?
  3. Why This Structure? Justify your choice (Option A or B) in terms of the algorithmic needs of your graph project. What would happen if you used a simpler structure (e.g., unsorted array)?
  4. Traversal Analysis: Which traversal(s) does your implementation use, and why? What is the time complexity of a full traversal?
  5. Trade-offs: What did you gain and what did you sacrifice by choosing this tree structure over the alternative option?

You do not need to write the Analysis Document now, but keeping these questions in mind as you implement will make the writing process much smoother in Week 17.
