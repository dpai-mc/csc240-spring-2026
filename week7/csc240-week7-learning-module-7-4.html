Password Security Analysis

Password security is one of the most direct applications of counting in computing. The strength of a password policy depends on the size of the password space — the total number of possible passwords. A larger space means a brute-force attacker must try more guesses.

The Counting Model

A password of length L drawn from an alphabet of size A (with repetition allowed and order matters) is a permutation with repetition:

Total passwords = Aᴸ

This is the Product Rule from Module 7.1: each of the L positions has A independent choices.

📘 Key Terms
  • Alphabet size (A): Number of distinct characters allowed (Sum Rule builds this from character classes)
  • Password length (L): Number of character positions
  • Password space: Total possible passwords = Aᴸ
  • Entropy: log₂(Aᴸ) = L × log₂(A) bits — measures information content
📝 Example 1.1: Basic Password Analysis

A system requires passwords of exactly 8 characters. Compare these policies:

Solution:

📝 Example 1.2: Variable-Length Passwords

A system allows passwords of length 6 to 10 using the 62-character alphanumeric alphabet. How many total passwords are possible?

Solution:

A password has length 6 or 7 or 8 or 9 or 10. These are mutually exclusive categories (a password has exactly one length), so we use the Sum Rule:

Total = 62⁶ + 62⁷ + 62⁸ + 62⁹ + 62¹⁰

= 56,800,235,584 + 3,521,614,606,208 + 218,340,105,584,896 + 13,537,086,546,263,552 + 839,299,365,868,340,224

≈ 8.53 × 10¹⁷

Counting techniques used: Product Rule (each length), Sum Rule (combining lengths).

📝 Example 1.3: Passwords with Constraints (Inclusion-Exclusion)

How many 8-character passwords using the 62-character alphabet contain at least one digit?

Solution:

Using complementary counting (Module 7.1):

Passwords with at least one digit = Total − Passwords with no digits

Total: 62⁸ ≈ 2.18 × 10¹⁴

No digits (letters only, alphabet = 52): 52⁸ ≈ 5.35 × 10¹³

At least one digit: 62⁸ − 52⁸ ≈ 2.18 × 10¹⁴ − 5.35 × 10¹³ ≈ 1.65 × 10¹⁴

Counting techniques used: Product Rule, Complementary Counting (Inclusion-Exclusion).

💡 Entropy and Security Standards

NIST (National Institute of Standards and Technology) recommends at least 80 bits of entropy for high-security passwords. Since entropy = L × log₂(A), you can calculate the minimum length for any alphabet. With the 95-character printable ASCII set: L ≥ 80 / log₂(95) ≈ 80 / 6.57 ≈ 12.2, so at least 13 characters.

IP Address Spaces

IP addresses are a perfect example of the Product Rule applied to networking. Understanding the size of address spaces explains why IPv4 addresses ran out and why IPv6 was designed.

IPv4 Addresses

📘 IPv4 Structure

An IPv4 address is a 32-bit binary number, typically displayed as four 8-bit octets in decimal notation (e.g., 192.168.1.1). Each octet ranges from 0 to 255.

📝 Example 2.1: IPv4 Address Space

How many unique IPv4 addresses exist?

Solution (Method 1 — Binary):

32 bit positions, each with 2 choices (0 or 1). Product Rule:

2³² = 4,294,967,296 ≈ 4.3 billion addresses

Solution (Method 2 — Octets):

4 octets, each with 256 values (0–255). Product Rule:

256⁴ = 256 × 256 × 256 × 256 = 4,294,967,296 ✓

Both methods give the same answer because 256 = 2⁸ and (2⁸)⁴ = 2³².

📝 Example 2.2: Subnet Analysis

A /24 subnet (like 192.168.1.0/24) fixes the first 24 bits. How many host addresses are available?

Solution:

32 − 24 = 8 free bits → 2⁸ = 256 total addresses

Subtract 2 reserved addresses (network address .0 and broadcast .255):

256 − 2 = 254 usable host addresses

General formula for a /n subnet: 2⁽³²⁻ⁿ⁾ − 2 usable hosts

IPv6 Addresses

📘 IPv6 Structure

An IPv6 address is a 128-bit number, displayed as eight groups of four hexadecimal digits (e.g., 2001:0db8:85a3:0000:0000:8a2e:0370:7334).

📝 Example 2.3: IPv6 Address Space

How many unique IPv6 addresses exist?

Solution:

2¹²⁸ ≈ 3.4 × 10³⁸

That's approximately 340 undecillion addresses — enough to assign about 5 × 10²⁸ addresses to every person on Earth.

For perspective: IPv6 / IPv4 ratio = 2¹²⁸ / 2³² = 2⁹⁶ ≈ 7.9 × 10²⁸. IPv6 has about 79 octillion times more addresses than IPv4.

📝 Example 2.4: MAC Addresses

A MAC (Media Access Control) address is 48 bits. How many unique MAC addresses exist? Is this enough for all devices ever manufactured?

Solution:

2⁴⁸ = 281,474,976,710,656 ≈ 281 trillion

With an estimated 30+ billion IoT devices projected by 2030, 281 trillion is sufficient for now — but the address space is not unlimited. This is why EUI-64 (64-bit) identifiers were developed.

Hash Functions and Collisions

Hash functions map data of arbitrary size to fixed-size values. Counting tells us why collisions are inevitable and helps us analyze collision probability.

📘 Key Definitions
  • Hash function: A function h: U → {0, 1, …, m−1} mapping a large universe U to m hash values (buckets)
  • Collision: When two distinct inputs x ≠ y produce the same hash: h(x) = h(y)
  • Load factor: n/m where n is the number of items stored and m is the number of buckets

The Pigeonhole Principle

📐 Pigeonhole Principle

If n items are placed into m containers and n > m, then at least one container holds more than one item.

Consequence for hashing: If the number of possible inputs exceeds the number of hash values, collisions are guaranteed.

📝 Example 3.1: Guaranteed Collisions

A hash function maps strings to 32-bit values. If we hash 2³² + 1 distinct strings, are collisions guaranteed?

Solution:

The hash function has 2³² ≈ 4.3 billion possible output values (pigeonholes). With 2³² + 1 distinct inputs (pigeons), by the Pigeonhole Principle, at least two strings must share a hash value.

Yes, collisions are guaranteed.

The Birthday Problem and Collision Probability

📐 Birthday Problem Framework

If n items are independently and uniformly assigned to m buckets, the probability that all items get distinct buckets is:

P(no collision) = m/m × (m−1)/m × (m−2)/m × … × (m−n+1)/m = P(m, n) / mⁿ

The probability of at least one collision is: 1 − P(no collision)

A collision becomes likely (≥ 50%) when n ≈ 1.2 × √m.

📝 Example 3.2: The Classic Birthday Problem

In a room of 23 people, what is the probability that at least two share a birthday? (Assume 365 equally likely birthdays.)

Solution:

m = 365 (buckets), n = 23 (items).

P(no shared birthday) = P(365, 23) / 365²³

= (365 × 364 × 363 × … × 343) / 365²³

= 365!/342! / 365²³ ≈ 0.4927

P(at least one shared birthday) = 1 − 0.4927 ≈ 0.5073 (50.7%)

Counting techniques: P(365, 23) counts the no-collision arrangements (permutations without repetition). 365²³ counts all possible arrangements (permutations with repetition). The ratio gives the probability.

📝 Example 3.3: Hash Table Collision Threshold

A hash table uses 16-bit hashes (m = 2¹⁶ = 65,536 buckets). Approximately how many items before a collision is likely?

Solution:

Using the birthday approximation: n ≈ 1.2 × √m = 1.2 × √65,536 = 1.2 × 256 ≈ 307 items

With only ~307 items in a 65,536-bucket table (load factor ≈ 0.5%), there's already a 50% chance of collision! This demonstrates why hash functions need large output spaces.

⚠️ Birthday Attack in Cryptography

The birthday problem has serious security implications. A hash function with an n-bit output can be "broken" (collision found) in approximately 2ⁿ/² operations, not 2ⁿ. This is called a birthday attack. It's why MD5 (128-bit) and SHA-1 (160-bit) are considered insecure — birthday attacks require only 2⁶⁴ and 2⁸⁰ operations respectively, which are feasible with modern hardware.

Counting Algorithm Inputs

Counting the number of possible inputs to an algorithm is essential for testing, complexity analysis, and understanding why brute-force approaches fail for many problems.

Boolean Function Inputs

📝 Example 4.1: Truth Table Size

A boolean function has n input variables. How many rows does its truth table have? How many distinct boolean functions of n variables exist?

Solution:

Truth table rows: Each variable has 2 values (T/F). Product Rule: 2ⁿ rows.

Distinct functions: Each of the 2ⁿ rows can output T or F independently. Product Rule again:

Number of boolean functions = 2^(2ⁿ)

  • n = 1: 2² = 4 functions (NOT, identity, always-true, always-false)
  • n = 2: 2⁴ = 16 functions (AND, OR, XOR, NAND, NOR, …)
  • n = 3: 2⁸ = 256 functions
  • n = 10: 2¹⁰²⁴ ≈ 1.8 × 10³⁰⁸ — astronomically more than atoms in the universe

This is why exhaustive testing of all boolean functions is infeasible for even moderate n.

Sorting Algorithm Inputs

📝 Example 4.2: Permutations as Sorting Inputs

A sorting algorithm receives an array of n distinct elements. How many distinct input orderings exist?

Solution:

Each input ordering is a permutation of the n elements: n!

  • n = 10: 10! = 3,628,800
  • n = 20: 20! ≈ 2.4 × 10¹⁸
  • n = 100: 100! ≈ 9.3 × 10¹⁵⁷

Lower bound on comparison sorts: Any comparison-based sorting algorithm must make at least log₂(n!) comparisons in the worst case. By Stirling's approximation: log₂(n!) ≈ n log₂ n − n log₂ e ≈ n log₂ n. This is why comparison sorts are Ω(n log n).

Subset Selection Problems

📝 Example 4.3: The Knapsack Problem

In the 0/1 Knapsack problem, you have n items and must decide which to include. How many possible selections exist?

Solution:

Each item is either included or excluded. Product Rule: 2ⁿ possible subsets.

  • n = 20: 2²⁰ = 1,048,576 (feasible to enumerate)
  • n = 50: 2⁵⁰ ≈ 1.13 × 10¹⁵ (borderline)
  • n = 100: 2¹⁰⁰ ≈ 1.27 × 10³⁰ (completely infeasible)

This exponential growth is why the Knapsack problem is NP-hard — no known polynomial-time algorithm exists.

Graph Problems

📝 Example 4.4: Counting Graphs and Paths

(a) How many simple undirected graphs on n labeled vertices exist?

(b) How many Hamiltonian paths (visiting each vertex exactly once) could exist in a complete graph on n vertices?

Solution:

(a) There are C(n, 2) = n(n−1)/2 possible edges. Each edge is either present or absent:

Number of graphs = 2^(C(n,2)) = 2^(n(n−1)/2)

  • n = 5: 2¹⁰ = 1,024 graphs
  • n = 10: 2⁴⁵ ≈ 3.5 × 10¹³ graphs

(b) A Hamiltonian path is a permutation of the n vertices: n! possible paths to check. For n = 20, that's 20! ≈ 2.4 × 10¹⁸ — this is why the Hamiltonian path problem is NP-complete.

Counting techniques: C(n, 2) for edge count (combination), 2^(C(n,2)) for graph count (Product Rule), n! for path count (permutation).

The counting techniques from this week directly support algorithm analysis in later weeks. The number of possible inputs determines the brute-force baseline.
