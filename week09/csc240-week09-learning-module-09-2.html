1. Opening: What Changes When We Already Know Something?

In Module 9.1 you learned to assign probabilities to events in a sample space. Every outcome was equally possible at the start. But real systems rarely work that way — we almost always have partial information before we make a judgment.

"A program just failed its unit tests. Given that it failed, what is the probability that it also has a memory leak?"

The phrase "given that" is the key. It tells us that certain outcomes are already ruled out — the sample space has been restricted to only those outcomes consistent with what we know. This is conditional probability.

Motivating Scenario: Software Quality Team

A software team is reviewing a batch of 100 programs submitted for final release. Quality records show:

  • 40 programs failed at least one automated test (event F).
  • 25 programs contain a memory leak (event M).
  • 15 programs both failed a test and contain a memory leak (F ∩ M).

A QA engineer randomly picks one program and is told it failed a test. What is the probability it also has a memory leak? Without the "failed a test" information, the answer would be 25/100 = 0.25. But knowing the program failed a test changes the relevant universe — we now look only at the 40 programs that failed, and ask how many of those also have a leak. That gives us 15/40 = 0.375. The extra information raised the probability.

By the end of this module, you will have the formal definition and tools to compute this — and to recognize when additional information has no effect at all (independence).

🔗 Connection — Module 9.1

2. Conditional Probability

📘 Definition — Conditional Probability

Let A and B be events in a probability space with P(B) > 0. The conditional probability of A given B is:

P(A | B) = P(A ∩ B) P(B)

The condition P(B) > 0 is required because we cannot condition on an impossible event. When B has occurred, B becomes the new "universe," and we measure what fraction of B is also in A.

The diagram below shows what conditioning does geometrically. On the left, Ω contains everything. On the right, knowing B occurred means we discard all outcomes outside B — the oval B becomes the new universe, and A ∩ B is the portion of interest within it.

▶ Example 1 — Software QA: Computing P(M | F)

Setup (from the opening scenario): 100 programs; P(F) = 40/100 = 0.40, P(M) = 25/100 = 0.25, P(F ∩ M) = 15/100 = 0.15. Find the probability that a program has a memory leak given it failed a test.

  • Identify which formula to use. We want P(M | F) — the probability of M given B = F. Use the definition: P(M \mid F) = P(M ∩ F) P(F)
  • Substitute known values. P(M \mid F) = 0.15 0.40 = 15 40 = 0.375
  • Interpret the result. Among programs that failed a test, 37.5% also have a memory leak — higher than the unconditional 25%. The failure information updated our probability upward, because memory leaks are disproportionately common among programs that fail tests.

Takeaway: Conditional probability is just a ratio within the restricted universe. Always check that P(B) > 0 before applying the formula.

3. The Multiplication Rule

Rearranging the definition of conditional probability by multiplying both sides by P(B) gives an immediately useful result:

📘 The Multiplication Rule

For any two events A and B with P(B) > 0:

P(A ∩ B) = P(A | B) · P(B)

Equivalently, by symmetry of intersection: P(A ∩ B) = P(B | A) · P(A)  (when P(A) > 0). This rule lets you compute the probability of both events occurring when you know a conditional probability and a marginal probability.

▶ Example 2 — Two Sequential Code Reviews

Setup: A pull request must pass two sequential automated reviews. The probability it passes the first review is 0.80. Given it passes the first, the probability it passes the second is 0.90. What is the probability it passes both?

  • Define events. Let A = "passes first review," B = "passes second review." We want P(A ∩ B).
  • Identify the given values. P(A) = 0.80 and P(B | A) = 0.90.
  • Apply the multiplication rule. P(A ∩ B) = P(B \mid A) · P(A) = 0.90 × 0.80 = 0.72
  • Interpret. There is a 72% chance the pull request clears both automated reviews. Note that the second review's probability (0.90) was stated conditional on passing the first — we cannot simply multiply two unconditional probabilities unless the events are independent (covered in Section 5).

4. Law of Total Probability

Sometimes an event A can happen via several distinct pathways, each associated with a different background condition. The law of total probability lets us add up the contributions from each pathway.

📘 Law of Total Probability

Let {B1, B2, …, Bn} be a partition of Ω — that is, the Bi are pairwise mutually exclusive (Bi ∩ Bj = ∅ for i ≠ j) and their union covers the entire sample space (B1 ∪ B2 ∪ ⋯ ∪ Bn = Ω). Then for any event A:

P(A) = P(A | B1)·P(B1) + P(A | B2)·P(B2) + ⋯ + P(A | Bn)·P(Bn)

Intuitively: A can only happen "through" one of the Bi. We weight each pathway's conditional probability by how likely that pathway is, then sum.

Worked Example: Three Compilers and Bug Introduction

A development team uses three different compilers for different parts of their codebase. Each compiler has a different probability of introducing a bug during compilation. The setup:

Compiler Share of Compilations P(bug introduced | this compiler)
C1 (legacy) 50%   →   P(C1) = 0.50 P(B | C1) = 0.10
C2 (standard) 30%   →   P(C2) = 0.30 P(B | C2) = 0.05
C3 (new) 20%   →   P(C3) = 0.20 P(B | C3) = 0.02

Question: What is the overall probability that a randomly selected compilation introduces a bug?

▶ Example 3 — Three Compilers: Full Solution

Goal: Find P(B), the overall probability that a compilation introduces a bug.

  • Verify the partition. {C1, C2, C3} partitions Ω: the compilers are mutually exclusive (each compilation uses exactly one) and exhaustive (0.50 + 0.30 + 0.20 = 1.00). ✓
  • Apply the law of total probability. P(B) = P(B \mid C_1)\cdot P(C_1) + P(B \mid C_2)\cdot P(C_2) + P(B \mid C_3)\cdot P(C_3)
  • Substitute and compute each term. P(B) = (0.10)(0.50) + (0.05)(0.30) + (0.02)(0.20) P(B) = 0.050 + 0.015 + 0.004 = 0.069
  • Interpret. There is approximately a 6.9% chance that any given compilation introduces a bug. The legacy compiler (C1) contributes the most (0.050 out of 0.069 ≈ 72% of the total bug risk) despite having the same 50% usage share — its bug rate is 5× higher than C2.

Takeaway: The law of total probability is the tool for "un-conditioning" — computing an overall probability when you only know conditional probabilities within each partition element. The probability tree makes each pathway visible and helps catch arithmetic errors.

5. Independence

Conditional probability measures how much knowing B changes the probability of A. What if it changes nothing? Then knowing B is irrelevant to A — the events are independent.

📘 Definition — Independence

Two events A and B are independent if and only if:

P(A ∩ B) = P(A) · P(B)

Equivalently, when P(B) > 0, independence means:

P(A | B) = P(A)

The second form is the intuitive one: learning that B occurred provides zero information about whether A occurs. Both forms are equivalent; use whichever is more convenient for a given problem.

⚠️ Checking Independence
To test whether two events are independent, compute P(A) · P(B) and compare it to P(A ∩ B). If they are equal, the events are independent. If they differ — even slightly — the events are dependent. Do not assume independence; always verify it from the numbers.
▶ Example 4 — Two Independent Network Packets

Setup: A network router forwards two packets independently. Each packet is dropped (lost) with probability 0.05, regardless of what happens to the other. Let D1 = "packet 1 is dropped" and D2 = "packet 2 is dropped."

Given: P(D1) = 0.05, P(D2) = 0.05, and the packets are dropped independently.

  • Verify independence is stated. The problem says "independently" — this is an assumption we accept. In practice, you would verify independence from data; here it is given.
  • Compute P(both packets dropped). Because D1 and D2 are independent: P(D_1 \cap D_2) = P(D_1) \cdot P(D_2) = 0.05 \times 0.05 = 0.0025
  • Compute P(at least one packet dropped). Use the complement: P(D_1 \cup D_2) = 1 - P(D_1^c \cap D_2^c) = 1 - (0.95)(0.95) = 1 - 0.9025 = 0.0975 (The complement step also uses independence: P(D1c ∩ D2c) = P(D1c) · P(D2c) because independence of the original events implies independence of their complements.)
  • Interpret. There is only a 0.25% chance both packets are dropped, but nearly a 10% chance at least one is lost. This is why network protocols use acknowledgment and retransmission: even small individual drop rates compound into noticeable loss rates across many packets.

Takeaway: Independence allows you to multiply individual probabilities directly. This simplification is powerful — and it is only valid when independence actually holds.

6. Watch Out: Mutual Exclusivity vs. Independence

⚠️ Most Common Misconception in Introductory Probability

Students frequently confuse mutually exclusive events with independent events. These are not the same concept — in fact, for events with positive probability, they are mutually exclusive if and only if they are dependent. The table below shows the key differences.

Intuition They compete — only one can win. They are unrelated — one's outcome tells you nothing about the other.

Counterexample: Mutually Exclusive Events Are Dependent

Roll a fair six-sided die. Let A = "roll a 1" and B = "roll a 2."

  • A ∩ B = ∅, so A and B are mutually exclusive.
  • P(A) = 1/6, P(B) = 1/6, so P(A) · P(B) = 1/36.
  • But P(A ∩ B) = 0 ≠ 1/36.

Therefore A and B are not independent. Intuitively: if you know a 2 was rolled (B occurred), you know for certain a 1 was not rolled — so knowing B absolutely changes your assessment of A.

🔗 Key Insight to Remember

Mutually exclusive events with positive probability are always dependent — knowing one occurred tells you the other definitely did not. True independence requires the events to be able to co-occur, with their joint probability exactly equal to the product of their individual probabilities.

8. Module Summary

📋 Module 9.2 — Key Takeaways
  • Conditional probability P(A | B) = P(A ∩ B) / P(B) formalizes "restricting the sample space to B." The multiplication rule P(A ∩ B) = P(A | B) · P(B) follows immediately by rearrangement and lets you compute joint probabilities from conditional ones. The law of total probability sums contributions across a partition when only conditional probabilities are known.
  • Two events are independent when P(A ∩ B) = P(A) · P(B), equivalently P(A | B) = P(A): knowing one event occurred provides no information about the other. Independence must be verified from probabilities — never assumed without justification.
  • Mutual exclusivity and independence are not the same and are in fact incompatible for events with positive probability. Mutually exclusive events (A ∩ B = ∅) are always dependent because knowing one occurred rules out the other. Keep the comparison table from Section 6 as a reference.
