1. Probability in Computing Applications

The probability tools you have built this week — sample spaces, conditional probability, expected value — are not abstract exercises. They are the mathematical foundation for analyzing how algorithms behave on real inputs. This section applies the Week 9 toolkit to three computing contexts: average-case algorithm analysis, randomized algorithms, and the birthday paradox and its connection to hashing.

1A. Average-Case Algorithm Analysis

When we analyze an algorithm, we typically ask three questions about its running time. These correspond directly to probability concepts from Module 9.4:

Case Meaning Probability Connection
Best case Minimum possible steps over all inputs of size n The outcome with the smallest cost — one element of the sample space
Worst case Maximum possible steps — the traditional Big-O bound The outcome with the largest cost — a bound, not an expectation
Average (expected) case Expected number of steps over a probability distribution on inputs Expected value E[X] from Module 9.4
📘 Linear Search — Expected Case

In linear search, we scan an array of n elements one by one looking for a target. Assume the target is present and equally likely to be in any position 1, 2, \ldots, n.

The cost of finding the target at position i is i comparisons. Since each position is equally likely with probability 1/n, the expected number of comparisons is:

E[comparisons] = 1n · (1 + 2 + 3 + ⋯ + n) = 1n · n(n + 1)2 = n + 12

So on average, linear search finds the target after examining about half the array. The worst case is n comparisons (target is last); the best case is 1 (target is first).

🔗 Connection — Module 9.4 Expected Value
The formula above is exactly the expected value of a discrete random variable X that takes values 1, 2, \ldots, n each with probability 1/n. The summation 1 + 2 + \cdots + n = n(n+1)/2 is the arithmetic series identity from Week 5. Expected value, counting, and summation all converge here.

Randomized Quicksort: Expected vs. Worst Case

Standard quicksort chooses a fixed pivot rule (e.g., always the first element). On an already-sorted array, this produces the worst case: O(n^2) comparisons. Randomized quicksort selects the pivot uniformly at random from the current subarray, breaking the adversarial pattern.

The informal argument for the expected case uses linearity of expectation (Module 9.4). Define an indicator random variable X_{ij} = 1 if elements i and j are ever compared during the sort, and 0 otherwise. The total comparisons equal \sum_{i < j} X_{ij}. By linearity of expectation and a careful probability argument, the expected total is:

E[total comparisons] = O(n log n)

The worst case is still O(n^2) — it can happen, just with vanishingly small probability for a random pivot. This is the defining characteristic of a Las Vegas algorithm (see Section 1B below): the answer is always correct, but the running time is a random variable.

Algorithm Complexity Comparison

Algorithm Best Case Expected (Average) Case Worst Case Notes
Linear Search O(1) O(n) — specifically (n+1)/2 comparisons O(n) Unsorted array; target present
Randomized Quicksort O(n \log n) O(n \log n) — via linearity of expectation O(n^2) Worst case has negligible probability with random pivot
Binary Search O(1) O(\log n) O(\log n) Requires sorted array; deterministic
⚠️ Watch Out — Average Case ≠ Typical Case
"Average case" means the expected value over a probability distribution on inputs. If real-world inputs are not drawn from that distribution (e.g., nearly-sorted data is common in your application), the average-case bound may not reflect actual performance. Always ask: what distribution was assumed?

1B. Randomized Algorithms: Las Vegas vs. Monte Carlo

Randomized algorithms use random choices as part of their logic. There are two fundamentally different contracts they can offer:

📘 Definition — Las Vegas Algorithm

A Las Vegas algorithm always produces a correct answer. Its running time is a random variable — it may be fast or slow depending on random choices, but it never returns a wrong answer.

Canonical example: Randomized quicksort. The sorted output is always correct; only the number of comparisons varies randomly.

📘 Definition — Monte Carlo Algorithm

A Monte Carlo algorithm runs in deterministic (or bounded) time but may return an incorrect answer with some small error probability \varepsilon.

Canonical example: The Miller-Rabin primality test. Given a number n, it runs in polynomial time and declares n prime or composite. If it says "composite," it is always correct. If it says "prime," it is correct with probability at least 1 - \varepsilon for a single round.

Key property — error reduction by repetition: If a single round has error probability \varepsilon, then running k independent rounds reduces the error probability to at most \varepsilon^k. For \varepsilon = 1/4 and k = 10, the error probability is at most (1/4)^{10} \approx 10^{-6} — negligible in practice.

📘 Error Reduction Formula
P(\text{error after } k \text{ rounds}) \leq \varepsilon^k

This follows directly from the independence rule (Module 9.2): if rounds are independent and each has error probability \varepsilon, the probability that all k rounds err is \varepsilon \cdot \varepsilon \cdots \varepsilon = \varepsilon^k.

🔗 Connection — Bayes' Theorem and Algorithm Correctness

When analyzing the correctness of a Monte Carlo algorithm in practice, Bayes' theorem (Module 9.3) provides a more nuanced picture. If we have prior knowledge about how often inputs are truly prime (or truly in some class), we can compute the posterior probability that the algorithm's output is correct given that output. A "probably prime" declaration from Miller-Rabin carries a different real-world meaning depending on whether we are testing random integers or specially constructed adversarial ones.

▶ Las Vegas vs. Monte Carlo — Side-by-Side Comparison
Property Las Vegas Monte Carlo
Correctness Always correct Correct with probability 1 - \varepsilon
Running time Random variable (expected bound given) Deterministic or bounded
Error control No error to control Run k times: error ≤ \varepsilon^k
Canonical example Randomized quicksort Miller-Rabin primality test
When preferred Correctness is non-negotiable Speed is critical; small error is acceptable

1C. The Birthday Paradox and Hashing

The birthday paradox asks: how many people must be in a room before the probability that at least two share a birthday exceeds 50%? The surprising answer is just 23 people — far fewer than most people expect.

Using the complement rule (Module 9.1), the probability that all k people have distinct birthdays (out of 365 equally likely days) is:

P(\text{all distinct}) = 365365 · 364365 · 363365 ⋯ 365 − k + 1365

So P(\text{at least one shared birthday}) = 1 - P(\text{all distinct}). At k = 23, this exceeds 0.50; at k = 57 it exceeds 0.99.

🔗 Connection — Week 8 Discussion and Hashing

You discussed the birthday paradox in the Week 8 discussion board in the context of counting. Here we see its probabilistic formulation and its direct application in computer science.

Hash table collisions: Replace 365 with the hash table size N. The birthday paradox tells us that after inserting approximately \sqrt{N} keys, the probability of at least one collision exceeds 50%. For a table of size N = 10{,}000, collisions become likely after only about \sqrt{10{,}000} = 100 insertions — long before the table is full. This is why good hash functions and collision-resolution strategies matter even in lightly loaded tables.

2. Monte Carlo Simulation Activity Ungraded

A Monte Carlo simulation uses random sampling to estimate a mathematical quantity that may be difficult or impossible to compute directly. The key idea: if a quantity equals the probability of some geometric or combinatorial event, we can simulate that event millions of times and use the observed frequency as an estimate.

Estimating π by Random Sampling

Consider the unit square [0,1] \times [0,1] and the quarter of the unit circle centered at the origin that lies inside it. A point (x, y) drawn uniformly at random from the square falls inside the quarter-circle if and only if x^2 + y^2 \leq 1.

The area of the quarter-circle is \pi/4, and the area of the unit square is 1, so:

P(x^2 + y^2 \leq 1) = π/41 = π4   ⟹   π ≈ 4 · points inside circle total points sampled

By the Law of Large Numbers, as the number of samples grows, this estimate converges to \pi. The simulation below lets you see this convergence in action.

⚠️ Common Pitfalls — PS-06
  • Forgetting to verify mutual exclusivity before adding probabilities. The simple addition rule P(A \cup B) = P(A) + P(B) only applies when A \cap B = \emptyset. In general, subtract P(A \cap B).
  • Confusing P(A|B) with P(B|A). These are not equal in general. Bayes' theorem exists precisely to convert one into the other. Write out the formula before substituting numbers.
  • Not identifying the correct sample space. Before computing any probability, write \Omega explicitly (or state its size and confirm uniformity). Many errors trace back to an incorrectly defined sample space.
  • Skipping the Law of Total Probability when applying Bayes' theorem. The denominator P(E) usually requires expanding via P(E) = P(E|H)P(H) + P(E|H^c)P(H^c).
  • Reporting expected value without units or interpretation. State what the expected value represents in context, not just the number.

6. Module Summary

📋 Module 9.5 — Key Takeaways
  • Probability powers algorithm analysis. The expected case of an algorithm is the expected value of its cost random variable. For linear search, this gives (n+1)/2 comparisons; for randomized quicksort, linearity of expectation yields O(n \log n) expected comparisons versus an O(n^2) worst case.
  • Las Vegas algorithms are always correct; Monte Carlo algorithms trade a small error probability for bounded runtime. Repeating a Monte Carlo algorithm k times drives error probability to \varepsilon^k — an application of independence from Module 9.2.
  • Monte Carlo simulation estimates quantities by random sampling. The π estimation activity demonstrates that accuracy improves slowly (roughly as 1/\sqrt{n}) — a fundamental accuracy/computation trade-off that reappears in the Probability Calculator module of Project 2.
