Proving Recursive Algorithms Correct:
Recursion and Mathematical Induction

Modules 11.1 and 11.2 introduced recursive algorithm design and execution tracing. This module reveals why recursive algorithms work — and why mathematical induction is not merely a proof technique applied to algorithms, but the very logical structure that makes recursive correctness arguments possible.

CSLO 4 Prereqs: Modules 11.1 & 11.2 Core: PS-07
1

The Structural Parallel

Mathematical induction and recursive algorithms are isomorphic in structure: every component of an induction proof maps directly onto a corresponding component of the recursive algorithm. This is not an analogy we impose — it is a logical identity we discover. The table below makes the correspondence explicit.

Table 1 — Induction Structure ↔ Recursive Algorithm Structure
Component Mathematical Induction Recursive Algorithm
Base Case Prove P(0) — establish the claim for the smallest value directly, without using any prior instance of the claim. Handle the simplest input (e.g., n = 0) directly — return a result with no recursive call.
Inductive Hypothesis Assume P(k) is true for some arbitrary k ≥ 0. You borrow this truth temporarily — you are not proving it here. Assume the recursive call on a strictly smaller input already returns the correct answer. The correctness proof borrows this assumption.
Inductive Step Using P(k), prove P(k+1). Show constructively how truth at k forces truth at k+1. Show that the current call (input size k+1) correctly combines the recursive result (input size k) with the current input to produce the right answer.
Conclusion By the principle of induction, P(n) is true for all n ≥ 0. The algorithm returns the correct result for all valid inputs.
🔑 Key Insight

This is not merely an analogy. When you write a correctness proof for a recursive algorithm, you are literally constructing a proof by mathematical induction. The base case of the proof is the base case of the algorithm. The inductive hypothesis is the assumption that the recursive call is correct. The inductive step is the argument that the algorithm correctly computes its result using that recursive call.

Proving the algorithm correct IS the induction proof.

ℹ️ Why This Shapes How You Code

Knowing this parallel enforces disciplined design. Before writing any recursive function, ask: Have I identified the base case? Is my recursive call on a genuinely smaller input? Does combining the recursive result with the current state produce the right answer? These are the exact questions a correctness proof must answer — and they are the same questions that prevent bugs.

Correctness Proofs — Two Worked Examples

Each proof follows the same four-part structure: Base Case → Inductive Hypothesis → Inductive Step → Conclusion. These labels are not decorative — each satisfies a specific, distinct logical obligation.

Proof 1 — Recursive Factorial

function factorial(n):
  if n = 0 then
    return 1                          -- base case
  else
    return n × factorial(n − 1)       -- recursive case
Claim For all integers n ≥ 0, factorial(n) returns n!
Proof — by weak induction on n
Base Case (n = 0)

The if-branch executes, returning 1.

By definition, 0! = 1.

Therefore factorial(0) = 1 = 0!  ✓

Inductive Hypothesis

Assume factorial(k) = k! for some arbitrary integer k ≥ 0.

Inductive Step (show factorial(k+1) = (k+1)!)

Since k+1 > 0, the else-branch executes:

factorial(k+1) = (k+1) × factorial(k) else-branch; k+1 > 0
= (k+1) × k! Inductive Hypothesis
= (k+1)! definition: (k+1)! = (k+1) · k!

So factorial(k+1) = (k+1)!  ✓

Conclusion

By the principle of mathematical induction, factorial(n) = n! for all integers n ≥ 0.

□

Proof 2 — Euclid’s GCD Algorithm

function gcd(a, b):
  if b = 0 then
    return a                            -- base case
  else
    return gcd(b, a mod b)             -- recursive case
Claim For all integers a ≥ 0 and b ≥ 0, gcd(a, b) returns the mathematical greatest common divisor of a and b.
ℹ️ Proof Strategy

We induct on b (the second argument). The recursive call passes a mod b as the new second argument. Because a mod b can be any value from 0 to b−1 — not necessarily b−1 exactly — we need strong induction: assume correctness for all second-argument values strictly less than b.

Proof — by strong induction on b
Base Case (b = 0)

The if-branch executes, returning a.

By the mathematical definition, gcd(a, 0) = a for any a ≥ 0.  ✓

Inductive Hypothesis

Assume that for all a′ ≥ 0 and all b′ with 0 ≤ b′ < b, the call gcd(a′, b′) correctly returns gcd(a′, b′).

Inductive Step (b > 0)

Step 1 — Validate the recursive call.
Since b > 0, the property of integer division gives: 0 ≤ a mod b < b. So the second argument of the recursive call is strictly less than b. The IH applies. ✓

Step 2 — Apply the IH.
The else-branch executes. Set a′ = b and b′ = a mod b (where b′ < b). By the Inductive Hypothesis, the recursive call gcd(b, a mod b) correctly returns the mathematical value gcd(b, a mod b).

Step 3 — Apply the GCD identity.
The mathematical identity states: gcd(a, b) = gcd(b, a mod b) for all b > 0. Therefore:

value returned = gcd(b, a mod b) Step 2: IH applied to recursive call
= gcd(a, b) GCD identity; b > 0

Therefore gcd(a, b) returns gcd(a, b).  ✓

Conclusion

By strong induction on b, gcd(a, b) returns gcd(a, b) for all a ≥ 0 and b ≥ 0.

□
⚠️ Common Pitfall — GCD Inductive Step

Step 1 of the inductive step — establishing that 0 ≤ a mod b < b — is not optional detail. It is the prerequisite that permits the application of the strong inductive hypothesis. Omitting it leaves the IH application unjustified and the proof incomplete.

Observe also that this same inequality guarantees termination: the second argument strictly decreases toward 0 on every call. The inductive step doubles as the termination argument. This connection is developed further in Section 3.

3

Proving Termination

A correctness proof establishes that if the algorithm terminates, it returns the right answer. But we also need to prove that it does terminate. Termination and correctness are separate logical obligations — neither implies the other. The standard technique is the termination measure argument.

🔑 Definition — Termination Measure

A termination measure (also called a ranking function) for a recursive algorithm is a function m that maps each possible input to a non-negative integer and satisfies:

  1. Non-negative: m(input) ≥ 0 for every valid input.
  2. Strictly decreasing: For every recursive call, m(arguments of the recursive call) < m(current arguments).

If such a measure exists, the algorithm terminates on all valid inputs.

ℹ️ Why This Guarantees Termination — Well-Ordering

A non-negative integer cannot decrease strictly forever. This follows from the Well-Ordering Principle: every non-empty set of non-negative integers has a minimum element. A strictly decreasing sequence of non-negative integers is therefore finite — it must eventually reach the minimum, at which point the base case fires and no further recursive call is made.

Worked Example 1 — Factorial Termination

Proposed measure: m(n) = n

  1. Non-negative: All valid inputs satisfy n ≥ 0, so m(n) = n ≥ 0. ✓
  2. Base case fires at 0: When n = 0, factorial returns immediately with no recursive call. ✓
  3. Strictly decreasing: The recursive call is factorial(n−1). Its measure is m(n−1) = n−1 < n = m(n). ✓

Conclusion: By the termination measure argument, factorial terminates for all n ≥ 0.

Worked Example 2 — GCD Termination

Proposed measure: m(a, b) = b (the second argument)

  1. Non-negative: All valid inputs satisfy b ≥ 0, so m(a, b) = b ≥ 0. ✓
  2. Base case fires at 0: When b = 0, gcd returns a immediately with no recursive call. ✓
  3. Strictly decreasing: The recursive call is gcd(b, a mod b). Its measure is m(b, a mod b) = a mod b.
    • Since b > 0 in the recursive case: 0 ≤ a mod b < b.
    • Therefore m(b, a mod b) = a mod b < b = m(a, b). ✓

Conclusion: By the termination measure argument, gcd terminates for all a ≥ 0 and b ≥ 0.

💡 Strategy Tip

For most algorithms in this course, the termination measure is the value of one integer argument or a simple expression over it. Start by asking: which parameter gets strictly smaller on every recursive call? If more than one parameter changes, look for one that is (a) bounded below by 0 and (b) strictly decreases each call. If no single argument works, consider sums, maximums, or lexicographic pairs of arguments.

Structural Induction — A Conceptual Introduction

Sections 1–3 used induction over the natural numbers (n = 0, 1, 2, …). This is natural when a recursive call decrements an integer argument. But many algorithms in computer science operate on structured data — lists, trees, binary strings, expressions — whose “size” is determined by their recursive structure, not a single integer counter.

For such settings we use structural induction: rather than inducing over a numeric index, we induct over the recursive definition of the data object itself.

🔑 Structural Induction — Core Pattern

Let S be a recursively defined set and P a property of its elements. To prove P holds for every element of S:

  1. Base case: Prove P holds for all atomic (smallest) elements of S.
  2. Inductive step: Prove that if P holds for every sub-component of a compound element, then P holds for the whole element.

Validity rests on the well-foundedness of the recursive definition: every element is eventually decomposed into atomic parts through a finite number of steps.

Example — Binary Strings

Let Σ = {0, 1}. The set Σ* of binary strings is defined recursively:

  • Basis: ϵ (the empty string) ∈ Σ*.
  • Recursive rule: If w ∈ Σ*, then 0w ∈ Σ* and 1w ∈ Σ*.
Claim For every w ∈ Σ*,  length(reverse(w)) = length(w).
Proof — by structural induction on w
Base Case (w = ϵ)

reverse(ϵ) = ϵ  (reversing the empty string gives the empty string).

length(reverse(ϵ)) = length(ϵ) = 0. ✓

Inductive Hypothesis

Assume length(reverse(w′)) = length(w′) for some arbitrary w′ ∈ Σ*.

Inductive Step (w = cw′, c ∈ {0,1})
length(reverse(cw′)) = length(reverse(w′) · c) def. of reverse: rev(cw) = rev(w) · c
= length(reverse(w′)) + 1 length of concatenation
= length(w′) + 1 Inductive Hypothesis
= length(cw′) def. of length: len(cw) = len(w) + 1

So length(reverse(cw′)) = length(cw′). ✓

Conclusion

By structural induction, length(reverse(w)) = length(w) for every w ∈ Σ*.

□
🗓️ Forward Reference — Trees (Weeks 12–15)

Structural induction generalizes naturally to richer data structures. In Weeks 12–15 you will work with binary trees, expression trees, and parse trees. Properties such as tree height, node count, and traversal correctness are all established by structural induction on the tree definition. The pattern is the same as here: base case on leaf nodes, inductive step on internal nodes using the sub-tree hypotheses. Keep this string example as your concrete reference point when those proofs arise.

Mini-Case: Find the Error

The following is a complete-looking correctness proof — but it contains a subtle flaw in the inductive step. Your task is to identify the error, explain precisely why the proof fails, and describe how to correct it. This activity is ungraded and is direct preparation for PS-07.

🔍 Activity — Identify the Flaw in This Proof

Consider this recursive algorithm for computing Fibonacci numbers:

function fib(n):
  if n = 0 then return 0
  if n = 1 then return 1
  else          return fib(n − 1) + fib(n − 2)
Claim For all n ≥ 0, fib(n) = Fn, where F0 = 0, F1 = 1, and Fn = Fn−1 + Fn−2 for n ≥ 2.

A student submits the following proof:

Base Case (n = 0):
fib(0) = 0 = F0. ✓

Inductive Hypothesis:
Assume fib(k) = Fk for some arbitrary k ≥ 0.

Inductive Step (show fib(k+1) = Fk+1):

For k+1 ≥ 2, the else-branch executes:

fib(k+1) = fib(k) + fib(k−1)
= Fk + fib(k−1) Inductive Hypothesis
= Fk + Fk−1 … (see note below)
= Fk+1 definition of Fibonacci numbers

The “…” in line 3 silently replaces fib(k−1) with Fk−1. The student treats this as justified by the Inductive Hypothesis.

Conclusion: By induction, fib(n) = Fn for all n ≥ 0. □

Your Task

In the response box, address all three questions:

  1. What is the specific logical error in the inductive step?
  2. Why does the stated Inductive Hypothesis fail to justify replacing fib(k−1) with Fk−1?
  3. What corrected form of inductive hypothesis would fix the proof? State it precisely.

Hint: The IH says “for some arbitrary k ≥ 0.” What exactly does that give you — and what does it not give you?

The Error: Weak Induction Cannot Support a Two-Step Recursion

The stated hypothesis — “Assume fib(k) = Fk for some arbitrary k ≥ 0” — is weak induction. It asserts the claim for exactly one specific value, k.

The inductive step, however, requires two facts simultaneously:

  • fib(k) = Fk  ← provided by the IH ✓
  • fib(k−1) = Fk−1  ← NOT provided by the IH ✗

The IH says nothing about k−1. Line 3 silently substitutes Fk−1 for fib(k−1) without any logical basis. The proof simply assumes what it needs to prove, making the argument circular and invalid.

Fixes:

  • Strong induction: Assume fib(j) = Fj for all j with 0 ≤ j ≤ k. This gives both fib(k) = Fk (setting j = k) and fib(k−1) = Fk−1 (setting j = k−1) as special cases.
  • Two-step induction: Prove both n = 0 and n = 1 as base cases (both already verified in the student's proof), and take the hypothesis to be the joint assumption fib(k) = Fk and fib(k−1) = Fk−1.

General principle: match the strength of your induction to the depth of your recursion. A two-step recursion (calling at both k and k−1) requires a two-step — or strong — inductive hypothesis.

function collatz(n):
  if n ≤ 1 then return n

Since 3n+1 > n, the measure m(n) = n strictly increases on this branch. The proposed measure fails immediately on any odd input greater than 1.
✓ Correct! m(n) = n fails because 3n+1 > n for any odd n > 0. This is the Collatz conjecture (1937): despite being verified for every tested input up to at least 268, no proof of universal termination has ever been found. It is one of the most celebrated open problems in all of mathematics. The key lesson: empirical testing is not a termination proof. A function can behave correctly on every tested input and still lack a valid formal termination argument.

Module Summary & Bridge to Module 11.4

Module 11.3 established the logical core of recursive algorithm verification. Carry these six ideas forward:

The Structural Parallel

Proving a recursive algorithm correct is constructing an induction proof. Base case ↔ algorithm base; IH ↔ recursive-call assumption; inductive step ↔ combining result with current input.

Four-Part Proof Format

Every correctness proof must label all four parts: Base Case, Inductive Hypothesis, Inductive Step, Conclusion. Each satisfies a distinct logical obligation and is not cosmetic.

Termination is Separate

Correctness (right answer, given termination) and termination (the algorithm stops) are independent obligations. A non-negative, strictly decreasing integer measure guarantees termination.

Match IH to Recursion Depth

A two-step recursion (calling at depths k and k−1) requires strong induction or a two-step hypothesis. Weak induction provides only a single-step guarantee and is insufficient.

Structural Induction

For algorithms over structured data (strings, trees), induct on the object’s structure: base case on atomic elements; step on compound objects using sub-component hypotheses.

Termination Can Be Hard

Some recursive functions resist all formal termination arguments (Collatz). Empirical testing is never a substitute for a proof — this is precisely why the termination measure argument exists.
