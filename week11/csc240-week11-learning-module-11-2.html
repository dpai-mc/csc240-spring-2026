💻 Classic Recursive Algorithms

Four algorithms appear in virtually every discrete structures course because they cleanly illustrate different recursive structures: Factorial (linear recursion), Fibonacci (binary / branching recursion), Euclid's GCD (tail recursion — previewed here, detailed in Section 3), and Towers of Hanoi (exponential recursion that produces output rather than returning a computed value).

For each algorithm: pseudocode → collapsible Java implementation → hand-trace table showing every call, return value, and stack depth.

✨ Factorial — f(n) = n!

Pseudocode

FACTORIAL(n):
  if n = 0
    return 1                     // base case: 0! = 1
  return n × FACTORIAL(n − 1) // recursive case
/** Returns n! for n >= 0. */
public static long factorial(int n) {
    if (n < 0) throw new IllegalArgumentException("n must be ≥ 0");
    if (n == 0) return 1;            // base case
    return n * factorial(n - 1);   // recursive case
}

Hand Trace — factorial(4)

The call stack grows as frames are pushed (↓ descend), then unwinds as each frame returns its value (↑ return). Every intermediate frame holds a pending multiplication that cannot complete until the recursive call below it returns.

📈 Fibonacci — F(n) = F(n−1) + F(n−2)

Pseudocode

FIB(n):
  if n = 0  return 0              // base case 1
  if n = 1  return 1              // base case 2
  return FIB(n − 1) + FIB(n − 2) // binary recursion — two sub-calls
/** Returns the n-th Fibonacci number. F(0)=0, F(1)=1. */
public static int fib(int n) {
    if (n == 0) return 0;
    if (n == 1) return 1;
    return fib(n - 1) + fib(n - 2);
}

🔢 Euclid's GCD — gcd(a, b)

Pseudocode

GCD(a, b):
  if b = 0   return a       // base case: gcd(a, 0) = a
  return GCD(b, a mod b)   // recursive case — this is a tail call
/** Returns the greatest common divisor of a and b (a, b >= 0). */
public static int gcd(int a, int b) {
    if (b == 0) return a;      // base case
    return gcd(b, a % b);     // tail call — no pending work after this
}

Hand Trace — gcd(48, 18)

GCD is a linear recursion — each call makes exactly one recursive call. Because the recursive call is the only thing returned, there is no pending arithmetic. This makes GCD naturally tail-recursive (Section 3). Notice how the answer 6 propagates back unchanged through every unwind frame.

🔎 Why the Answer Propagates Unchanged

Once the base case returns 6, each frame above it simply passes 6 upward — no computation is needed. This is the hallmark of tail recursion: the deepest return value flows back through every waiting frame without modification. A compiler with tail-call optimization collapses all four frames into one (see Section 3).

🏛️ Towers of Hanoi — hanoi(n, from, to, via)

Hanoi does not return a computed value — it produces output (the sequence of disk moves). The structure is: to move n disks from peg from to peg to, first move the top n−1 disks to the temporary peg via, then move the large disk, then move the n−1 disks from via to to.

Pseudocode

HANOI(n, from, to, via):
  if n = 1
    print "Move disk 1 from " + from + " to " + to
    return
  HANOI(n − 1, from, via, to)    // move top n-1 out of the way
  print "Move disk " + n + " from " + from + " to " + to
  HANOI(n − 1, via, to, from)    // move top n-1 to destination
/** Prints the moves to transfer n disks from 'from' to 'to' using 'via'. */
public static void hanoi(int n, char from, char to, char via) {
    if (n == 1) {
        System.out.println("Move disk 1 from " + from + " to " + to);
        return;
    }
    hanoi(n - 1, from, via, to);   // move top n-1 to via
    System.out.println("Move disk " + n + " from " + from + " to " + to);
    hanoi(n - 1, via, to, from);   // move n-1 from via to to
}

🔐 Exponential Growth

For n disks, Hanoi requires exactly 2n − 1 moves. Moving 64 disks would require 264 − 1 ≈ 1.8 × 1019 moves — at one move per second, roughly 585 billion years. This is the concrete meaning of exponential time complexity.

🌳 Recursion Tree Diagrams

What Is a Recursion Tree?

A recursion tree is a rooted tree where each node represents one function call, and the children of a node are the recursive sub-calls it spawns. Reading top-down shows how calls multiply; reading bottom-up shows how results propagate back to the root.

📝 Anatomy of a Recursion Tree
  • Root: the initial call (e.g., fib(4)).
  • Internal nodes: calls that make further recursive calls.
  • Leaves: base-case calls — they return without recursing.
  • Depth / level L: the number of edges from the root to a node.
  • Total nodes: the total number of function invocations.

How to Draw a Recursion Tree

  1. Root: write the initial call at the top.
  2. Expand: draw one child for each recursive call the current node makes.
  3. Label: annotate each node with its argument(s) and return value.
  4. Stop at leaves: base-case calls are leaves; mark them (e.g., ✓).
  5. Annotate returns: show how values propagate back toward the root.

⚙️ Tail Recursion and Tail-Call Optimization

📖 Definition — Tail Call & Tail Recursion

A recursive call is a tail call if it is the last operation performed before the function returns — there is no pending work (no arithmetic, no assignment, no further computation) that must happen after the call returns.

A function is tail-recursive if every recursive call it contains is a tail call.

Quick Test: Is It a Tail Call?

Return StatementTail Call?Reason
return gcd(b, a % b);✅ YesNothing happens after the call returns.
return n * factorial(n-1);❌ NoMultiplication by n is pending after the call.
return fib(n-1) + fib(n-2);❌ NoAddition (and the second call) are pending.
return helper(n-1, acc+n);✅ YesAll computation is done before passing as argument; nothing remains after.

Side-by-Side: GCD — Tail-Recursive vs. Iterative

Because GCD is already tail-recursive, converting it to an iterative version is mechanical: replace the recursive call with a loop that updates the parameters directly.

🔄 Tail-Recursive GCD
public static int gcd(int a, int b) {
    if (b == 0) return a;
    return gcd(b, a % b); // ◀ tail call
}
🔄 Iterative GCD
public static int gcdIter(int a, int b) {
    while (b != 0) {
        int tmp = b;
        b = a % b;
        a = tmp;
    }
    return a;
}
💡 The Conversion Pattern for Tail Recursion
  • The base case (if b == 0 return a) becomes the loop exit condition (while b != 0).
  • The tail call gcd(b, a%b) becomes a parameter update: store b in a temp, update b = a%b, then a = tmp.
  • The loop body executes exactly what the recursive step used to do.

Tail-Call Optimization (TCO)

Many compilers and runtimes (Scheme, Haskell, modern JavaScript engines) apply tail-call optimization: recognizing that a tail-recursive call will never need the current frame again, the runtime reuses the current stack frame instead of pushing a new one — turning recursion into an implicit loop.

⚠️ Java Does Not Automatically Apply TCO

The standard JVM does not perform tail-call optimization for Java source code. A tail-recursive Java method still allocates a new stack frame per call. For large inputs this risks a StackOverflowError. When stack depth or performance matters, convert tail-recursive Java to an explicit loop, as shown for GCD above.

🔄 Converting Recursion to Iteration

Every recursive algorithm can be rewritten as an iterative one. For tail-recursive functions the conversion is mechanical (Section 3). For non-tail-recursive functions, two strategies cover most cases: the accumulator pattern (for linear recursion) and the rolling-variable pattern (for two-term recurrences).

✨ Factorial — Accumulator Pattern

The key insight: instead of building the product on the way back up the call stack, carry a running acc variable that accumulates the product on the way down (i.e., inside the loop). The base-case value seeds the accumulator.

Step-by-Step Conversion

  1. Seed the accumulator at the base-case value: factorial(0) = 1, so acc = 1.
  2. Identify the step computation: the recursive case multiplies by n and decrements toward 0. In the loop, multiply acc ×= i for each i from 1 to n.
  3. Loop while the base case is not yet reached: for (int i = 1; i <= n; i++).
  4. Return the accumulator after the loop exits.
Recursive Factorial
public static long factorial(int n) {
    if (n == 0) return 1;
    return n * factorial(n - 1);
    // pending × n on every frame
}
Iterative Factorial (Accumulator)
public static long factIter(int n) {
    long acc = 1;  // seed = 0! = 1
    for (int i = 1; i <= n; i++) {
        acc *= i;  // acc holds i! after each step
    }
    return acc;
}

📈 Fibonacci — Rolling-Variable Pattern

Fibonacci cannot use a single accumulator because each term depends on two preceding values. Instead, maintain two variables (prev2 and prev1) as a sliding window through the sequence: at every step they hold the two most recently computed values.

Step-by-Step Conversion

  1. Handle base cases directly: if (n <= 1) return n — returns 0 for n=0, 1 for n=1.
  2. Initialize the window at the two base cases: prev2 = 0 (F0) and prev1 = 1 (F1).
  3. Loop from i = 2 to n: compute curr = prev1 + prev2, then shift the window: prev2 ← prev1, prev1 ← curr.
  4. Return prev1 after the loop — it holds Fn.
Recursive Fibonacci
public static int fib(int n) {
    if (n <= 1) return n;
    return fib(n-1) + fib(n-2);
    // O(2ⁿ) calls; redundant work
}
Iterative Fibonacci (Rolling Vars)
public static int fibIter(int n) {
    if (n <= 1) return n;
    int prev2 = 0, prev1 = 1;
    for (int i = 2; i <= n; i++) {
        int curr = prev1 + prev2;
        prev2 = prev1;
        prev1 = curr;
    }
    return prev1; // O(n); no redundancy
}

⚠️ When an Explicit Stack Is Required

The accumulator and rolling-variable patterns work because Factorial and Fibonacci have small, well-defined state — one or two scalar values. Algorithms with complex branching (graph DFS, tree traversal, backtracking search) cannot be collapsed into scalar variables.

For those cases, replace the implicit call stack with an explicit stack data structure (e.g., Stack<T> in Java). Push subproblems onto the stack instead of making recursive calls, and pop them when ready to process. You will apply this technique when implementing iterative DFS in Week 13.

📊 Complexity Preview (Informal)

Before Module 11.4 introduces recurrence relations as a formal tool, you can estimate time complexity directly from the recursion tree.

📝 The Counting Rule

Each node in the recursion tree represents one function call that does O(1) work (ignoring the recursive sub-calls themselves). Therefore:

Time complexity ≈ O(total number of nodes in the recursion tree)

Algorithm Tree Shape Branching Total Calls (n) Complexity
factorial(n) Linear chain 1 child per node n + 1 O(n)
gcd(a, b) Linear chain 1 child per node O(log n) O(log n)
fib(n) (naïve) Binary tree 2 children per internal node ≈ 2n O(2n)
hanoi(n) Full binary tree 2 children per internal node 2n+1 − 1 O(2n)

Why Fibonacci Is O(2n)

At each level of the fib(n) tree, the number of nodes roughly doubles because every internal node spawns exactly two children. Starting from 1 node (the root), after k levels there are at most 2k nodes. The tree has depth n, so total nodes ≤ 20 + 21 + … + 2n = 2n+1 − 1 = O(2n).

The iterative fibIter runs exactly n − 1 loop iterations — O(n). Eliminating redundant sub-calls collapses exponential cost to linear. This is the formal cost of the repeated sub-problems identified in the recursion tree diagram.

Why Factorial Is O(n)

Factorial's tree is a straight chain: each call spawns exactly one recursive call (no branching). There are n + 1 total calls, each doing O(1) work → O(n) total.

👀 Bridge to Module 11.4

The informal counting above becomes a formal proof via recurrence relations. The recurrence for factorial is T(n) = T(n − 1) + c, which solves to T(n) = O(n). The recurrence for naïve Fibonacci is T(n) = T(n − 1) + T(n − 2) + c, which solves to T(n) = O(2n). Module 11.4 introduces the characteristic equation method that derives these closed-form solutions exactly, and Module 11.5 adds the Master Theorem for divide-and-conquer algorithms like merge sort.
