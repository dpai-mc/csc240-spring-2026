Recursive Definitions and Structures

Mendocino College · Spring 2026 · Foundation for recursion, recurrence relations, and recursive algorithms

1 · What Is a Recursive Definition?

A recursive definition defines an object — a function, a sequence, or a set — partly in terms of itself. This is not circular because the self-reference always points to a simpler or smaller version of the same object, and a separate base case provides a direct answer that stops the chain.

📘 Definition
A recursive definition consists of exactly two parts:
  1. Base case — gives the value (or membership) directly for the simplest input(s), with no self-reference whatsoever.
  2. Recursive case — expresses the value (or membership) for a general input in terms of the same function or set applied to a strictly smaller input.

The Two-Part Structure

Part 1
Base Case
Anchors the definition. Gives a direct answer for the smallest input — no self-reference, no infinite chain.
Part 2
Recursive Case
Reduces the problem. Expresses a larger input in terms of a smaller one, always moving toward the base case.
🔑 Well-Definedness Principle
A recursive definition is well-defined if and only if two conditions hold:
  • At least one base case exists — providing a direct, non-recursive answer for the smallest input(s).
  • Every recursive chain terminates — the argument becomes strictly smaller with each call, so every chain reaches a base case in finitely many steps.
When both conditions hold, the definition assigns a unique, unambiguous value to every valid input.

The critical intuition: each recursive call must make the problem strictly smaller. This same progress guarantee is what you will analyze formally as algorithm termination in Module 11.2.

1 The Factorial Function

The factorial of a non-negative integer n counts the number of ways to arrange n distinct objects. Its recursive definition:

// Base case
0! = 1

// Recursive case  (n ≥ 1)
n! = n · (n−1)!

The argument decreases by 1 on every recursive call, so every chain reaches 0! in exactly n steps.

2 The Fibonacci Sequence

Fibonacci requires two base cases because the recursive case references the two preceding terms:

// Base cases
F(0) = 0
F(1) = 1

// Recursive case  (n ≥ 2)
F(n) = F(n−1) + F(n−2)

This example demonstrates that more than one base case can be necessary — a key point revisited in the Knowledge Check.

3 The Set of Binary Strings of Length ≤ n

Recursive definitions work for sets too. Let Bn denote the set of all binary strings whose length is at most n:

// Base case: strings of length 0
B0 = { ε }           // ε is the empty string

// Recursive case (n ≥ 1)
Bn = Bn−1
     ∪ { s·"0" | s ∈ Bn−1, |s| = n−1 }
     ∪ { s·"1" | s ∈ Bn−1, |s| = n−1 }

Here s·"0" means appending the character 0 to string s. The recursive case says: keep all shorter strings already in Bn−1, then add every string of length exactly n by extending each length-(n−1) string with a 0 or a 1.

Show Trace — building up to B2

Step 1 — B0 (base case):

B0 = { ε }

Step 2 — B1 (extend ε with "0" and "1"):

B1 = B0 ∪ { "0", "1" }
    = { ε, "0", "1" }

Step 3 — B2 (extend "0" and "1" each with "0" and "1"):

B2 = B1 ∪ { "00", "01", "10", "11" }
    = { ε, "0", "1", "00", "01", "10", "11" }

|B2| = 7 = 20 + 21 + 22 = 23 − 1. This counting pattern returns in Module 11.3.

💡 Set Recursion Pattern
Set-based recursive definitions follow the same two-part structure: the base case establishes the smallest members; the recursive case builds new, larger members from existing ones. This approach — called structural recursion — is the foundation for the formal grammar and tree material in Weeks 12–13.

3 · Well-Defined vs. Ill-Defined Recursion

Not every recursive-looking definition actually works. Two failure modes are especially common:

⚠ Failure Mode 1
Missing base case
No direct answer is ever given. The recursion unwinds indefinitely with nothing to anchor it.
⚠ Failure Mode 2
Recursive case moves away from the base case
A base case exists, but the argument grows (or never shrinks) with each call, so the chain is still infinite.

Annotated Counter-Examples

Ill-Defined · Failure Mode 1 · Missing Base Case
f(n) = f(n−1) + 3   // for all integers n
Problem: There is no base case. Computing f(5) requires f(4), which requires f(3), and so on, forever. No starting value is ever given, so f is undefined for every input.
Ill-Defined · Failure Mode 2 · Recursive Case Diverges
g(0) = 1                   // Base case
g(n) = 2 · g(n+1)       // for n ≥ 0
Problem: The base case is at n = 0, but the recursive case calls g(n+1) — a larger argument. Computing g(0) requires g(1), which requires g(2), and so on. The argument moves away from the base case and never terminates.

🧪 Mini-Case Activity

For each definition below, decide whether it is well-defined or ill-defined and justify your answer before revealing the explanation.

Activity · Is It Well-Defined?
Case 1. Consider this definition for non-negative integers:
h(0) = 1
h(n) = h(n−1) + h(n−2)  // for n ≥ 1

Is this definition well-defined for all non-negative integers? Justify your reasoning first, then reveal.

Ill-defined. The recursive case for n ≥ 1 references both h(n−1) and h(n−2). When n = 1, it requires h(0) — which is defined as 1 — and h(−1), which lies outside the non-negative integer domain and has no base case covering it. So h(1) cannot be computed.

Fix: Add a second base case h(1) = 1, and restrict the recursive case to n ≥ 2. Then both h(n−1) and h(n−2) are non-negative integers smaller than n, and every chain terminates. This produces the Fibonacci-like sequence 1, 1, 2, 3, 5, …
Case 2. Consider this definition for positive integers:
p(1) = 0
p(n) = p(n / 2) + 1   // for n ≥ 2, n a positive integer

Is this definition well-defined for all positive integers? Justify your reasoning first, then reveal.

Ill-defined as stated. When n is odd (e.g., n = 3), the expression n / 2 = 1.5 is not a positive integer, placing it outside the domain. The definition is silent about fractional inputs.

Fix: Use integer (floor) division: p(n) = p(⌊n/2⌋) + 1 for n ≥ 2. Under this reading, ⌊n/2⌋ < n for all n ≥ 2, so every chain terminates at the base case p(1) = 0. This is precisely the recursive definition of the number of bits required to represent n in binary.

4 · Structural Preview — Recursive Data Structures

The two-part pattern extends naturally to the data structures you will study in Weeks 12–15. As a brief preview, consider the linked list.

📘 Recursive Definition — Linked List
A linked list over a value set V is defined recursively:
  • Base case: The empty list (null) is a linked list.
  • Recursive case: If L is a linked list and v ∈ V, then a node holding value v with its next-pointer set to L is a linked list.

The concrete list [3 → 7 → 2 → null] is built by three applications of the recursive case on top of the base case:

Every operation on a linked list — length, search, insertion, reversal — can be written as a recursive algorithm that mirrors this definition directly. The central insight: when a data structure is defined recursively, algorithms that operate on it are naturally recursive too.

Three questions to consolidate Module 11.1 concepts. Responses are not recorded. Take your time and read the feedback for each question carefully.

Question 1 of 3 · Open Response
Write a recursive definition for the function f(n) = 2n, where n is a non-negative integer. Use the two-part format: state the base case and the recursive case. Draft your answer below, then reveal the model answer to compare.
📋 Model Answer

// Base case
f(0) = 1            // 20 = 1

// Recursive case (n ≥ 1)
f(n) = 2 · f(n−1)  // 2n = 2 · 2n−1
Why it works:
  • Base case: f(0) = 1 because 20 = 1 by definition of integer exponentiation.
  • Recursive case: f(n) = 2 · f(n−1) follows from 2n = 2 · 2n−1. The argument decreases by 1 each call, so every chain terminates at f(0).

✔ Spot check: f(3) = 2·f(2) = 2·2·f(1) = 2·2·2·f(0) = 2·2·2·1 = 8 = 23 ✓

6 · Summary and Bridge

  • ✦Recursive definition — defines an object in terms of a simpler version of itself, anchored by at least one base case.
  • ✦Base case — provides a direct, non-recursive answer for the simplest input(s); no self-reference allowed.
  • ✦Recursive case — reduces the input strictly, guaranteeing every chain eventually reaches the base case.
  • ✦Well-definedness — requires at least one base case and finitely terminating chains. Multiple base cases are sometimes necessary (Fibonacci is the canonical example).
  • ✦Two failure modes — (1) missing base case; (2) recursive case that moves away from the base case rather than toward it.
  • ✦Recursive structures — linked lists, trees, and grammars obey the same two-part pattern, making recursive algorithms a natural fit for them.
