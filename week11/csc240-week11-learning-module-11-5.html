1Divide-and-Conquer Recurrences — Setup

A divide-and-conquer algorithm solves a problem of size n by splitting it into a subproblems each of size n/b, solving them recursively, and combining the results. Every such algorithm has a running time that satisfies a standard recurrence form.

Standard Form

T(n) = a · T(n/b) + f(n) with base case T(1) = Θ(1)

Annotated Parameter Meanings

Symbol Constraint Algorithmic Meaning Merge Sort Example

bb > 1, realFactor by which input size is reduced per subproblem2 — each half has n/2 elements
f(n)Asymptotically positiveWork done outside the recursive calls (divide + combine steps)Θ(n) — the merge step
n/b—Size of each subproblem (floor or ceiling — same asymptotic result)n/2 elements per half
ℹ Exactness vs. Asymptotics
In practice n/b may not be an integer. The Master Theorem holds for both ⌊n/b⌋ and ⌈n/b⌉ — the asymptotic class is unchanged. We write T(n/b) as convenient shorthand throughout this module.

Worked Setup — Merge Sort

Algorithm description: Split the array in half, sort each half recursively, then merge the two sorted halves into one sorted array.

  1. Count subproblems (a): Two recursive calls are made — one on the left half, one on the right half. → a = 2
  2. Determine size reduction (b): Each half contains n/2 elements. → b = 2
  3. Identify combine work f(n): Merging two sorted halves requires a single linear pass — Θ(n) comparisons and element moves. → f(n) = Θ(n)
  4. Write the recurrence:
T(n) = 2 · T(n/2) + Θ(n) a = 2  |  b = 2  |  f(n) = Θ(n)

The Master Theorem resolves this recurrence in Section 3. The result — Θ(n log n) — confirms merge sort is optimal for comparison-based sorting.

Worked Setup — Binary Search

Algorithm description: Compare the target to the middle element of a sorted array; recurse on the left half or the right half — never both.

  1. Count subproblems (a): Only one recursive call is made — either left or right, never both. → a = 1
  2. Determine size reduction (b): The search space is halved each call. → b = 2
  3. Identify combine work f(n): One comparison per call — constant work outside the recursion. → f(n) = Θ(1)
  4. Write the recurrence:
T(n) = T(n/2) + Θ(1) a = 1  |  b = 2  |  f(n) = Θ(1)

Note: a = 1 is perfectly valid — it means only one subproblem is created, which is characteristic of search-reduction algorithms.

2The Master Theorem

The Master Theorem provides a direct, closed-form asymptotic result for any recurrence of the form T(n) = a·T(n/b) + f(n), provided the relationship between f(n) and the critical function np falls into one of three cases. It replaces the need to expand the recursion tree or apply the characteristic equation for this entire class of algorithms.

Define: p = logb a p is the critical exponent — the rate at which total subproblem work accumulates across all recursion levels

Formal Statement — Three Cases

Case 1 — Recursion Dominates

If f(n) = O(np − ε)
for some constant ε > 0,

then T(n) = Θ(np)

f(n) grows polynomially slower than np. The total cost of all recursive subproblems at the leaves dominates; the combining work is negligible by comparison. The recursion tree is "bottom-heavy."

Case 2 — Work is Balanced

If f(n) = Θ(np · logk n)
for some constant k ≥ 0,

then T(n) = Θ(np · logk+1 n)

f(n) and np grow at the same rate. Every level of the recursion tree contributes roughly equal cost — they share the work, adding exactly one log factor. Most common subcase: k = 0 → T(n) = Θ(np log n).

Case 3 — Combining Dominates

If f(n) = Ω(np + ε) for ε > 0
AND a·f(n/b) ≤ c·f(n) for some c < 1
(regularity condition),

then T(n) = Θ(f(n))

f(n) grows polynomially faster than np. The work at the top level of the recursion tree swamps the combined cost of all sub-levels. The tree is "top-heavy."

✔ Most Common Subcase of Case 2
When k = 0 (i.e., f(n) = Θ(np)), Case 2 simplifies to T(n) = Θ(np log n). This is the version you will encounter most frequently — including merge sort and binary search.
⚠ Regularity Condition — Case 3
The condition a·f(n/b) ≤ c·f(n) ensures f(n) does not oscillate. For all standard functions (polynomials, polynomials × logs) it holds automatically whenever f(n) = Ω(np+ε). You rarely need to verify it explicitly for textbook problems — but always mention it when stating the theorem formally.

💡 Quick Check Technique
After computing p, ask: "Is the exponent of n in f(n) less than, equal to, or greater than p?" For any polynomial f(n) = Θ(nc), compare c to p directly. You do not need ε — it only appears in the formal statement to establish the polynomial gap.

Step 3: Same rate, k = 0. → Case 2.

T(n) = Θ(n log n) — same complexity class as merge sort, even though the branching factor is different.

4When the Master Theorem Does Not Apply

The Master Theorem is powerful within its domain — but that domain has real boundaries. Before applying it, verify that none of the following disqualifying conditions holds.

Disqualifying Conditions

  1. Subtraction form — not divide-and-conquer. Recurrences such as T(n) = T(n−1) + f(n) use T(n−c), not T(n/b). The subproblem size decreases by a constant rather than a multiplicative factor. The Master Theorem is not applicable.
  2. a < 1. You cannot create fewer than one subproblem. Values of a < 1 have no valid algorithmic interpretation in a divide-and-conquer context.
  3. b ≤ 1. If b = 1, each subproblem has the same size as the original — the algorithm never terminates. If b < 1, the subproblem grows, which is impossible in this model.
  4. f(n) is not asymptotically positive. The theorem requires f(n) > 0 for all sufficiently large n. Oscillating or sign-changing functions violate this condition.
  5. f(n) falls in the gap between cases. There exist functions — such as n / log n when p = 1 — that are neither polynomially smaller than, equal to, nor polynomially larger than np. For such recurrences, none of the three cases applies.

Illustrative Examples

Recurrence: T(n) = T(n−1) + 1, with T(1) = 1.

Why the Master Theorem does not apply: The recursive call is on T(n−1), not T(n/b). There is no multiplicative size reduction — the problem shrinks by a constant at each step. Condition 1 is violated.

Correct method — direct unrolling:

T(n) = T(n−1) + 1 = T(n−2) + 2 = T(n−3) + 3  ⋮ = T(1) + (n−1) = 1 + (n−1) = n
Actual Result
T(n) = Θ(n) — This models a simple linear scan, such as counting elements one by one.

Recurrence: T(n) = 2·T(n/2) + n / log n.

Parameters: a = 2, b = 2, f(n) = n / log n

Critical exponent: p = log2 2 = 1, so np = n.

Check Case 1: Is f(n) = O(n1−ε) for some ε > 0?  No — n / log n grows faster than n1−ε for every fixed ε > 0, because log n grows slower than nε.

Check Case 2: Is f(n) = Θ(n)?  No — n / log n < n asymptotically; f(n) ≠ Θ(n).

Check Case 3: Is f(n) = Ω(n1+ε)?  No — n / log n < n < n1+ε.

Verdict — Not Applicable
f(n) = n / log n falls in the gap between Case 1 and Case 2. None of the three cases applies. The Akra–Bazzi method or direct recursion-tree analysis is required. The correct result (for reference only) is T(n) = Θ(n log log n).
💡 Rule of Thumb
If f(n) contains any function other than a pure polynomial or a polynomial multiplied by an integer power of log — for example, f(n) = n / log n, f(n) = n log log n, or f(n) = 2n — verify all three cases carefully before concluding the Master Theorem applies.

Practical Algorithm Selection

✔ When D&C Helps
  • The problem naturally splits into independent, equal-sized subproblems.
  • The combining step is cheap relative to the subproblem work (Case 1 or Case 2 outcome).
  • A logarithmic number of recursion levels is the key insight — eliminating half the work per level compounds into a huge advantage.
⚠ When to Reconsider
  • The combining step is as expensive as the original problem — you may recover the same complexity (as in naive matrix multiply).
  • Subproblems share overlapping computations — dynamic programming is likely superior.
  • The recurrence form is non-standard — asymptotic behavior may be worse than anticipated.
📊 Growth Rate Ordering
For algorithm selection, keep this ordering in mind — every step represents an exponential practical difference at large n:
Θ(1) < Θ(log n) < Θ(n) < Θ(n log n) < Θ(n²) < Θ(n³) < Θ(2n)
Divide-and-conquer algorithms typically land in the Θ(log n) to Θ(n log n) range — far superior to the Θ(n²) or Θ(n³) of naive iterative approaches on the same problems.

Implementation Tips

  1. Represent f(n) as an exponent. The simplest input model for Part 4 asks the user to enter the exponent c where f(n) = Θ(nc). Accept c as a double. This avoids complex symbolic expression parsing while covering all standard examples.
  2. Compute p with the change-of-base formula. Use Math.log(a) / Math.log(b) (Java) or log(a) / log(b) (C++) to compute p = logb a in floating-point. Never assume base 2 unless b = 2.
  3. Use a tolerance for case comparison. Define a small epsilon (e.g., 1e-9) and test: if c < p - eps → Case 1; if Math.abs(c - p) < eps → Case 2 (k=0); if c > p + eps → Case 3. Never use == for floating-point equality.
  4. Print results symbolically, not numerically. Output strings such as "Theta(n^3)", "Theta(n log n)", or "Theta(log n)". A numeric approximation is insufficient — the question asks for the complexity class.
  5. Handle the log n special case. When Case 2 applies with p = 0 (binary search), the result is Θ(log n), not Θ(n0 log n). Add a conditional to print "Theta(log n)" when p ≈ 0.
  6. Validate b > 1. If the user enters b ≤ 1, print an error: "Invalid: b must be greater than 1 for divide-and-conquer recurrences." Do not attempt to apply the theorem.

Common Mistakes to Avoid

⚠ Most Common PE-08 Errors
  • Floating-point equality for case selection — Always use a tolerance, never ==.
  • Wrong log base — logb a ≠ log2 a unless b = 2. Use the change-of-base formula.
  • Not mentioning the regularity condition for Case 3 — Even if you do not verify it, include a printed note that it should be checked.
  • Numeric output for complexity class — Print symbolic strings like "Theta(n log n)", not a decimal approximation of p.
  • Off-by-one in term generation — If T(1) is your base case, index your output array starting at 1, or document clearly that you start at index 0.
  • Fewer than three test cases submitted — Include at least merge sort, binary search, and one additional example. All three cover different cases.

Proof-Writing Checklist

Verify every item below for each proof before you submit PS-07. Missing any item is the most common source of point deductions.

For ALL proof types:

  • The claim to be proved is stated explicitly at the top of your solution.
  • The proof technique is named (direct proof, contradiction, weak induction, strong induction).
  • Every logical step follows clearly from the previous one — no unexplained jumps.
  • Definitions are cited when first used: "by the recursive definition of factorial…", "by the induction hypothesis…"
  • The final line explicitly states that the proof is complete and what was shown.

For Induction Proofs (primary PS-07 type):

  • Base case: Verified explicitly for the smallest valid input (usually n = 0 or n = 1). Never omit it even if "obvious."
  • Induction hypothesis: Stated as a formal assumption: "Assume the claim holds for n = k" (weak) or "Assume the claim holds for all values up to k" (strong).
  • Inductive step: The induction hypothesis is invoked explicitly — mark exactly where you use it in your proof.
  • Strong vs. weak induction: If the recursive case depends on values other than just k, switch to strong induction and restate the hypothesis accordingly.
  • The conclusion of the inductive step matches the statement of the claim at n = k+1 exactly.

For Correctness Proofs (Module 11.3 style):

  • Pre-condition stated: what is guaranteed about the input before the function call.
  • Post-condition stated: what the function is required to return or produce.
  • Correctness argument: given that recursive sub-calls return correct results (induction hypothesis), show the full function returns the correct result.
  • Termination proof: name a strictly decreasing non-negative integer measure (e.g., the value of n) and show each recursive call strictly reduces it.

For Recurrence Setup and Solving Problems (Module 11.4 style):

  • Recurrence written with all terms shown (not just the final form).
  • Characteristic equation derived and written out explicitly.
  • All factoring steps shown — do not skip from equation to roots.
  • General solution written with undetermined constants c₁, c₂, …
  • Initial conditions applied and constants solved — all algebra shown.
  • Final solution verified by substituting back into the original recurrence for at least the base case and one recursive step.

⚠ Most Common PS-07 Deductions
  • Missing base case — A proof without a verified base case is structurally incomplete, even when the inductive step is correct.
  • Circular reasoning — Treating the claim you are trying to prove as an established step within the same proof.
  • "It is clear that…" without justification — Every algebraic manipulation must be shown. What is clear to the author is not always clear to the reader.
  • Using weak induction when strong is needed — If your recursive function calls itself on two or more smaller values (e.g., Fibonacci), weak induction only covers one of those calls. State the strong induction hypothesis explicitly.
